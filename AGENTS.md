# AI エージェント向けガイド

このドキュメントは、AI エージェントがこのリポジトリで記事執筆を支援する際の指示をまとめたものです。

## 記事（article）と本（book）

このリポジトリでは Zenn の「記事」と「本」の両方を扱いますが、**基本は記事**です。
作業対象のパス（`articles/` か `books/`）でどちらかを判断できます。

`books/` 配下を編集するとき（本の執筆・チャプター追加など）は、`.claude/rules/books.md` を参照してください。本固有の構成・命名規則・メタデータはそちらにまとめています。

## 記事執筆時の必須事項

### 1. 執筆スタイルの参照

記事を執筆する際は、**必ず `writing-style.md` を参照してください**。

このファイルには、過去の記事から抽出された執筆スタイルの特徴が詳細にまとめられています。

- 文体的特徴
- 構造パターン
- トーンと視点
- 技術的な表現方法
- よく使われる表現
- 記事タイプ別の特徴

### 2. 執筆の流れ

1. `writing-style.md` を読んで、執筆スタイルを理解する
2. 記事のトピックと対象読者を明確にする
3. 記事の構成を計画する（結論先行型を推奨）
4. スタイルガイドに沿って執筆する
5. コード例は Before/After や NG/OK で対比する
6. 技術用語には初出時に説明を加える

### 3. 重要な執筆原則

以下の 3 つの原則を常に意識してください：

1. **結論を先に書く**: 読者が記事の価値をすぐに理解できるように、要点を冒頭で示す
2. **具体例を豊富に使う**: 抽象的な概念は必ず具体的なコード例や身近な例で説明する
3. **読者への配慮を忘れない**: 親しみやすく、でも正確に、技術的な内容を伝える

## 記事のメタデータ

Zenn の記事には以下のメタデータが必要です：

```markdown
---
title: "記事のタイトル"
emoji: "📝"
type: "tech" # または "idea"
topics: ["javascript", "typescript"] # 関連するトピック
published: false # 公開状態
---
```

**重要**: 新規記事作成時は、`published` を必ず `false` に設定してください。`true` に変更するのは人間が行うため、AI が変更してはいけません。

## ファイル命名規則

- 新規記事のファイル名: `articles/` 配下に作成
- ファイル名は意味のある slug (例: `react-hooks-guide.md`)
  - メイントピックをプレフィックスとして使用し、ソートしたときに見出し順に並ぶようにする
  - 例： CSS を扱った記事は `css-` プレフィックスを使用

## トピック選定のルール

- **テスト関連のtopicは `test` を使用**してください（`testing` ではなく `test` を使う）

## リンクのルール

- **MDN へリンクするときは日本語版 (`/ja/`) の URL を使用**してください（例: `https://developer.mozilla.org/ja/docs/...`）。`/en-US/` ではなく `/ja/` にします。

### Zenn のカード表示（リンクカード）

- Zenn では、**URL だけを 1 行で書き、前後を空行で挟む**と、タイトルや OGP 画像つきのカード表示になる。それ以外（文中の URL や、行頭に `参考: ` などのテキストが付く URL）は通常のテキストリンクになる。
- カード表示の例（前後に空行が必要）:

  ```markdown
  参考:

  https://example.com/article
  ```

- テキストリンクになる例（カードにならない）:

  ```markdown
  参考: https://example.com/article
  ```

- 使い分け: カード表示は視覚的に目立つため、**流し見でも見つけてほしい主要な情報・出典**に使う。すべてのリンクをカード化するのではなく、補足程度のリンクは 1 行のテキストリンクのままにして強弱をつける。

## 見出しと目次

- **Zenn の記事および本の目次には h2 (`##`) と h3 (`###`) だけが表示されます**（h4 以降は目次に出ません）。
  - 主要な項目は h2・h3 で全体像を把握できるように構成する（読者は目次で見通しを得る）。
  - 一方で、h3 を増やしすぎない。細かい区分や、1 つの項目が複数の方式に分かれる場合などは h4 (`####`) に下げるか、箇条書きでまとめて、目次が長くなりすぎないようにする。
