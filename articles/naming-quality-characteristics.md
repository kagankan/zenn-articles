「よい命名」「可読性の高い名前」と言っても、様々な観点が存在します。
この記事では、命名を決定する上で考慮すべき観点、コードの可読性を上げるために名前が持つべき性質を **命名品質特性** として分類します。ソフトウェア品質特性に倣って、 名前そのものがもつ **16の品質** と、使った結果として現れる **8の効果**（利用時の品質）として整理しました。

## はじめに

私は普段の開発で、可読性の高いコードになるよう、変数や関数などの名前にはこだわっています。自分が書くコードはもちろん、他のメンバーが書いたコードをレビューするときも同様です。良い名前は理解がしやすくなり保守性を高めますし、逆に不適切な名前は読者（他の開発者）の誤解を生みやすくなります。

「可読性の高い名前を」とは言いますが、実際に名前を選ぶときに意識している観点はいくつもあります。意図が伝わるか、他の名前と揃っているか、使い方を間違えないか、……などなど。これらをまとめて「可読性」と呼んでしまうこともありますが、中身はかなり違いますし、ときにはトレードオフも発生します。

そこで、これらの観点を一度体系的に整理してみたくなりました。どんな観点が存在するのか、網羅的に並べ、まとめてみます。

### 分類の参考：ソフトウェア品質特性

分類の参考にしたのは **ソフトウェア品質特性**（ISO/IEC 25010 など）です。これ自体の詳しい解説は他の記事に譲りますが、ソフトウェアの品質を機能性・信頼性・保守性……などに分類する枠組みです。

https://www.veriserve.co.jp/helloqualityworld/media/20250618001/

これに倣って名前の良さの観点を分類し、「**命名品質特性**」と名付けました。
ここで示す分類基準・用語は筆者個人が整理した**この記事独自のもの**です。分け方や分類名には議論の余地があると思います。

### この記事が対象とする名前

本記事が主に対象とするのは、開発者が名付ける **プログラム・コードの名前** です。

- 変数名・関数名
- ディレクトリ名・ファイル名
- API の名前（エンドポイント・パラメータなど）
- DB カラム名
- リポジトリ名

ただし、この記事で挙げる観点の多くは、発展的にプログラム以外の命名にも応用できるはずです。

- UI 上の文言（ボタンラベルなど）
- ページタイトル
- サービス名・プロダクト名
- 技術ブログの記事タイトル

…などなど。

:::message

なお、筆者は Web フロントエンド領域の人間なので、例は HTML・CSS・JavaScript・TypeScript・React のコードが中心です。
ただし **命名の考え方は特定の言語に依存しない**ですし、コード例では、当該言語を知らなくても雰囲気で読める程度のものです。

:::

### この記事で対象としないこと

以下の 2 点は命名以前の **前提** として、この記事では扱いません。

- **何に名付けるか**
  - マジックナンバーや複雑な条件式、概念の塊を、どこで切り出して名前付きの単位にするか、という問題です。
  - これはこれで重要なテーマで、名付けにくさが設計を見直すきっかけにもなりますが、この記事の対象外とします（この点はおわりにで触れます）。
- **言語仕様上、動くか動かないか**
  - 言語仕様上、使用できる文字で構成されていること
  - 予約語とコンフリクトして使用できない名前でないこと
  - 名前によって挙動が変わるもの（React コンポーネントは大文字始まり、Next.js でページとして公開するには `page.tsx` など）

### AI時代の命名の重要性

前述した通り命名の重要性は昔から語られていましたが、**AI というテキスト処理を得意とする存在** が開発に入ってきたことで、近年その必要性はさらに増しています。

- 名前が曖昧だと、AI が意図を理解するためのコンテキスト（説明）が増える。**単体のワードで意図が伝わる名前** は、それだけでコンテキストを節約する。
- 「個人開発で自分しか読まないコードだから可読性を気にする必要がない」ということもこれまでならあり得たが、AI と共同で書くなら、**AI に伝わること** がそのまま開発効率に影響する。

コードの読み手は、もはや人間だけではありません。人にも AI にも伝わる名前を命名することが重要です。

## カテゴリ分け：名前そのもの / 利用時

さて、まずは命名の品質をどのように分類するか考えます。
今回以下のようにカテゴリ分けしています。

```
命名の品質
├── 名前そのものの品質
│   ├── 自立品質
│   ├── 内部相対品質
│   └── 外部相対品質
└── 利用時の品質
```

- **名前そのものの品質**
  - 名前（やその集まり）を静的に眺めるだけで判定できる性質。
  - 「スペルが正しいか」「指示対象の中身と合っているか」など
- **利用時の品質**
  - 実際に読む・書く・探す場面で初めて現れる結果・効果。
  - 「読み手が意味を正しく当てられたか」「必要な人がその名前にたどり着けたか」など。
  - 複数の「名前そのものの品質」が寄与して生まれる。

この 2 層構成は、ISO が製品品質（ISO/IEC 25010）と利用時の品質（同 25019）を分けているのに倣ったものです。

### 自立 / 内部相対 / 外部相対

「名前そのものの品質」については、**「名前のよさを、何と照らし合わせて判断するか」** によってさらに 3 つに分けています。

- **自立品質**: 名前とその対象、そしてあらかじめ想定した前提（使う言語や対象読者）だけで判断できる品質。他の名前を見に行かなくても判断できる。
- **内部相対品質**: コード内の **他の名前・スコープ** との関係で決まる。
- **外部相対品質**: コードの外にある **規約・慣習・社会** との関係で決まる。

:::message

なお、この 3 つの境界は **何を「前提」として固定するかで変動します**。

たとえば後述の「言語的正確性」は主に「英語として」正しいかを見る観点で、突き詰めれば英語というコードの外部の存在との関係（外部相対）とも言えます。ですが、英語の綴り・文法は、ほぼすべてのコードの前提となっており、どのプロジェクトでも答えが変わらないので「自立品質」として扱っています。

:::

ここからは具体的に個別の性質を紹介していきます。

:::message

以降のコード例で 🪫 は「その節の観点である〇〇性が低い」、🔋 は「その節の観点である〇〇性が高い」を表します。
あくまでその観点の中での比較であり、**絶対的な正解・不正解ではありません**。ある性質が高い名前でも、他の性質で評価するとむしろ劣ることもあります。
その名前が使われる文脈によっても良し悪しの評価は変わります。

:::

## 自立品質 (standalone quality)

名前とその対象、そして固定された前提（使う言語や想定読者）だけで判断できる品質です。

### 意味的正確性 (semantic accuracy)

非常に基本的な項目です。名前が **指示対象と意味的に合っているか** の基準です。

- 名前が指す対象の事実と合っているか
- 型と名前が矛盾しない

特に避けたいのは「嘘をつく名前」です。例えば、`getName` が実際には name ではないものを取得しているとしたら、読み手を積極的に誤解させる分、「情報がない名前」よりたちが悪いです。

あえて間違った命名をしようとする人はいないでしょうが、すでにあるコードを変更したときに、名前を変え忘れると名前と中身が乖離することはよくあります。

```ts:事例: 嘘つき変数名
// 🪫 低: 内容と合っていない（age を返すのに name と言っている）
function getName() {
  return user.age;
}

// 🔋 高: 内容と合っている
function getAge() {
  return user.age;
}
```

```ts:事例: 型と合っていない名前
// 🪫 低: 型と合っていない
const userCountString: number = 10;

// 🔋 高
const userCountNumber: number = 10;
```

:::message

繰り返しますが、「🔋 高」で示す名前の例は、この節の観点である「意味的正確性」が「🪫 低」の名前に対して高いということを示しており、必ずしもこの名前が良いということではありません。
実際のケースでは、`userCount` のように型情報を省略したほうがよいことが多いでしょう。

:::

### 言語的正確性 (linguistic accuracy)

名前が **自然言語として正しいか** どうか。自然言語というのは英語や日本語のこと。
意味的正確性が「指示対象と意味が合っているか」だったのに対し、こちらは語そのものの形式が正しいかを見ます。プログラムにおいては多くの場合英語が使われるため、主に英語の話で、大きく「綴り（スペル）」と「文法」に分かれます。

#### 綴り（スペル）

英単語として正しいスペルで記述されているか。

```ts:事例: スペルミス
// 🪫 低: スペルミス
savedSuccessfly = true;

// 🔋 高: 正しいスペル
savedSuccessfully = true;
```

```ts:事例: 存在しない単語
// 🪫 低: regist という単語は存在しない
//        "register" が動詞+er に見えることで勘違いする人が多い
function regist() {}
// 🔋 高: register という動詞が存在する
function register() {}
```

このようなスペルミスは Code Spell Checker（VSCode 拡張）/ CSpell（CLI ツール）で機械的に検出できます。

https://streetsidesoftware.com/vscode-spell-checker/

https://cspell.org/

#### 文法

英語の文法上正しい表現を使っているかどうか。
日本人（英語を母語としない人）、特に英語が苦手な人は間違えがちです。

```ts:事例: is に縛られた命名「ボタンを表示するかどうか」
// 🪫 低: ブール値ならば is、を意識しすぎて文法がおかしい
// 「ボタンを表示するかどうか」を表したい
isShowButton = true;
// 🔋 高
// 「するかどうか」なので should
shouldShowButton = true;
// is で始めるなら、受動態
isButtonShown = true;
// もしくは動詞
showButton = true;
```

```ts:事例: is に縛られた命名「編集可能かどうか」
// 🪫 低: ブール値ならば is、を意識しすぎて英語がおかしい
// 「編集可能かどうか」を表したい
isCanEdit = true;
// 🔋 高
// 「できるかどうか」なので can
canEdit = true;
// is で始めるなら、-able で表現
isEditable = true;
```

<!-- 逆に、**データ（名詞）なのに動詞句**で名付けると関数に見える。送信予定のメッセージを `sendMessage` とすると「メッセージを送る動作」に読めてしまう。データなら名詞句で `messageToSend`（あるいは `outgoingMessage`）とする。`userName`（関数なのに名詞）とちょうど逆向きの取り違えで、英語を母語としない開発者がやりがちだ。 -->

```ts:事例: する予定の〇〇
// 🪫 低: データなのに動詞
// 「このあとの処理で送信するメッセージのデータ」を表したいのに、
// 「メッセージを送る関数」に読めてしまう
sendMessage = { to: "a@example.com", body: "Hello" };
// 🔋 高: データは名詞句で
messageToSend = { to: "a@example.com", body: "Hello" };
```

「複数形には `s` がつく」のも英語のルールです。

```ts:事例: 複数形
// 🪫 低: 配列（複数の値を内包する）なのに単数形
user = ["Alice", "Bob", "Carol"]
// 🔋 高: 複数の値を持つので複数形になっている
users = ["Alice", "Bob", "Carol"]
```

### 明確性 (clarity)

名前だけで **意図・使い方** が分かるか。

- 具体性があること。抽象的な名前だと、実際に何を意味するコードか読み取れない
- 「どのように（実装）」ではなく「何のために（目的）」で名付ける

```ts:事例: 実装でなく目的で名付ける
// 🪫 低: 抽象すぎる
getDate();
// 🪫 低: 具体的にはなったが、役割がわからない
// 「3か月後を取得」= 実装の説明
getThreeMonthsLater();
// 🔋 高: 役割・目的のわかる名前
// 「有効期限を取得」= 目的
getExpiredDate();
```

```ts:事例: 状態・時制を名前に反映させる
// 🪫 低: load だけでは進行中か完了済みか曖昧
load = false;
// 🔋 高: 時制・相で状態を言い分ける
isLoading = false; // 読み込み中か
isLoaded = false; // 読み込みが完了したか
```

#### 機械的な名前でなく中身を語る名前にする

連番のように機械的につけられた名前は明確性が低い名前の例です。例えば `img1`/`img2` のように名付けられた画像ファイルは、何の画像かわかりません。中身の内容で名付ければ、名前だけで中身が推測できます。

AI にコーディングをさせる際には、意味のある名前にしておくことで、詳細な指示をしなくてもファイル名から推察して適切な画像を使ってくれるといった場合も多くなります。

```ts:事例: 機械的な名前でなく中身を語る名前にする
// 🪫 低: 連番。中身が分からない
fileNames = ["img1.png", "img2.png"];
// 🔋 高: 中身を名前で語る
fileNames = ["user-icon.png", "product-image.png"];
```

#### 単位を名前で補う

名前だけで使い方が分かれば、コメントやドキュメントを読みにいかなくて済みます。数値なら単位を名前に足すのはよく使われるテクニックです。

```ts:事例: 単位を名前で補う
// 🪫 低: 単位が分からない（秒かもしれないしミリ秒かもしれない）
setTimeout(fn, duration);
// 🔋 高: 単位を名前に含める
setTimeout(fn, durationMs);
```

#### 使うときの注意（副作用・リスク）を名前で伝える

使うときに注意が必要な場合、それを名前で伝えることも明確性を上げることの 1 つです。

実例として、React には `dangerouslySetInnerHTML` というプロパティが存在します。この機能にはクロスサイトスクリプティング (XSS) を実行させうるセキュリティ上の危険性があります。そのため、あえて `dangerously`（危険な）から始まる長ったらしい名前にしています。こうすることで、**たとえドキュメントを読んでいない人でも、この機能を使おうとしたときには必ず名前を目にすることになる** ので、「なんか危なそうだ」と思わせ軽率に使用しないようにさせています。

https://ja.react.dev/reference/react-dom/components/common#dangerously-setting-the-inner-html

```tsx:事例: 危険性を名前で伝える dangerouslySetInnerHTML
// 🪫 低: XSS の危険性が名前からは分からない
setHTML={{ __html: '<script>alert("XSS!")</script>' }}

// 🔋 高: 名前で危険性を明示（React に実際に存在する名前）
dangerouslySetInnerHTML={{ __html: '<script>alert("XSS!")</script>' }}
```

ライブラリに実験的に追加された機能で見られる `unstable〇〇`, `experimental〇〇` といった命名も似たような思想です。動作が保証されていなかったり、仕様が変わる可能性があったりする実験的機能であることを名前で示しています。

```ts:事例: 不安定な機能であることを名前で伝える
// 🪫 低: 安定した機能であるように見える
import { LinkControl } from "@wordpress/block-editor";

// 🔋 高: 名前に実験的な機能であることを示す
// WordPress のブロックエディタのコンポーネントで実際に存在した名前
import { __experimentalLinkControl } from "@wordpress/block-editor";
```

#### コラム: 「間違ったコードを間違って見せる」

良い名前の働きのひとつに、**間違ったコードを間違って見せる** というものがあります。
これは Joel Spolsky 氏の「Making Wrong Code Look Wrong」という記事からの引用です。

https://www.joelonsoftware.com/2005/05/11/making-wrong-code-look-wrong/

この記事で用いられている例は、文字列がサニタイズ済み（クロスサイトスクリプティングの危険性を取り除く処理がされている）か否かを `s`（safe）/`us`（unsafe）で区別するものでした（本記事では読みやすさのため `safe`/`unsafe` と置き換えています）。

```ts:事例: 間違ったコードを間違って見せる
// 🪫 低: どちらもただの文字列で、安全・危険の区別がつかない
input = req.body.comment;
html = sanitize(input);

// 🔋 高: サニタイズ前後を名前で言い分ける（ちぐはぐな代入が目立つ）
unsafeInput = req.body.comment;
safeHtml = sanitize(unsafeInput);
```

このような命名ルールにしておけば、正しくない代入がされていると、**処理の全体を追いかけなくてもその行のみを見るだけで間違っていることがわかります**。

```ts
// ❌️ 入力値をサニタイズせずに "safe" に入れているのはおかしい
safeInput = req.body.comment;
// ❌️ "unsafe" な値をサニタイズせずに "safe" に入れているのはおかしい
safeHtml = unsafeInput;
```

「意味や種別を接頭辞・接尾辞で添える」やり方は、**ハンガリアン記法** の一種であり、特に **アプリケーションハンガリアン** と呼ばれます。前述の `durationMs`（秒かミリ秒かを名前で補う）も、型では表せない「単位」を名前に込めている点で、この仲間です。

この方法が有効なのは、**型システムでは区別できない違い** があるときです。

- どちらも文字列だが、サニタイズ前後が違う
- どちらも日付の文字列だが、日付だけか時刻まで含むかが違う
- どちらも数値だが、秒とミリ秒が違う
- どちらも数値だが、円とドルが違う

こうした「型は同じだが意味が違う」差は、名前で補うと取り違えに気づけます。

「わわわ（「分かりそう」で「分からない」でも「分かった」気になれる IT 用語辞典）」のハンガリー記法のページにわかりやすい事例があったのでこちらも引用します。

https://wa3.i-3-i.info/word13959.html

```ts:事例: 型では区別できない違いを名前で補う（ドル・円）
// 🪫 低: 実は単位が違うのに、名前から気付けない
売上_全体 = 売上_本店 + 売上_支部1 + 売上_支部2
// 🔋 高: 名前に単位が明示され、間違っているコードが間違って見える
// ❌️ 円にドル単位の値をそのまま足しており、式の見た目で誤りに気付ける
円_売上_全体 = 円_売上_本店 + ドル_売上_支部1 + ドル_売上_支部2
```

なお、単位などの意味的なものではなく、データ型を付加する形式もハンガリアン記法と呼びます。アプリケーションハンガリアンに対し「システムハンガリアン」と呼ばれます。こちらは `strName`（string）・`iCount`（int）のように型を接頭辞にするものです。
型は型システムが保証してくれる情報なので名前に書くと重複し、リファクタで `int` を変えたのに `iCount` のまま……と **名前が嘘をつく** 事故も招きます。IDE の機能が十分でなかった時代には意味があったかもしれませんが、コーディング中にホバーするだけで型情報が見られる環境では、デメリットの方が大きいと言えます。静的な型のない言語では意味があるかもしれませんが、型のある言語を使っている限りは利点はあまりありません。

というわけで、ハンガリアン記法と言っても、 **型のある言語では、型を含めるシステムハンガリアンは基本的に要らず、意味を含めるアプリケーションハンガリアンは役に立つ** ということです。

ただし、モダンな型システムがあるなら **型そのもので区別する** ほうが強力です。TypeScript の [branded types](https://typescriptbook.jp/reference/values-types-variables/structural-subtyping#%E3%83%96%E3%83%A9%E3%83%B3%E3%83%89%E5%9E%8B) のように `SafeHtml` と `UnsafeInput` を別の型にすれば、取り違えを型レベルで検出できます。**型で表せるならまず型で、表せない（あるいは型を定義するほどでもない）ときに名前で補う** とよいでしょう。

### 平易性 (simplicity)

簡単に読める名前であり、速く正しく読めるか。言い換えると、読むときの認知的な負荷が低いか。

- 文字数が短いこと（目で認識しやすい）
- 平易な語であること（辞書を引かずに理解できる）
- 読み上げられること（会話やレビューで困るため）
- 単語の区切りが視認できること（どこで区切れるか迷わない）
- 否定・反転がなく、意味を素直に追えること

など、パッと読めることに寄与する観点全般を含みます。

```ts:事例: 長過ぎる変数名
// 🪫 低: 長すぎて一目で読み取れない
numberOfUsersWhoHaveNotVerifiedEmailYet
// 🔋 高: 短くして読みやすくする
unverifiedUserCount
```

```ts:事例: 馴染みのない単語
// 🪫 低: 日常的でない語で、辞書を引かないとわからない
commenceUpload()
// 🔋 高: 平易な語で表現する
startUpload()
```

```ts:事例: 口に出して読みにくい名前
// 🪫 低: 省略形で読めない・発音できない
ymdhms = "2024-06-22 03:03:52";
// 🔋 高: 辞書に載っている一般的な単語
timestamp = "2024-06-22 03:03:52";
```

```ts:事例: 略語を使わない
// 🪫 低: 略語は読み手によって解釈が分かれる
btn
bnr
ttl // total? title?
// 🔋 高: 省略せずに書く
button
banner
title
```

```ts:事例: 否定形を避ける
// 🪫 低: 否定形で命名されている
isNotEnabled = false;
if (!isNotEnabled) ... // 否定で判定するとさらに混乱

// 🔋 高: あらかじめ反転させた意味で命名する
isDisabled = false;
```

```ts:事例: 区切りがわかりにくい名前
// 🪫 低: 区切りがわかりにくい
contenteditable
// 🔋 高: 区切りが明確
content_editable
```

:::message

`contenteditable` は実際に存在する HTML のグローバル属性のひとつです。
https://developer.mozilla.org/ja/docs/Web/HTML/Reference/Global_attributes/contenteditable
HTML の属性名の仕様上、区切り文字もなくすべて小文字で連結されています。仕様上しかたないのですが、読みにくい。
初見で「contented/itable」と区切って読んでしまい、content の過去形ってなんやねんと思ってしまいました。
個人的に読みにくいワードランキングの上位です。

:::

#### 日本語を活用する

日本語を母語とするメンバーが中心の開発チームであれば、日本語の変数名を使うのも「平易性」を向上させる有効な手段です。
プログラミングでは変数名・関数名などを英語で命名することが一般的ですが、特に専門用語・ドメイン用語などで英語にするとわかりにくい場合、ローマ字書きで日本語読みの名前を用いてもいいでしょう。
さらに、現代のプログラミング言語では日本語（非 ASCII 文字）で書くことも仕様上問題なくなっています。

```ts
// 🪫 低: （日本語話者にとって）subject=教科のことと認識しづらい
// また、subject には「件名」という意味もあるため、誤解を生む
subject = "国語";
// 🔋 高: ローマ字書きにすることで、日本語話者にはわかりやすい
kyouka = "国語";
// 🔋 高: 日本語を変数名に使うことも可能（少なくとも JavaScript で）
教科 = "国語";
```

```ts:事例: 日本語変数名の活用 その1

};
```

### 選択容易性 (selectability)

名前を **一語としてまとめて扱える** かどうか。
テキスト上にカーソルを合わせてダブルクリックすると単語をまとめて選択できます。また、`option`+`←` (macOS)、`Ctrl`+`←` (Windows) のような単語単位のカーソル移動のショートカットがあります。これらの操作をしたときの利便性を考慮する観点です。

`-`（ハイフン）はそこが切れ目とみなされるため、`my-value`（ケバブケース）はダブルクリックすると `my` か `value` の片方しか選べません。`my_value`（スネークケース）や `myValue`（キャメルケース）であれば、一語として選択・移動できます。

ピンポイントな観点ではありますが、編集時の操作性や利用時の利便性に影響するため、意外と重要です。

```css:事例: 一語として扱えるか
/* 🪫 低: ケバブケースでは `-` が単語区切りになり、ダブルクリックで全体を選べない */
.my-component
/* 🔋 高: スネークケースやキャメルケースなら一語として選択・移動できる */
.my_component
.MyComponent
```

以前私が書いた CSS の命名設計に関する記事では、この観点を考慮に入れて `BlockName__ElementName` のような形式の命名を採用していました。

https://zenn.dev/kagan/articles/css-brain-bemding

## 内部相対品質 (internal-relative quality)

自立品質ではその名前単体で評価できる性質を見ましたが、内部相対品質では、**他の名前と比べて**評価される性質を見ていきます。
同じ関数の中の他の変数名、同じファイルの中の他の関数名、同じリポジトリ内の他のファイル名……など、**あるスコープの中の相対的な関係性**で評価される品質です。

### 内部識別性 (internal identifiability)

コードの中で「これは何か」を正しく見分けられるか。これはさらに 2 つの側面に分けられます。

- **一貫性**: 同じものには同じ名前
- **弁別性**: 違うものには違う名前

:::message

この「識別性」という概念は、WCAG の達成基準 3.2.4「[一貫した識別性](https://waic.jp/translations/WCAG22/Understanding/consistent-identification.html)」も参考にしています。
WCAG はウェブコンテンツアクセシビリティガイドライン。すなわち UI に登場する要素のアイコンや文言の話ですが、コードの命名においてもユーザー（＝開発者・AI）にとっての認知的負荷として捉えられます。

:::

#### 一貫性

同じ概念には同じ名前を使うこと。一度決めた名前を、他の場所でも一貫して使用する。

##### 同じ概念には同じ名前

```ts:事例: 同じ概念なのに異なる命名（「ユーザー」）
// 🪫 低: 同じ「ユーザー」の英訳がバラバラ
userId
newCustomer
MemberList
createAccount()

// 🔋 高: 同じ概念は同じ単語で揃える
userId
newUser
UserList
createUser()
```

簡単な用語・概念ならいいのですが、難しい概念や日本人に馴染みの薄い英単語だとブレがちです（実装者各自が「〇〇␣英語」のような Google 検索をしてそのとき出てきた単語を使うことで起こりがち）。

```ts:事例: 同じ概念なのに異なる命名（「手数料」）
// 🪫 低: 同じ「手数料」の英訳が fee / charge でバラバラ
getFee();
calculateCharge();

// 🔋 高: 同じ概念は同じ単語で揃える
getFee();
calculateFee();
```

##### 並列な存在には同じ構造の名前

全く同じ概念を指していない場合でも、並列な存在であれば似たような構造の名前を選択することで、近い存在であることを示唆できます。

```ts:事例: 他と違って見える命名
// 🪫 低: 同じ山手線の駅なのにカタカナを含む異なる名前
目黒
五反田
大崎
品川
高輪ゲートウェイ

// 🔋 高: 似た構造の名前であれば、並列な存在と認識できる
目黒
五反田
大崎
品川
高輪
```

山手線新駅の名称公募の結果の 1 位は「高輪」だったそうです。既存の駅名からすればそういう名前にしたいと思いますよね。
わざわざ「高輪ゲートウェイ」にしたのにも意図はあるのでしょうが、他の駅名と一貫性のない名前（異質な名前）であることは違いないでしょう。異質な名前にすることであえて新駅を目立たせたい意図があったんだと推測できます。

https://news.yahoo.co.jp/expert/articles/921ff3a6299a6c991464b029000c3f7e9c108947

https://president.jp/articles/-/33946

名前の構造の考え方はプログラミングでも活かせます。
並列な存在（変数・関数・コンポーネント）には同じ構造の名前をつけておくと理解しやすいでしょう。
たとえば、`Button` という名前で赤いボタンがすでにあったとします。後から色違いで緑のボタンを追加して `ButtonGreen` と名付けたとします。こうすると名前の構造が違ってしまうため、単なる色違いの並列な存在として見えなくなります（上下関係があるようにも見える）。元のボタンも `ButtonRed` に変えて、構造を揃えるべきでしょう。

```tsx
// 🪫 低: 実際は並列なバリアントなのに、違う構造
Button;
ButtonGreen;

// 🔋 高: 同じ構造の名前であれば、並列な存在と認識できる
ButtonRed;
ButtonGreen;
```

逆に、あえて構造の違いを作ることで、並列でない、従属関係・包含関係を示すこともできます。たとえば、`List` / `ListItem` のような名前であれば「List の中にある Item」という関係性を示唆できます。

```tsx
// 🪫 低: 従属・包含関係がわからない
List;
Item;
// 🔋 高: 関係性が分かる名前
List;
ListItem;
```

#### 弁別性

似ているが異なるものを区別できるようにすること。取り違えを防ぎたいときは、あえて冗長な名前を選ぶことも必要です。

```ts:事例: 似すぎた名前
// 🪫 低: 同じ概念ではあるが、変換前後で異なるデータ形式
options = "?a=1&b=2"
// ...また別の箇所で
options = { a: 1, b: 2};

// 🔋 高: 同じ概念だとしても、区別のために違う名前にする
rawOptions = "?a=1&b=2"
parsedOptions = { a: 1, b: 2};
```

##### コラム: 医療事故につながる似た名前の薬

プログラミングの文脈ではない事例ですが、識別性の観点で関連する事例として紹介します。
医療現場において名前・字面が似た薬剤が多いようで、しばしば薬剤を取り違える医療事故が発生するとのこと。
アメリカでは ISMP という団体が推奨する **Tall man lettering** という記法があるようです。似た名前の薬剤の、**異なる部分を大文字にして** 取り違えを防ぐ対策です。

```ts
// 🪫 低: 似た名前で間違えやすい
prednisone;
prednisolone;
// 🔋 高: 差のある部分を大文字で目立たせる
predniSONE;
prednisoLONE;
```

https://www.ismp.org/sites/default/files/attachments/2017-11/tallmanletters.pdf

また、そもそも名前を変える対応もあり、日本では糖尿病薬「アマリール」と似ていた高血圧症薬「アルマール」の名前を「アロチノロール」に変更したそうです。

### スタイル一貫性 (style consistency)

名前の形式・記法の流儀を、自分のコード内で揃えているか。命名規則・コーディング規約を定め、それに従っていること。
例えば、「ディレクトリ名は `kebab-case`」「変数名は `camelCase`」「定数は `UPPER_SNAKE`」など。どんなスタイルを選ぶにせよ、コードベース全体で一貫させていること。独自ルールでも、一貫してさえいれば読み手はそこから手がかりを得られます。

内部識別性が「語彙」で見分ける話だったのに対し、こちらは「形式」を揃える話です。
たとえば定数を `UPPER_SNAKE` に統一すると、読み手は形だけで「これは定数だ」と察することができます。

```ts:事例: 定数を見分けられるようにする命名
// 🪫 低: 記法がバラバラ
const tax_rate = 10; // 消費税率10%
const ShippingFee = 500; // 送料500円
...
// どれが変動する値で、どれが固定された値か把握しにくい
price * (100 + tax_rate) / 100 + ShippingFee;

// 🔋 高: 定数（あらかじめ固定された値）は UPPER_SNAKE で一貫させる
const TAX_RATE = 10; // 消費税率10%
const SHIPPING_FEE = 500; // 送料500円
...
// 変動する値が price だけであることが把握できる
price * (100 + TAX_RATE) / 100 + SHIPPING_FEE;
```

:::message

ここではコード内での一貫の話であり、エコシステムでの慣習に従う観点は後述の外部相対品質「スタイル適合性」で扱います。

:::

### 適量性 (proportionality)

スコープに対して情報量が過不足ないか。不足していても、逆に多すぎても良くないため、その 2 側面に分かれます。なお、どちらに傾けるべきかはスコープによって変わります。

- **簡潔性**: 多すぎる情報を削る
- **十分性**: 足りない情報を足す

#### 簡潔性

スコープが既に与えている情報は繰り返さない。狭いスコープなら短くてよい。
以下の例では、`userName` というそれ単体では間違ってもいないし明確に説明していますが、`user` オブジェクトというスコープの中では冗長になります。

```ts:事例: オブジェクトのプロパティ
// 🪫 低: user スコープ内なのに user を繰り返す（冗長）
user = {
  userName: "Alice",
  userId: 123,
};
// 🔋 高: user の話であることはわかりきっているので、省略する
user = {
  name: "Alice",
  id: 123,
};
```

他にも、ループ変数に短い（1 文字の）名前を使うのもこの観点です。数行のループの中だけという短いスコープなら、短い名前でよいです。

```ts:事例: ループ変数の名前
// 🪫 低: 単なるループ変数に長い名前を与えている
for (let userArrayIndex = 0; userArrayIndex < users.length; userArrayIndex++) {
  user = users[userArrayIndex];
}
// 🔋 高: 狭いスコープなら短く
for (let i = 0; i < users.length; i++) {
  user = users[i];
}
```

#### 十分性

逆に、広い・公開スコープでは、名前だけで意味が取れるよう具体性を足す必要があります。
スコープが文脈を与えてくれない分、名前に詰め込む必要があります。

```ts
// 🪫 低: モジュールから公開する名前にしては漠然としすぎ
export const data = ...;

// 🔋 高: 広いスコープには十分な具体性を
export const activeUsers = ...;
```

### 内部検索性 (greppability)

grep（ソースコード内での検索）で一意に当たるかの観点です。

**"greppability"** は英単語としては存在しない造語ですが、すでにウェブ上ではこの単語を使っている記事・投稿がいくつもあり、検索性の観点を表す言葉として定着しつつあります。

https://nippondanji.blogspot.com/2010/09/linux.html

https://ikesyo.hatenablog.com/entry/2020/10/28/130000

```ts:事例: 一意に grep できる名前に
// 🪫 低: grep すると大量にヒットして埋もれる
// user.ts
export const KEY = "KEY_FOR_USER";
// admin.ts
export const KEY = "KEY_FOR_ADMIN";

// 🔋 高: 一意に検索でき、補完でも絞り込める
export const KEY_FOR_USER = "KEY_FOR_USER";
export const KEY_FOR_ADMIN = "KEY_FOR_ADMIN";
```

#### grep できる定型マーカー（TODO コメント）

コメントをどんな文言で書くかも命名の 1 つです。
コメントの目印は、自由文よりも `TODO:` / `FIXME:` / `HACK:` のような **決まった語に揃える** と、後から横断検索・集計できます。

```ts:事例: 検索性を高める TODO コメント
// 🪫 低: 表記がバラバラで拾えない
// バリデーションをあとで追加する
// 使用していないので削除する
// 要修正！

// 🔋 高: 定型マーカーに揃えて grep できる
// TODO: バリデーションを追加する
// TODO: 使用していないので削除する
// FIXME: 境界値で破綻する
```

#### grep できるエラーコード

エラーメッセージの文言も重要な命名です。
たとえばユーザーから「エラーが出た」と問い合わせを受けたとき、画面やログに残るのが共通のメッセージだけだと、コードのどこで起きたのかを特定できません。

一意のメッセージになるように文言を変えたり、より確実に区別するために一意のエラーコードを併記したりしておけば、問い合わせに添えられたコードでそのまま grep でき、発生箇所を特定できます。

```ts:事例: grep できるエラーコード
// 🪫 低: 共通の文言だけ。どの箇所で起きたか grep で特定できない
throw new Error("エラーが発生しました");
// （別の箇所にも同じ文言があり、絞り込めない）
throw new Error("エラーが発生しました");

// 🔋 高: 一意のメッセージ＋一意のコードを併記
// 問い合わせのコードから発生箇所へ直行できる
throw new Error("E_AUTH_TOKEN_EXPIRED: セッションの有効期限が切れています");
throw new Error("E_PAYMENT_CARD_DECLINED: カードが承認されませんでした");
```

#### コラム: 検索性を下げないための工夫

命名とは別の観点にはなりますが、リポジトリ内での検索性を高めるには文字列連結による処理をできるだけ避けるのが望ましいです。

```tsx
// 🪫 低: ファイルの名前を連結して生成する
iconType = "book";
return <img src={`/icon_${iconType}.svg`} alt="" />;
```

このコードが実行されると `icon_book.svg` ファイルが参照されることになりますが、`icon_book` という名前でコード内を検索しても引っかかりません。
このファイルを使っていないと判断して間違って削除してしまうかもしれません。
未来の誰かが探したときにたどり着けるよう、ファイル名全体を文字列で持っておくようにするといいでしょう。

```ts
// 🔋 高: ファイル名全体を文字列で持つことで検索性を高める
iconType = "book";
ICON_FILE_MAPPING = {
  "book": "icon_book.svg",
  ...
};
return <img src={ICON_FILE_MAPPING[iconType]} alt="" />;
```

### 整列性 (sortability)

主にファイル名の命名で考慮する観点です。
一覧で名前順（アルファベット順）に並べられたときに、有用な並び順になるか。

- 関連する名前が固まるか
- 意味のある順番に並ぶか
- 特別なものが先頭や末尾に来るか（`_` で始めるなど）
- 連番が数字の大小でソートされるか

#### 関連する名前が固まるか

```ts
// 🪫 低: 関連する名前が離れてしまう
// ここでは MainSection と SubSection がセット
MainSection; // ★
Modal;
Navbar;
Pagination;
Spinner;
SubSection; // ★

// 🔋 高: 関連する名前が固まる
Modal;
Navbar;
Pagination;
SectionMain; // ★
SectionSub; // ★
Spinner;
```

関連するファイルが近くに並んでいれば、以下のようなメリットがあります。

- どちらにも同じ修正を加える必要がある場合、片方のファイルを編集した人（開発者）がもう片方のファイルの存在にも気づきやすくなる
- ファイルを利用する人（開発者）が、似たファイルを見つけやすくなる
  - たとえば、`ButtonPrimary` を使おうとした人がファイルを探したとき、近くに `ButtonSecondary` があれば目に入る。実はその人が使うべきファイルは `ButtonSecondary` だった、ということもあり得る

#### 意味のある順番で並ぶか

ディレクトリ設計においては、しばしばファイル・ディレクトリのレイヤー間の依存関係にルール（方向性）を設けている場合があります。
フロントエンドアーキテクチャでは [Feature-Sliced Design](https://feature-sliced.design/) や [Atomic Design](https://atomicdesign.bradfrost.com/) などがその例です。これらの設計では、依存関係の方向が決まっており、上位レイヤーは下位レイヤーに依存することはあっても、その逆はないというルールがあります。
この関係性が名前の順序性（ファイルエクスプローラーでの並び）で表すと、直感的に捉えやすくなります。

たとえば Feature-Sliced Design では、レイヤーが上位 → 下位の順にこう定義されています（依存は下方向にのみ許可されています）。

```text:依存の順序（上ほど上位）
app → pages → widgets → features → entities → shared
```

https://feature-sliced.design/docs/reference/layers

ところが、この層をフォルダ名としてそのままアルファベット順に並べると、依存の順序とは一致しません。

```text:アルファベット順では依存順にならない
app
entities   ← 依存では下のほうなのに 2 番目
features
pages
shared
widgets    ← 依存では上位なのに最後
```

FSD では依存方向をフォルダの並び順では表さず、[リンター (ESLint) の import ルール](https://github.com/feature-sliced/eslint-config/tree/master/rules/layers-slices)で担保しています。

逆に、**アルファベット順がそのままレイヤー順になるよう意図的に命名する**こともできます。私がまとめた独自の CSS 設計ルールである ABC-CSS は、レイヤー名の頭文字を並び順に対応させたものです。

```text:アルファベット順＝レイヤー順（ABC-CSS）
abstract
base
components
features
pages
utils
```

https://zenn.dev/kagan/articles/css-alphabetical-layers-css

#### 特別なものが先頭や末尾に来るか（`_` で始めるなど）

記号や数字は英字より前に並ぶことが多いので、名前の先頭に付けると一覧の上位に集まります。

たとえば Sass のパーシャル `_variables.scss` や Next.js の `_app.tsx` の `_` は、一覧でまとまって表示され、他のファイルと区別しやすくなります（これらはそもそもファイルの役割を区別するための規約であって整列のための仕様ではありませんが、結果として並び順が変わる効果もあります）。

```text
styles/
  _variables.scss  ← _ 始まりは先頭に集まる（区別の副産物）
  _mixins.scss
  base.scss
  layout.scss
```

#### 連番が数字の大小でソートされるか

```ts:事例: 連番が数字の大小でソートされる名前
// 🪫 低: 数字の順に並ばない
'image-1.png'
'image-10.png'
'image-2.png'
...

// 🔋 高: ゼロ埋めすることで数字の順に並ぶ
'image-01.png'
'image-02.png'
...
'image-10.png'
```

とは書いたものの、最近では数字列を数値の大小でソートしてくれるシステムが多くなっています（VSCode のエクスプローラー、Windows のエクスプローラー、macOS の Finder では image-01.png, image-02.png, image-10.png の順で並ぶことを確認しました）。
純粋な辞書順ソートがされる環境では、ゼロ埋めをしないと意図した順序に並びません。

### 補完容易性 (completion discoverability)

IDE 補完で絞り込めるか。共通プレフィックスで候補がまとまると探しやすくなります。

```ts:事例: 共通プレフィックスで補完を絞る
// 🪫 低: 補完で候補が散らばる
theme = {
  red: "#f00",
  blue: "#00f",
  green: "#0f0",
  small: "8px",
  medium: "16px",
  large: "32px",
};

// 🔋 高: 共通プレフィックスで候補がまとまる
theme = {
  colorRed: "#f00",
  colorBlue: "#00f",
  colorGreen: "#0f0",
  sizeSmall: "8px",
  sizeMedium: "16px",
  sizeLarge: "32px",
};
```

上記の例では、`theme.color` で補完すると色だけが、`theme.size` で補完するとサイズだけが候補に出るようになります。
`large` だったか `big` だったか `huge` だったか忘れても、`size` の一種であることだけ思い出せればたどり着けます。

### 連続性 (continuity, temporal consistency)

すでに広まった名前の変更を考える際の観点です。名前を変えないこと、もしくは変えたとしても関連性が認識できること。

開発者の間ですでに浸透している名前の場合、名前を変更すると必要な変数・関数を見つけられなくなるかもしれません。

公開 API やサービス名の場合は、昔から知っている人に伝わり続けることに価値があります。これは「内部一貫性」（ある時点における他のコードとの一貫性）に対する「時間軸の一貫」といえます。

#### 事例: GitHub のデフォルトブランチ名の変更

GitHub のデフォルトブランチ名が `master` から `main` に変更されましたが、これは結構連続性のある変更の例だなと勝手に思っています。
意味的に近い単語としては `primary` や `default` などもありますが、`main` は master と同じく **`ma`** からはじまる単語です。すでに `master` が定着している人にとっても視覚的に発見しやすく、master のつもりで ma... と入力したときに補完されることなども考えると、いい変更だと感じています（決定の際にこれが考慮されていたのかは知りませんが）。

```ts:事例: GitHub のデフォルトブランチ名の変更
// 🔋 高: 連続性
master → main
```

:::message

なお、そもそもこの名称変更が発生した背景は後述する「包摂性」の観点です。

:::

#### 事例: Twitter → X（連続性を手放した例）

いったん世界中に広まった名前を連続性のない名前に変更すると、既存の知識や期待が通じにくくなります。2023 年に Twitter が X に改称されましたが、この名前からは「Twitter のこと」とは連想しにくいです。
（さらに「X」は一般語すぎて Web 検索でも埋もれる（検索性の低下）とか、他と紛れて固有性も低い（識別性の低下）とか、色々問題はあるわけですが…。）
その結果、「X（旧 Twitter）」という表記をみんながする必要が発生しました。

```ts:事例: Twitter → X（連続性を手放した例）
// 🪫 低: 連続性がない名称変更
Twitter → X
```

## 外部相対品質 (external-relative quality)

内部相対品質が「コードの中の他の名前との関係」で決まる品質だったのに対し、この外部相対品質は、コードベースや開発チームの**外**にある、**規約・慣習・社会との関係**で決まる品質です。

### 外部識別性 (external identifiability)

コミュニティの慣習に照らして、「これは何か」が正しく見分けやすいかどうか。内部識別性と同じく **一貫＋弁別** の対となる 2 つの側面で検討します。

- **一貫性**：既存の慣習に照らし、同じ概念には同じ名前をつける
  - 世間一般の概念名
  - 標準的な略語（`id`, `url`, ...）
  - デザインパターン語彙（Adaptor, Observer, ...）
  - 慣習的な対義語ペア（`open`/`close`, `get`/`set`, ...）
- **弁別性**：前述のような慣習に照らし、違う概念には違う名前をつける

慣習と一致させることで、新しくプロジェクトに参画したエンジニアでも、その慣習さえ知っていればすぐに馴染むことができます。
AI も同じで、世の中のコードを学習しているので、慣習に沿った名前のほうが理解してもらいやすくなります。

#### 「負けを認めて」誤解されない名前を選ぶ

『リーダブルコード』の第 3 章「誤解されない名前」、3.7 節「ユーザの期待に合わせる」の言葉が好きなので引用します。

> たとえ別の意味で使っていたとしても、ユーザが先入観を持っているために誤解を招いてしまうことがある。こういうときは「負けを認めて」、誤解されない名前に変えたほうがいい。

（『リーダブルコード』第 3 章（34 ページ）より）

例も同じページから引用します。多くのプログラマは `get` で始まるメソッド（関数）が「軽量アクセサ」であるという規約に慣れ親しんでいます。そのため、重い（計算コストが高い）処理には `get` を使わないほうがいいでしょう。

```ts:事例: 重い処理に get を使わない
// 🪫 低: 重い（計算コストの高い）処理なのに `get`
getMean()
// 🔋 高: 計算コストの高さを名前で表す
computeMean()
```

### 外部検索性 (googlability、ググラビリティ、エゴサビリティ)

コードの外（Web 検索や SNS）で見つかるか。特に **公開パッケージ名・リポジトリ名・公開 API・サービス名** で考慮すべき観点です。
内部検索性が自分のリポジトリの中での話であったのに対し、こちらは Web という空間の話です。

- 一意にヒットすること
- その機能を求めている人が検索したときにヒットすること

例えばライブラリを公開したとき、使ってほしい人に見つけてもらうためには、検索でヒットする名前であることが助けになります。必ずしも名前に含めなくても、説明文章や紹介記事の内容から検索にヒットするとは思いますが、名前に機能が含められていると、その機能を求めている人がたどり着きやすくなります。

```ts:事例: 名前に具体性を持たせる
// 🪫 低: 抽象的で Web 検索でも埋もれる
super_utility
// 🔋 高: 機能を表しており、探している人が見つけやすい
date_calculation_functions
```

:::message

ここで扱うのはあくまで「見つかるか」であって、「覚えやすいか」「好印象か」（コピーライティング・ブランディング）は本記事の対象外です。

:::

### スタイル適合性 (style conformance)

使用している言語のコミュニティ、フレームワークのエコシステムにおける **形式・スタイルの流儀・慣習** に合うか。
前述のスタイル一貫性が「自分のコード内で形式を揃える」話だったのに対し、こちらはその揃え先を世間の慣習にする話です。

#### React における例

React では以下のような命名の慣習があります。

- イベントハンドラの変数名は `handleClick` (`handle〇〇`)
- イベントハンドラの props は `onClick` (`on〇〇`)

> 慣習的に、イベントハンドラは handle の後ろにイベントの名称をつなげた名前にすることが一般的です。

> 慣習として、イベントハンドラのプロップは on で始まり、次に大文字の文字が続くようにします。

https://ja.react.dev/learn/responding-to-events

これは別に仕様ではないので、全く違う名前にしても動作の上では問題ありません。ただ、これに合わせておくことで他の開発者や AI も、把握がしやすくなります。

```tsx:事例: ライブラリ（React）の慣習に合わせる
// 🪫 低: 動くが React の慣習から外れる
<button whenClick={clickHandler} />;
// 🔋 高: props は on*、変数名は handle*
<button onClick={handleClick} />;
```

### 包摂性 (inclusivity)

「インクルーシブ」や「インクルージョン」というカタカナ語で馴染みがあるかもしれません。これは「あらゆる人が孤立したり、排除されたりしないよう援護し、社会の構成員として包み、支え合う」という理念を表します。^[引用元: https://www.hurights.or.jp/japan/learn/terms/2011/10/new.html]
命名においては、差別的な含意のある語を避け、中立的な語を使用することをこの観点に分類します。

特に有名なものは、GitHub のデフォルトブランチが `master` から `main` に変更された例です。`master` はかつての奴隷制度を連想させる用語として見直しが進んでいます。
こういった言葉は **[inclusive language](https://en.wikipedia.org/wiki/Inclusive_language)** と呼ばれます。

プログラミングの文脈で代表的な例は以下のようなものです。

| 不適切な語            | 推奨される語（一例）  |
| --------------------- | --------------------- |
| master                | main                  |
| master / slave        | primary / secondary   |
| blacklist             | deny list, block list |
| whitelist             | allow list            |
| dummy（変数・データ） | placeholder           |

<!-- 日本においても、ジェンダー平等への配慮で「女優」ではなく「俳優」、「婦警」ではなく「（女性）警官」という表現を用いるように進んでいます^[『記者ハンドブック』第14版には「ジェンダー平等への配慮」が追加されている]。 -->

## 利用時の品質：名前を使った結果

ここまでの 3 大分類は「名前そのものの品質」（ISO でいう製品品質にあたる）でした。ISO/IEC 25010 が製品品質とは別に「利用時の品質」（使った結果として現れる品質）を分けているように、命名でも「使った結果として生まれる品質」を考えてみましょう。
これらは、これまでに分類してきた複数の特性が寄与して生まれます。

<!-- ISO の利用時の品質（有効性・効率性・リスク回避性・利用状況網羅性）になぞらえた 7 つに、modifiability 由来の変更容易性を加えた 8 つ。 -->

### 予測容易性 (predictability)

読み手が「この名前はこういう意味だろう」と正しく当てられること。内部識別性・外部識別性・スタイル一貫性・スタイル適合性などが寄与します（定数が `UPPER_SNAKE` なら「定数だ」と当てられる、など。「誤解を招かない」はこの裏返し）。

<!-- 有効性に対応 -->

規約が一貫していれば、初見の名前でも中身を確かめる前に種類を当てられます。逆にバラバラだと予測がきかなくなります。

```ts:事例: 定数を見分けられるようにする命名
// 🪫 低: 記法がバラバラ
const tax_rate = 10; // 消費税率10%
const ShippingFee = 500; // 送料500円
...
// どれが変動する値で、どれが固定された値か把握しにくい
price * (100 + tax_rate) / 100 + ShippingFee;

// 🔋 高: 定数（あらかじめ固定された値）は UPPER_SNAKE で一貫させる
const TAX_RATE = 10; // 消費税率10%
const SHIPPING_FEE = 500; // 送料500円
...
// 変動する値が price だけであることが予測できる
price * (100 + TAX_RATE) / 100 + SHIPPING_FEE;
```

```ts:事例: 値であることが予測できる（関数に誤解しない）名前
// 🪫 低: データなのに動詞で関数のように見える
sendMessage = { to, body }

// 🔋 高: データは名詞句
messageToSend = { to, body }
```

### 理解容易性 (understandability)

速く・楽に読み解けること。予測容易性が「正しく分かる」なら、こちらは「速く分かる」。平易性（短さ・平易な語）と適量性（情報の過不足のなさ）が主に寄与します。

<!-- 効率性に対応 -->

意味は同じでも、長く込み入った名前は読み解くのに時間がかかります。正確さを保ちつつ短くまとまっていると、理解しやすい。

```ts:事例: 理解しやすい名前
// 🪫 低: 正確だが、読み解くのに時間がかかる
theListOfUsersWhoAreCurrentlyActive = getUsers();
// 🔋 高: 一目で頭に入る
activeUsers = getUsers();
```

### 伝達容易性 (communicability)

名前を **共通の語彙** として、チームの会話・レビューや AI への指示で、正確かつ素早く指し示せること。平易性（読める・発音できる）と内部識別性の弁別性（`data` のような何でも屋にせず、指すものが一意に決まる）が寄与します。

<!-- 効率性に対応 -->

- 難しい単語だと読みにくい・間違えやすい
- 何でもかんでも同じ名前（たとえば `data`）だと、どのことを指しているかわからない

### 到達容易性 (reachability)

必要な人が、その名前に **たどり着ける** こと。

<!-- 有効性に対応 -->

- 同僚が既存のユーティリティ関数を見つけて再利用する（重複実装を防ぐ）
- 使ってほしいライブラリが Web 検索で見つかって採用される

など。

内部検索性・外部検索性があれば見つけやすくなり、明確性があれば「これが探していたものだ」と感じられる。

#### 既存の関数にたどり着く

「日付を整形する関数、どこかにあった気がする」というとき、慣習どおりに名付けられていれば、「`format` とか入ってるんじゃないかな～」と、補完や grep で当たりをつけてたどり着け、重複実装を避けられます。場当たりな名前だと存在に気づかず作り直してしまいます。

```ts
// 🪫 低: 場当たりな名前。存在を予測できず重複実装してしまう
function toStr2(d: Date) {}
// 🔋 高: 慣習に従った、挙動を表した名前。
// `format` で補完すれば候補に出てたどり着ける
function formatDate(d: Date) {}
```

こういった探し方は AI（私が主に使っているのが Claude Code なのでそれを念頭にしています）もよくやっています。

ためしに、手元にあるリポジトリで「このリポジトリに日付をフォーマットする関数はありますか？」と Claude Code に調査を依頼しました。その後 Claude Code 自身に「どうやって（どんなコマンドで）探しましたか？」と聞いてみました。
すると以下のように説明してくれました。

```sh
# 1. ファイル名で当たりをつける（find + grep）
find /repo \
  -type f \( -name "*.ts" -o -name "*.tsx" -o -name "*.js" -o -name "*.jsx" \) \
  | grep -E "(util|lib|helper|format)" | head -20

# 日付フォーマットが置かれそうなファイル名（util, lib, helper, format）でファイル一覧を絞り込み。

# 2. 関数名パターンで全文検索（grep -r）
grep -r "formatDate\|dateFormat\|toDateString\|format.*Date" \
  /repo \
  --include="*.ts" --include="*.tsx" --include="*.js" --include="*.jsx" \
  | grep -v node_modules | head -30

# よくある関数名パターン（formatDate, dateFormat, toDateString, format.*Date）を OR で横断検索。

# ...以降も具体的な定義箇所の調査が続いていたが省略
```

Claude Code 自身が説明しているとおりですが、
「そういうフォーマット関数は `util` `lib` `helper` `format` といった名前のついているファイルにありがちだ」、という推測をもとにファイルを探し、
「そういうフォーマット関数には `formatDate`, `dateFormat`, `toDateString`, `format〇〇Date` といった名前がついていがちだ」という推測を元に関数定義を探していました。

このとき検証したリポジトリには実際に `toDateString` という名前の関数があり、発見に成功していました。

機能を表すような名前（明確性を持つ名前）をつけ、慣習に従った名前（外部識別性のある名前）にすることで、このように発見しやすくなります。

### 安全性 (safety)

紛らわしい名前による **誤用・バグを防げる** こと。使う人（ある名前の関数を使ってコードを書く人、ある名前の API を実行する人など）が、使い方を間違えることを未然に防ぐ観点です。内部識別性の弁別性（紛らわしい名前で取り違えない）と明確性（危険の存在を名前で知らせる）が寄与します。

<!-- リスク回避性に対応 -->

- 取り違えを起こさない
- 危険の存在を知らせる

```ts:事例: 取り違えやすい名前を避ける
// 🪫 低: 1 文字違いで紛らわしく、取り違えやすい
deleteAt; // 削除予定時刻？ 削除済み時刻？
deletedAt;
// 🔋 高: 紛れない名前にして取り違えを防ぐ
scheduledDeletionAt; // 削除予定
deletedAt; // 削除済み
```

### 安定性 (stability)

将来の変更で名前を **変えずに済む** こと。適量性（拡張の余地を残す）や、将来も事実に反しない名付け（意味的正確性）が寄与します。
なお、公開した名前を約束として変えない **連続性** とは別の概念です。連続性が「変えたくても変えない」なのに対し、安定性は「そもそも変える必要が出ない」という結果を指します。

<!-- リスク回避性に対応？ -->

```ts:事例: And で列挙した名前
// 🪫 低: 返すフィールドを名前に列挙している
function getNameAndId(user) {
  return {
    name: user.name,
    id: user.id,
  };
}
// → email も返したくなると、名前まで変えることになる
function getNameAndIdAndEmail(user) {
  return {
    name: user.name,
    id: user.id,
    email: user.email,
  };
}

// 🔋 高: 役割で名付ければ、返す項目が増えても名前は有効なまま
function getUserSummary(user) {
  return {
    name: user.name,
    id: user.id,
    email: user.email,
  };
}
```

#### 拡張性の低い命名は安定性も低い

たとえばサイズ段階に名前を付けるとき、`largest` のような最上級の語を選ぶと、より大きい段階を増やしたくなったときに名前が嘘になり、改名の必要が出てきてしまいます。`s`/`m`/`l` のような相対スケールにしておけば、あとから `xl` や `2xl` を足すだけで済み、既存の名前は変えずに済みます。

```ts
// 🪫 低:
sizes = {
  large: "30px",
  larger: "40px",
  largest: "50px",
};
// すでに最上級の名前を使ってしまったが、
// "60px" を追加したいときどうする？
// 既存の名前を変更する？

// 🔋 高:
sizes = {
  lg: "30px",
  xl: "40px",
  "2xl": "50px",
  "3xl": "60px",
};
```

:::message

「**レポート\_最終版\_final\_修正\_v2.docx**」みたいな名前もこの系統の問題ですね。
最初から「レポート\_v1.docx」「レポート\_v2.docx」みたいな名前をつけていけば数を増やすだけで済みます。

:::

### 変更容易性 (modifiability)

いざ名前を変えるとき、**楽かつ安全に直せる** こと。安定性が「そもそも変えずに済む」状態、安全性が「使うときに取り違えない」ことであるのに対し、こちらは「名前を変える作業を安全・確実にする」目線です。
一意でよく弁別された名前は、grep で全使用箇所に漏れなく当たり（内部検索性）、似た名前と紛れず誤認しない（弁別性）ので、リネーム（一括置換）が安全・確実になります。逆に `data` や `id` のような頻出語は、置換しようとすると無関係な箇所まで巻き込みかねません。

<!-- リスク回避性＋効率性。modifiability は ISO だと保守性（製品側）だが、ここでは利用時の効果として借用 -->

### 読み手網羅性 (reader coverage)

**ベテランでも新人でも、人間でもAIでも、誰が読んでも通じる** こと。平易性（普通の語を選ぶ）・明確性・外部識別性（慣習どおりの語）などが寄与します。

チーム内では通じる独自の略語も、新しく入った人・非母語話者・AI には通じないことがある。読み手の幅を広げたいなら、場当たりな省略より普通の語を選ぶ。

```ts:事例: 略語を避ける
// 🪫 低: 内輪でしか通じない省略（Ttl は total? title?）
usrAddr = "";
calcTtl = () => {};
// 🔋 高: 誰が読んでも分かる
userAddress = "";
calculateTotal = () => {};
```

## トレードオフとの向き合い方

ここまで分類してきた命名品質特性は、**全部を同時には満たせません**。なぜなら、特性間に**トレードオフがある**からです。最終的には優先順位を決める必要があります。

```ts:性質間のトレードオフ
d; // 簡潔だが意味が薄い
daysSinceLastLogin; // 明確だが長い
```

どちらが正解かは単体では決められません。判断の基準は「**影響範囲（スコープ）**」です。
3 行のループ内なら `d` で十分ですし、公開 API の引数なら `daysSinceLastLogin` のような明確さがあったほうが望ましいです。
ローカル変数のリネームはいくらでもできます。一方で、外部に公開する API・URL・DB カラム・JSON キーなどは変更が難しくなります。影響範囲が広い名前ほど、意味的正確性・一貫性・明確性の側を重視する必要があります。

「全特性を満たす名前」ではなく **「その文脈で優先すべき特性を選んだ名前」が良い** ということです。

### 事例: 短いスコープの変数名（平易性 vs 明確性）

前述の `d` と `daysSinceLastLogin` で見たように、
最も基本的な衝突は、**平易性（短さ・簡潔さ）vs 明確性** です。

```ts:短いスコープでは短い名前でよい（ループ変数）
for (let index = 0; index < 9; index++) {
  console.log(index);
}

for (let i = 0; i < 9; i++) {
  console.log(i);
}
```

```ts:短いスコープでは短い名前でよい（補足されたエラーの変数）
try {
  ...
} catch (error) {
  console.error(error);
}

try {
  ...
} catch (e) {
  console.error(e);
}
```

狭いスコープなら短い名前で問題ありませんが、行数が長くなってきたり、他の変数が多くなったりしたら考え直してもいいでしょう。
このトレードオフはすでに紹介した **適量性** によって測る基準です。

### 事例: 「アンケート」（平易性 vs 言語的正確性）

たとえば、「アンケート」機能があるようなときの命名について。日本では「アンケート」として馴染みがありますが、これはもともとフランス語。`enquete` のような変数名を書くと Code Spell Checker ではスペルミス扱いされます（英語辞書で設定していれば）。英語では **"questionnaire"** と言います。
だからといって questionnaire が絶対的な正解かというとそういうことでもなく、日本語母語話者（「アンケート」で認識している人）が中心の開発チームであれば、直感的にアンケートのことだとわかりやすい `enquete` を使うのも十分選択肢になるでしょう。もしくは `アンケート` でもいいかもしれません。

```ts:事例: 「アンケート」の命名
// 英語における「言語的正確性」を取るならこちら
questionnaire;
// 日本人にとっての「平易性」を取るならこちら
enquete;
// より日本人にとっての「平易性」を優先し、
// 「スタイル適合性」（変数名はアルファベットで書くという慣習）を捨てるならこちら
アンケート
```

### 事例: 関連ファイルを近くに並べる名前（整列性 vs 言語的正確性）

英語の語順的な正しさで言えば形容詞を先に、名詞をあとに並べるのが正しいですが、「整列性」のコード例で紹介した通り、ファイル名をアルファベット順に並べたときに関連ファイルが並ぶようにすることを考慮するなら、逆順になります。

```ts
// 英語として正しい語順
MainSection;
SubSection;

// 整列性を考えた語順
SectionMain;
SectionSub;
```

### 事例: `referer` （言語的正確性 vs 連続性）

HTTP ヘッダの `referer` は、本来英単語として正しい綴りは `referrer`（r が足される）です。
これは仕様策定時にスペルミスをしてしまった結果、それが今でも使われているためです。
いったん広まった綴りを変えないために **連続性**（後方互換）を優先し、**言語的正確性** を捨てている（捨てざるを得ない）例です。

https://zenn.dev/motch/articles/567a456b42cc4f

### 事例: `dangerouslySetInnerHTML` （明確性・安全性 vs 平易性）

React の `dangerouslySetInnerHTML` は、シンプルに書けば `setHtml` で済むところを、あえて長く物々しくして XSS（クロスサイトスクリプティング）の危険があることを **明確** に伝える名前で、**安全性**（誤用への警告）を最優先した名前になっています。

## おわりに：名付けることは、対象を概念として認識すること

漠然としていた「良い命名」の観点を分類し、それぞれの特性にも名前を付けてきました。

**名前が付くと、漠然とした存在が形のある概念として認識できるようになります。** この、「名前を知ると、その存在に気づけるようになる」現象にもまた名前が付いていて、ジョシュアツリーの法則と呼ばれています。『[ノンデザイナーズ・デザインブック](https://book.mynavi.jp/nddb/)』のイントロで触れられている話です。

https://objectclub.jp/technicaldoc/object-orientation/principle/principle04

今回の記事ではすでに名前をつけることは決まっている前提でその名前の品質を語ってきましたが、そもそも「これに名前を付けるべきでは？」と気づくことが重要です。コード中には、まだ名前が付いていないけれどある概念を表している存在が多く眠っているはずです。たとえば、

- マジックナンバーがあるとして、その値は何らかの意味・役割があって指定されているはずです
- if 文に長い条件式があるとして、その式は何らかの状態を表しているはずです

その概念を切り出して、名前を付けましょう。

この記事で取り組んだ分類自体が、その実践でもあります。観点に名前が付いていれば、「ここはなんとなく読みにくい」ではなく「ここは明確性が高いが適量性が低い」みたいな捉え方ができます。レビューでも自分の頭の中でも、議論の解像度が上がります。

そして、名前が対象の認識・理解につながる効果は逆向きにも作用します。つまり、**命名が難しいときは、正しく対象を切り出せていないかもしれない**ということです。そういうときは**設計を見直すべきときかもしれません**。

- `getNameAndId` のようにやっていることを羅列する名前しか浮かばないのは、その処理を切り出す意味がないからかもしれません。
- `〇〇Manager` のように広く漠然とした名前しか浮かばないのは、その対象が多くの責務を抱えていて、一語で言い表せないからではないでしょうか。

名前を付けようとする行為が、「ここは分けるべきだ」「これは別のあれと同じ概念だ」「この知識はチーム内で共有すべき概念だ」といった気づきを与えてくれます。
命名にこだわることは、コードを読みやすくするだけでなく、設計そのものを良くすることにもつながります。
