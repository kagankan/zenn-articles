コードを書くうえで、命名は非常に重要であるとよく語られます。とはいえ、よい命名、「可読性の高い名前」と言っても、様々な観点が存在します。
この記事では、命名を決定する上で考慮すべき観点、コードの可読性を上げるために名前が持つべきを **命名品質特性** として分類します。ソフトウェア品質特性に倣って、 名前そのものがもつ **16**の品質と、使った結果として現れる「利用時の品質」**8** として整理しました。

## はじめに

私は普段の開発で、可読性の高いコードになるよう、変数や関数などの名前にはこだわっています。自分が書くコードはもちろん、他のメンバーが書いたコードをレビューするときも同様です。良い名前は理解がしやすくなり保守性を高めますし、逆に不適切な名前は読者（他の開発者）の誤解を生みやすくなります。

「可読性の高い名前を」とは言いますが、実際に名前を選ぶときに意識している観点はいくつもあります。事実と合っているか、意図が伝わるか、他の名前と紛れないか、チームの慣習に沿うか……などなど。これらをまとめて「可読性」と呼んでしまうこともありますが、中身はかなり違いますし、ときにはトレードオフも発生します。

そこで、その観点を一度きちんと整理したくなりました。どんな観点が存在するのか、網羅的に並べ、まとめ上げる取り組みです。
整理して観点に名前を付けてみると、漠然とした存在から形のある概念として認識できるようになります。「ここはなんとなく読みにくい」ではなく「ここは _明確性_ は高いが _適量性_ が低い」のような表現ができれば、レビューでも自分の頭の中でも、議論の解像度が上がります。

### この記事で対象としないこと

以下の 2 点は命名以前の **前提** として、この記事では扱いません。

- **何に名付けるか**
  - マジックナンバーや複雑な条件式、概念の塊を、どこで切り出して名前付きの単位にするか、という問題です。
  - これはこれで重要なテーマで、**名付けにくさは設計を見直すきっかけ** でもありますが、この記事の対象外とします。（この点はおわりにで触れます）。
- **言語仕様上、動くか動かないか**
  - 言語仕様上、使用できる文字で構成されていること（コンパイルが通る）
  - 予約語とコンフリクトして使用できない名前でないこと
  - 名前によって挙動が変わるもの（React コンポーネントは大文字始まり、Next.js でページとして公開するには `page.tsx` など）

### AI時代の命名の重要性

前述した通り命名の重要性は昔から語られていましたが、**AI というテキスト処理を得意とする存在** が開発に入ってきたことで、近年その必要性はさらに増しています。

- 名前が曖昧だと、AI が意図を理解するためのコンテキスト（説明）が増える。**単体のワードで意図が伝わる名前** は、それだけでコンテキストを節約する。
- これまで「個人開発で、他人とコードを共有しないなら可読性は二の次」という割り切りもあり得た。だが AI と共同で書くなら、**AI に伝わること** がそのまま開発体験に効く。

コードの読み手は、もはや人間だけではありません。人にも AI にも伝わる名前を命名することが重要です。

## 主軸：「何と照らして」評価するか

さて、まずは命名の品質をどのように分類するか考えます。
今回以下のようにカテゴリ分けしています。

```
命名の品質
├── 名前そのものの品質
│   ├── 自立品質
│   ├── 内部相対品質
│   └── 外部相対品質
└── 利用時の品質
```

- **名前そのものの品質**
  - 名前（やその集まり）を静的に眺めるだけで判定できる性質。
  - 「スペルが正しいか」「規約に沿っているか」など
- **利用時の品質**
  - 実際に読む・書く・探す場面で初めて現れる **結果**。
  - 「読み手が意味を正しく当てられたか」「必要な人がその名前にたどり着けたか」など。単一の名前の性質には還元できず、複数の「名前そのものの品質」が寄与して生まれる **アウトカム**。

この 2 層構成は、ISO が製品品質（ISO/IEC 25010）と利用時の品質（同 25019）を別モデルに分けているのに倣ったものです。本記事では前者を、名前の話だと分かりやすいよう **名前そのものの品質** と呼び換えます。

**名前そのものの品質** については、**「名前のよさを、何と照らし合わせて判断するか」** によってさらに 3 つに分けています。

- **自立品質**: 名前とその対象、そして固定された前提（使う言語や読者層）だけで判断できる品質。他の名前を見に行かなくても判断できる。
- **内部相対品質**: コード内の **他の名前・スコープ・過去の自分** との関係で決まる。
- **外部相対品質**: コードの外にある **規約・慣習・社会** との一致で決まる。

<!-- なお **何を前提に固定するかで、この境界は動く**。たとえば「英語」や「React」を前提に固定すれば、本来は外部との一致だった観点も、固定ルールに従うだけの自立評価に近づく。この点はトレードオフの節で改めて触れる。 -->

ここからは具体的に個別の性質を紹介していきます。

:::message

以降のコード例で 🪫 は「その観点（〇〇性）が低い」、🔋 は「高い」を表します。
あくまでその観点での比較であり、**絶対的な正解・不正解ではありません**。
同じ名前でも観点ごとに評価は変わり、総合的な良し悪しは文脈（影響範囲や固定した前提）で決まります。

:::

:::message

コード例はあくまで命名の観点を示すためのものであり、実際に動くコードではありません。

:::

## 自立品質 (standalone quality)

名前とその対象、そして固定された前提（使う言語や読者層）だけで判断できる品質です。

### 意味的正確性 (semantic accuracy)

非常に基本的な項目です。名前が **指示対象（実体）と意味的に合っている** かの基準です。

- 名前が指す対象の **事実** と合っているか
- 型と名前が矛盾しない

特に避けたいのは「嘘をつく名前」です。例えば、`getName` が実際には name ではないものを取得しているとしたら、読み手を積極的に誤解させる分、「情報がない名前」よりたちが悪いです。

```ts:事例: 嘘つき変数名
// 🪫 低: 内容と合っていない（age を取っているのに name と言っている）
function getName() {
  return user.age;
}

// 🔋 高: 内容と合っている
function getAge() {
  return user.age;
}
```

```ts:事例: 型と合っていない名前
// 🪫 低: 型と合っていない
const userCountString: number = 10;

// 🔋 高
const userCountNumber: number = 10;
```

あえてこういった間違った命名をしようとする人はいないでしょうが、すでにあるコードを変更したときに、名前を変え忘れると名前と中身が乖離することはよくあります。

### 言語的正確性 (linguistic accuracy)

名前が **自然言語として正しい** かどうか。自然言語というのは英語や日本語のこと。
意味的正確性が「指示対象と意味が合っているか」だったのに対し、こちらは語そのものの形式が正しいかを見ます。プログラムにおいては多くの場合英語が使われるため、主に英語の話で、大きく 2 つの面に分かれます。

#### 綴り（spelling）

語が英単語として正しく綴られているか。

```ts:事例: スペルミス
// 🪫 低: スペルミス
savedSuccessfly = true;

// 🔋 高: 正しいスペル
savedSuccessfully = true;
```

```ts:事例: 存在しない単語
// 🪫 低: regist という単語は存在しない
//        "register" が動詞+er に見えることで勘違いする人が多い
function regist() {}
// 🔋 高: register という動詞が存在する
function register() {}
```

このようなスペルミスは Code Spell Checker（VSCode 拡張）/ CSpell（CLI ツール）で検出できます。

https://streetsidesoftware.com/vscode-spell-checker/

https://cspell.org/

#### 文法（grammar）

英語の文法上正しい表現を使っているかどうか。
日本人（英語を母語としない人）、特に英語が苦手な人は間違えがちです。

```ts:事例: is に縛られた命名 その1
// 🪫 低: ブール値ならば is、を意識すぎて英語がおかしい
// 「ボタンを表示するかどうか」を表したい
isShowButton = true;
// 🔋 高
// 「するかどうか」なので should
shouldShowButton = true;
// is で始めるなら、受動態
isButtonShown = true;
// もしくは動詞
showButton = true;
```

```ts:事例: is に縛られた命名 その2
// 🪫 低: ブール値ならば is、を意識すぎて英語がおかしい
// 「編集可能かどうか」を表したい
isCanEdit = true;
// 🔋 高
// 「できるかどうか」なので can
canEdit = true;
// is で始めるなら、-able で表現
isEditable = true;
```

<!-- 逆に、**データ（名詞）なのに動詞句**で名付けると関数に見える。送信予定のメッセージを `sendMessage` とすると「メッセージを送る動作」に読めてしまう。データなら名詞句で `messageToSend`（あるいは `outgoingMessage`）とする。`userName`（関数なのに名詞）とちょうど逆向きの取り違えで、英語を母語としない開発者がやりがちだ。 -->

```ts:事例: する予定の〇〇
// 🪫 低: 値なのに動詞句で、関数に見える
// 「このあとの処理で送信するメッセージのデータ」を表したいのに、
// 「メッセージを送る関数」に読めてしまう
sendMessage = { to: "a@mail.com", body: "Hello" };
// 🔋 高: データは名詞句で
messageToSend = { to: "a@mail.com", body: "Hello" };
```

### 明確性 (clarity)

名前だけで **意図・使い方** が分かるか。

- 具体性があること。抽象的な名前は、読み手が「何をするのか」を推測する余地を残す。
- 「どのように（実装）」ではなく「何のために（目的）」で名付ける

実装の詳細（型やデータ構造）を名前に埋め込まないこと、呼び出す側が欲しいもので名付けること（クライアント視点）も、すべてここに含まれる。

```ts:事例: 実装でなく目的で名付ける
// 🪫 低: 抽象すぎる
getDate();
// 🪫 低: 「3か月後を取得」= 実装の説明
getThreeMonthsLater();
// 🔋 高: 「有効期限を取得」= 目的
getExpiredDate();
```

**事例: 単位や型を名前で補う**

名前だけで使い方が分かれば、ドキュメントを読み返さずに済む。数値なら単位を名前に足すとよい。`duration` だけでは秒かミリ秒か分からないが、`durationMs` なら一目で伝わる。

```ts:事例: 単位を名前で補う
// 🪫 低: 単位が分からず、呼び出し側が取り違える
setTimeout(fn, duration);
// 🔋 高: 単位を名前に込める（ドキュメント不要）
setTimeout(fn, durationMs);
```

```ts:事例: 状態・時制を名前に映す
// 🪫 低: load だけでは進行中か完了済みか曖昧
load = false;
// 🔋 高: 時制・相で状態を言い分ける
isLoading = false; // 読み込み中か
isLoaded = false; // 読み込みが完了したか
```

**事例: 機械的な名前でなく中身を語る名前にする**

連番のように **中身を語らない名前** も明確性が低い。`img1`/`img2` は区別こそつくが「どの画像か」は分からない。意味で名付ければ、名前だけで中身が分かる（共通の接頭辞を残せば補完や検索でも探しやすい）。

意味のある名前にしておくと、AI に何も言わずに任せてもファイル名から推察して適切な画像を使ってくれるといった場合も多くなる。

```ts:事例: 機械的な名前でなく中身を語る名前にする
// 🪫 低: 連番。中身が分からない
fileNames = ["img1.png", "img2.png"];
// 🔋 高: 中身を名前で語る
fileNames = ["userIcon.png", "productImage.png"];
```

#### 事例：使うときの注意（副作用・リスク）を名前で伝える

また、**使うときの注意（副作用・リスク）も名前で伝える**と、誤用を未然に防げる。React の `dangerouslySetInnerHTML` は、名前にあえて「危険」と書くことで「サニタイズせず使うと XSS になる」と警告し、軽率な使用を思いとどまらせる。`unsafe`・`UNSAFE_`・`unstable_` なども同類だ。この“警告的命名”は、後述の利用時の品質「安全性」に直結する。

```ts:事例: 危険性を名前で伝える
// 🪫 低: XSS の危険性が名前からは分からない
setHTML={{ __html: '<script>alert("XSS!")</script>' }}
// 🔋 高: 名前に危険性を明示（React に実際に存在する名前）
dangerouslySetInnerHTML={{ __html: '<script>alert("XSS!")</script>' }}
```

```ts:事例: 危険性を名前で伝える その2
// 🪫 低: 安定した機能であるように見える
import { LinkControl } from "@wordpress/block-editor";
// 🔋 高: 名前に実験的な機能であることを示す
// WordPress のブロックエディタのコンポーネントで実際に存在した名前
import { __experimentalLinkControl } from "@wordpress/block-editor";
```

### 平易性 (simplicity)

**簡単に読める名前** であり、**速く正しく** 読めるか。

- 文字数が短いこと（目で認識しやすい）
- 平易な語であること（辞書を引かずに理解できる）
- 読み上げられること（会話やレビューで困るため）
- 理解できる言語（これは英語や日本語などの自然言語のこと）で表現されていること

```ts:事例: 長過ぎる変数名
// 🪫 低: 長すぎて読みにくい
// 🔋 高: 短くして読みにくさを減らす
```

```ts:事例: 平易な語でない名前
// 🪫 低: 辞書を引かないと分からない
// 🔋 高: 平易な語で表現する
```

```ts:事例: 口に出して読みにくい名前
// 🪫 低: 省略形で読めない・発音できない
const ymdhms = "2024-06-22 03:03:52";
// 🔋 高
const timestamp = "2024-06-22 03:03:52";
```

```ts:事例: 略語を使わない
// 🪫 低
btn
bnr
ttl // （Ttl は total? title?）
```

```ts:事例: 区切りがわかりにくい名前
// 🪫 低: 区切りがわかりにくい
contenteditable
// 🔋 高: 区切りが明確
content_editable
```

:::message

`contenteditable` は実際に存在する HTML のグローバル属性のひとつです。
https://developer.mozilla.org/ja/docs/Web/HTML/Reference/Global_attributes/contenteditable
HTML の属性名の仕様上、区切り文字もなくすべて小文字で連結されています。仕様上しかたないのですが、読みにくい。
初見で「contented/itable」と読んでしまい、content の過去形ってなんやねんと思ってしまいました。
個人的読みにくいワードランキングの上位です

:::

### 選択容易性 (selectability)

名前を **一語としてまとめて扱える** か。ダブルクリックでの選択や、`Alt+←`／`Ctrl+←` のような単語単位のカーソル移動は、**区切り文字で単語の境界が決まります**。ハイフン `-` はそこが切れ目とみなされるため、`my-value` はダブルクリックすると `my` か `value` の片方しか選べない。`my_value`（スネークケース）や `myValue`（キャメルケース）なら、一語として選択・移動できます。

ピンポイントな観点ではありますが、編集時の操作性や利用時の利便性に影響するため、意外と重要です。

```css:事例: 一語として扱えるか
/* 🪫 低: - が単語区切りになり、ダブルクリックで全体を選べない */
.my-component
/* 🔋 高: _ やキャメルケースなら一語として選択・移動できる */
.my_component
.MyComponent
```

以前書いた CSS の命名設計に関する記事では、この観点を考慮に入れて `BlockName__ElementName` のような形式の命名を採用していました。

https://zenn.dev/kagan/articles/css-brain-bemding

#### コラム: 主キーの選択

命名とは少しそれますが、DB の ID の採番を検討するときにこの観点を意識することもあります。ID の文字列が長すぎると、コピー＆ペーストで一部しか選択されず、意図しないレコードを参照してしまうことがあります。

```ts
// 🪫 低: ダブルクリックですべて選択できない
"e99f2011-1bf0-4b34-9888-6bd00afb9233"; // uuid
// 🔋 高: ダブルクリックでまとめて選択できる
"01ARZ3NDEKTSV4RRFFQ69G5FAV"; // ulid
"tz4a98xxat96iws9zmbrgj3a"; // cuid2
```

## 内部相対品質 (internal-relative quality)

自立品質ではその名前単体で評価できる性質を見ましたが、内部相対品質・外部相対品質では、**他の名前と比べて**評価される性質を見ていきます。
同じ関数の中の他の変数名、同じファイルの中の他の関数名、同じリポジトリ内の他のファイル名……など、**あるスコープの中の相対的な関係性**で評価される品質です。

### 内部識別性 (internal identifiability)

コードの中で「これは何か」を正しく見分けられるか。これはさらに 2 つの側面に分けられます。

- **一貫性**（同じものは同じ名前）
- **弁別性**（違うものは違う名前）

#### 一貫性

同じ概念には同じ名前を使う。分析→設計→実装の層をまたいでも語を保つ。さらに、**比喩・語彙の体系を混ぜない**（郵便の比喩で組んだなら `stream`/`flush` を持ち込まない）。

##### 同じ概念には同じ名前

```ts:事例: 同じ概念を指しているのに異なる単語で表現している
// 🪫 低: 同じ「ユーザー」の英訳がバラバラ
userId
newCustomer
MemberList
createAccount()

// 🔋 高: 同じ概念は同じ単語で揃える
userId
newUser
UserList
createUser()
```

```ts:事例: 同じ概念を指しているのに異なる単語で表現している
// 🪫 低: 同じ「手数料」の英訳が fee / charge でバラバラ
getFee();
calculateCharge();

// 🔋 高: 同じ概念は同じ単語で揃える
getFee();
calculateFee();
```

:::message

この「一貫した識別」という発想は、WCAG の達成基準 3.2.4「[一貫した識別性](https://waic.jp/translations/WCAG22/Understanding/consistent-identification.html)」とも通じる概念です。
WCAG はウェブコンテンツアクセシビリティガイドライン。すなわち UI に登場する要素のアイコンや文言の話ですが、コードの命名においてもユーザー（＝開発者・AI）にとってのわかりやすさとして捉えられます。

:::

##### 並列な存在には似た名前

```ts:事例: 並列な存在には並列の命名規則を使う
// 🪫 低
目黒
五反田
大崎
品川
高輪ゲートウェイ

// 🔋 高: 並列な存在には並列の命名規則を使う
目黒
五反田
大崎
品川
高輪
```

山手線新駅の名称公募の結果の 1 位は「高輪」だったそうです。
わざわざ「高輪ゲートウェイ」にしたのにも意図はあるのでしょうが、他の駅名と一貫性のない名前（異質な名前）であることは違いないでしょう。

https://news.yahoo.co.jp/expert/articles/921ff3a6299a6c991464b029000c3f7e9c108947

もう 1 つ、**名前の構造を要素どうしの関係に合わせる**のも一貫性だ。従属（包含）するものは従属と分かる名前に、並列なものは並列の命名規則で揃える。`List` / `ListItem` は「Item は List の子」と読めてよい。一方 `Button` / `ButtonSpecial` は、ButtonSpecial が Button の子のように見えて誤解を招く（実際は並列なバリアント）。並列なら対等に揃える。

```tsx
// 🪫 低: ButtonSpecial が Button の子要素に見える（実際は並列なバリアント）
Button;
ButtonSpecial;
// 🔋 高: 並列なものは対等な命名規則で揃える
ButtonCommon;
ButtonSpecial;
// 🔋 高: 従属（包含）は従属と分かる名前で
List;
ListItem;
```

#### 弁別性

- **弁別**：紛らわしい別物を区別する（`date` → `startDate`）。プロジェクト内の他のシンボルと衝突させない。

連番や似すぎた名前より、**互いに紛れない名前** のほうが弁別性は高い。取り違えを防ぎたいときは、あえて遠い名前を選ぶ。

```ts:事例: 似すぎた名前
// 🪫 低: 似すぎて紛れる
date1 = "2024-01-01";
date2 = "2024-02-01";
// 🔋 高: 遠い名前にして紛れないようにする
startDate = "2024-01-01";
endDate = "2024-02-01";
```

無線で使う NATO のフォネティックアルファベット（Alpha, Bravo, Charlie…）も、`B`・`D`・`P` などを聞き間違えないよう紛れない語に置き換える同じ発想だ。

**事例: トールマン表記（字形で差を立てる）**

プログラミングの文脈ではない事例ですが、識別性の良い事例として紹介します。
医療現場には、字面が似た薬剤名を **部分的に大文字にして** 取り違えを防ぐ「トールマン表記 (Tall man lettering)」という手法がある。差のある部分を視覚的に際立たせるやり方で、紛れやすい識別子ではコードでも同じ発想が効く。

```ts
// 🪫 低: 似た識別子が一覧で紛れる
prednisone;
prednisolone;
// 🔋 高: 差のある部分を大文字で立てる（トールマン表記）
predniSONE;
predniSOLONE;
```

糖尿病薬「アマリール」と似ていた高血圧症薬「アルマール」の名前を「アロチノロール」に変更したそうです。

### スタイル一貫性 (style consistency)

形式・記法の流儀を、**自分のコード内で揃えている** か。
例えば、「ディレクトリ名は `kebab-case`」「変数名は `camelCase`」「イベントハンドラは `handle*`」などなど。どんな流儀を選ぶかは自由でも、コードベース全体で一貫させる。外部標準のない独自ルールでも、一貫してさえいれば読み手はそこから手がかりを得られる。

内部識別性が **語彙** で見分ける話だったのに対し、こちらは **形式** を揃える話です（意味系と形式系の対をなしています）。たとえば定数を `UPPER_SNAKE` に統一すると、読み手は形だけで「これは定数だ」と察することができます。

**事例: カテゴリごとに記法を揃える**

```ts
// 🪫 低: 定数・コンポーネント・真偽値の記法がバラバラ
const maxCount = 100;
const max_retry = 3;
function userCard() {}
const enabled_flag = true;
// 🔋 高: カテゴリごとに記法を一貫させる
const MAX_COUNT = 100;
const MAX_RETRY = 3;
function UserCard() {}
const isEnabled = true;
```

:::message

ここではコード内での一貫の話であり、エコシステムでの慣習に従う観点は後述の外部相対品質「スタイル適合性」で扱います。

:::

### 適量性 (proportionality)

スコープに対して情報量が **過不足ない** か。

- **簡潔性**: 多すぎを削る
- **十分性**: 足りなければ足す

の双方向で、どちらに倒すかは **スコープが裁定** する。この 2 方向にそれぞれ名前をつけておく。

#### 簡潔性（減らす方向）

スコープが既に与えている情報は繰り返さない。狭いスコープなら短くてよい。

```ts
// 🪫 低: user スコープ内なのに user を繰り返す（冗長）
user = {
  userName: "Alice",
  userId: 123,
};
// 🔋 高: user の話であることはわかりきっているので、省略する
user = {
  name: "Alice",
  id: 123,
};
```

:::message

ここでの簡潔さは _スコープに照らした_ 冗長の排除であり、語そのものの平易さ（平易性）とは別だ。`userName` 単体は難しくないが、`user.` の中では冗長、という相対の判断になる。

:::

```ts:事例: ループ変数の名前
// 🪫 低: 単なるループ変数に長い名前を与えている
for (let userArrayIndex = 0; userArrayIndex < users.length; userArrayIndex++) {
  const user = users[userArrayIndex];
}
// 🔋 高: 狭いスコープなら短く
for (let i = 0; i < users.length; i++) {
  const user = users[i];
}
```

#### 十分性（増やす方向）

逆に、広い・公開スコープでは、名前だけで意味が取れるよう具体を **足す**。スコープが文脈を与えてくれない分、名前が背負う。

```ts
// 🪫 低: モジュールから公開する名前が漠然としすぎ
export const data = fetchAll();

// 🔋 高: 広いスコープには十分な具体を
export const activeUsers = fetchAll();
```

### 内部検索性 (greppability)

ツールの上での話。
grep（ソースコード内での検索）で一意に当たるか。

**"greppability"** は英単語としては存在しない造語ですが、すでにウェブ上ではこの単語を使っているエンジニアが多く、検索性の観点を表す言葉として定着しつつあります。

```ts:事例: 一意に grep できる名前に
// 🪫 低: grep すると大量にヒットして埋もれる
// user.ts
export const KEY = "KEY_FOR_USER";
// admin.ts
export const KEY = "KEY_FOR_ADMIN";

// 🔋 高: 一意に検索でき、補完でも絞り込める
export const KEY_FOR_USER = "KEY_FOR_USER";
export const KEY_FOR_ADMIN = "KEY_FOR_ADMIN";
```

静的解析で見つけることもできるが。

なお **弁別性が高いほど内部検索性も上がる**（紛れない名前は grep でも一意に当たる ── `id` は埋もれ、`KEY_FOR_USER` は当たる）。この「区別できる → 見つかる」という関係は外の世界でも同じで、そちらは後述の **外部検索性** で扱う。

```
const userId = user.id;
```

**事例: grep できる定型マーカー**

コメントをどんな文言で書くかも命名の 1 つです。
コメントの目印は、自由文よりも `TODO:` / `FIXME:` / `HACK:` のような **決まった語に揃える** と、後から横断検索・集計できます。

```ts
// 🪫 低: 表記がバラバラで拾えない
// 〇〇をあとで直す
// そのうち△△をやる
// 要修正！

// 🔋 高: 定型マーカーに揃えて grep できる
// TODO: バリデーションを追加する
// FIXME: 境界値で破綻する
```

#### コラム: 検索性を下げないために

命名とは別の観点にはなりますが、リポジトリ内での検索性を高めるには文字列連結による処理をできるだけ避けるのが望ましいです。

```tsx
// ファイルの名前を連結して生成する
const iconType = "close";
return <img src={`/icon_${iconType}.svg`} alt="" />;
```

このコードが実行されると `icon_close.svg` ファイルが参照されることになりますが、`icon_close` という名前でコード内を検索しても引っかかりません。
このファイルを使っていないと判断して間違って削除してしまうかもしれません。
未来の誰かが探したときにたどり着けるよう、ファイル名全体を文字列で持っておくようにするといいでしょう。

```ts
// 🔋 高: ファイル名全体を文字列で持つことで検索性を高める
const iconType = "close";
switch (iconType) {
  case "close":
    return <img src="icon_close.svg" alt="" />;
  case ...:
    ...
}
```

### 整列性 (sortability)

主にファイル名の命名で考慮する観点です。
一覧で名前順（アルファベット順）に並べられたときに、有用な並び順になるか。

- 連番が数字の大小でソートされるか
- 関連する名前が固まるか
- 意味のある順番に並ぶか
- 特別なものが先頭や末尾に来るか（`_` で始めるなど）

#### 連番が数字の大小でソートされるか

```ts:事例: 連番が数字の大小でソートされる名前
// 🪫 低: 数字の順に並ばない
'image-1.png'
'image-10.png'
'image-2.png'
...

// 🔋 高: ゼロ埋めすることで数字の順に並ぶ
'image-01.png'
'image-02.png'
...
'image-10.png'
```

とは書いたものの、最近では数字列を数値の大小でソートしてくれるシステムが多くなっています。
（VSCode のエクスプローラー、Windows のエクスプローラー、macOS の Finder では image-01.png, image-02.png, image-10.png の順で並ぶことを確認しました）
純粋な辞書順ソートがされる環境では、ゼロ埋めをしないと意図した順序に並ばない。

#### 関連する名前が固まるか

```ts
// 🪫 低: 関連する名前が離れてしまう
PrimaryButton;
SecondaryButton;

// 🔋 高: 関連する名前が固まる
ButtonPrimary;
ButtonSecondary;
```

関連するファイルが近くに並べば、

- 片方に加えた修正をもう片方にも適用することがある場合、見つけやすくなる
- 利用する人（開発者）が、似たファイルを見つけやすくなる
  - たとえば、`ButtonPrimary` を使おうとした人がファイルを探したとき、近くに `ButtonSecondary` があれば目に入る。実はその人が使うべきファイルは `ButtonSecondary` だった、ということもあり得る

#### 意味のある順番で並ぶか

ディレクトリ設計においては、しばしばファイル・ディレクトリのレイヤー間の依存関係にルール（方向性）がある場合があります。
フロントエンドアーキテクチャでは Feature-Sliced Design や Atomic Design などがその例です。これらの設計では、依存関係の方向が決まっており、上位レイヤーは下位レイヤーに依存することはあっても、その逆はないというルールがあります。
これを名前の順序性で表すと、依存関係の方向が捉えやすくなります。

#### 特別なものが先頭や末尾に来るか（`_` で始めるなど）

記号や数字は英字より前に並ぶことが多いので、名前の先頭に付けると一覧の上位に集まります。

たとえば Sass のパーシャル `_variables.scss` や Next.js の `_app.tsx` の `_` は、「部分ファイル」「フレームワークの特別なファイル」だと区別するためのであって整列のための仕様ではありませんが、結果として一覧でもまとまり、区別しやすくなります。

```text
styles/

  _mixins.scss
  base.scss
  layout.scss
```

### 補完容易性 (completion discoverability)

IDE 補完で絞り込めるか。共通プレフィックスで候補がまとまると探しやすい。

```ts:事例: 共通プレフィックスで補完を絞る
// 🪫 低: 補完で候補が散らばる
const theme = {
  red: "#f00",
  blue: "#00f",
  green: "#0f0",
  small: "8px",
  medium: "16px",
  large: "32px",
};

// 🔋 高: 共通プレフィックスで候補がまとまる
const theme = {
  colorRed: "#f00",
  colorBlue: "#00f",
  colorGreen: "#0f0",
  sizeSmall: "8px",
  sizeMedium: "16px",
  sizeLarge: "32px",
};
// → theme.color で補完すると色だけが、theme.size で補完するとサイズだけが候補に出る
// large だったか big だったか huge だったか忘れても、size の一種であることだけ思い出せればたどり着ける
```

### 連続性 (continuity, temporal consistency)

すでに広まった名前の変更を考える際の観点です。名前を変えないこと、もしくは変えたとしても関連性が認識できること。

公開 API やサービス名は、昔から知っている人に伝わり続けることに価値があります。これは「内部一貫性（ある時点における他のコードとの一貫性）」に対する「時間軸の一貫」といえます。

#### 事例: GitHub のデフォルトブランチ名の変更

master → main への変更は実は `ma` から始まる点でパッと見の印象が近いので、連続性を保った変更の例と勝手に思っています。
（それが理由になっているかは知りませんが）
primary や default でも意味的には近いですが、`ma` で視覚的に発見しやすく、master のつもりで入力したときに補完されることなども考えると、いい命名の変更だと感じています。

```ts:事例: GitHub のデフォルトブランチ名の変更
// 🔋 高: 連続性
master → main
```

:::message

そもそもこの名称変更が必要な背景は後述する「包摂性」の観点です。

:::

公開した名前ほど連続性は重く効く ── 後述の裁定基準 **影響範囲** が大きいからだ。HTTP ヘッダの `referer` が綴りミス（正しくは `referrer`）のまま保たれているのが典型で、いまさら直せば世界中の実装が壊れる。ここで「外部公開だから連続性が別の特性になる」わけではない点に注意したい。起きているのは複数の特性の合わせ技だ ── 過去の名前を保つ **連続性**、すでに外で定着した綴りに新規実装が合わせる **外部一貫性**、そして桁違いの **影響範囲**。1 つの命名判断が複数の特性にまたがる好例である。

**事例: Twitter → X（連続性を手放した例）**

いったん世界中に広まった名前を連続性のない名前に変更すると、既存の知識や期待が通じにくくなります。2023 年に Twitter が X に解消されましたが、この名前からは「Twitter のこと」とは連想しにくいです。
（さらに「X」は一般語すぎて Web 検索でも埋もれる（外部検索性）とか、他と紛れて固有性も低い（外部識別性）とか、色々問題はあるわけですが…。）
その結果、「X（旧 Twitter）」という表記をみんながする必要が発生しました。

```ts:事例: Twitter → X（連続性を手放した例）
// 🪫 低: 連続性がない名称変更
Twitter → X
```

## 外部相対品質 (external-relative quality)

内部相対品質が「コードの中の他の名前との関係」で決まる品質だったのに対し、この外部相対品質は、コードベースや開発チームの**外**にある、**規約・慣習・社会との一致**で決まる品質です。

### 外部識別性 (external identifiability)

コミュニティの慣習に照らして、「これは何か」が正しく見分けやすいかどうか。内部識別性と同じく **一貫＋弁別** の対となる 2 つの側面で評価します。

- **一貫性**：ドメイン用語・ユビキタス言語、世間一般の概念名、標準的な略語（`id`/`url`）、確立したパターン語彙（Factory／Validator）、慣習的な対義語ペア（`open`/`close`、`get`/`set`）に合わせる。語が世間で持つ含意にも逆らわない（`getSize()` が重い処理だと、`get`＝軽い、という期待を裏切る）。
- **弁別性**：組み込み・標準ライブラリのシャドーイングを避け、商標や既存サービスとも紛れないようにする。

**事例: 語の含意・対義語の慣習に合わせる**

```ts
// 🪫 低: get は軽い処理を期待させるのに重い（世間の含意に反する）
user.getRecommendations(); // 内部で API 通信＋重い計算
// 🔋 高: 取得元・重さを示す
user.fetchRecommendations();
```

```ts
// 🪫 低: open の慣習的な相方は close（hide は別ペア show/hide のもの）
openModal();
hideModal();
// 🔋 高: 確立した対義語ペアで揃える
openModal();
closeModal();
```

慣習と一貫させていることで、新しく入ってきたエンジニアでも、その慣習さえ知っていればすぐに馴染むことができる。
AI も同じで、世の中のコードを学習しているので、慣習に沿った名前のほうが理解してもらいやすい。

### 外部検索性 (googlability、ググラビリティ、エゴサビリティ)

コードの外（Web 検索や SNS）で見つかるか。特に **公開パッケージ名・リポジトリ名・公開 API・サービス名** で考慮すべき観点。
内部検索性が grep（自分のリポジトリ）の中での話であったのに対し、こちらは **Web という空間** の話しです。

- 一意にヒットすること
- その機能を求めている人が検索したときにヒットすること

例えばライブラリを公開したとき、使ってほしい人に見つけてもらうためには、検索でヒットする名前にする必要があります。
もちろん、必ずしもライブラリ名を説明的にせずとも見つけてもらう方法はありますが、

一般的すぎる語は埋もれて検索にかからない。プログラミング言語の Go は名前が一般語すぎて探しにくく、`golang` という別名で検索される。逆に固有性の高い造語（`TypeScript`・`Kubernetes`）は一意に見つかる。**外部弁別性（外で紛れない）が高いほど外部検索性も上がる** ──「区別 → 発見」の関係は内と同じだ。

**事例: 一般語は埋もれる**

公開するライブラリ・パッケージ名では、一般語だと検索で埋もれ、固有性の高い造語だと一意にたどり着ける。

```text
TypeScript / Kubernetes
```

### スタイル適合性 (style conformance)

使用している言語のコミュニティ、フレームワークのエコシステムにおける **形式・スタイルの流儀・慣習** に合うか。いわゆる _idiomatic_（その言語・フレームワークで「自然」とされる書き方）に沿うこと。
前述のスタイル一貫性が「自分のコード内で形式を揃える」話だったのに対し、こちらはその **揃え先を世間の流儀にする** 話です。

#### React における例

- イベントハンドラの変数名は `handleClick` (`handle〇〇`)
- イベントハンドラの props は `onClick` (`on〇〇`)

> 慣習的に、イベントハンドラは handle の後ろにイベントの名称をつなげた名前にすることが一般的です。

> 慣習として、イベントハンドラのプロップは on で始まり、次に大文字の文字が続くようにします。

https://ja.react.dev/learn/responding-to-events

```tsx:事例: ライブラリの慣習に合わせる

// 🔋 高: props は on*、変数名は handle*
<button onClick={handleClick} />;
```

## 利用時の品質：名前を使った結果

ここまでの 3 大分類は **名前そのものの品質**（ISO でいう製品品質にあたる）でした。ISO/IEC 25010 が製品品質とは別に「利用時の品質」（使った結果として現れる品質）を分けているように、命名にも **使った結果として生まれる品質** を考えてみましょう。
これらは、これまでに分類してきた複数の特性が寄与して生まれます。

<!-- ISO の利用時の品質（有効性・効率性・リスク回避性・利用状況網羅性）になぞらえた 7 つに、modifiability 由来の変更容易性を加えた 8 つ。 -->

<!-- つまり、個々の特性はバラバラに存在するのではなく、**少数のアウトカムに寄与・収束** している。 -->

### 予測容易性 (predictability)

読み手が「この名前はこういう意味だろう」と _正しく_ 当てられること。内部識別性・外部識別性・スタイル一貫性・スタイル適合性などが寄与する（定数が `UPPER_SNAKE` なら「定数だ」と当てられる、など。「誤解を招かない」はこの裏返し）。

**事例: 規約から種類を当てる**

規約が一貫していれば、初見の名前でも中身を確かめる前に種類を当てられる。逆にバラバラだと予測がきかない。

```ts
// 🪫 低: 規約がバラバラで、名前から種類を予測できない
const maxRetry = 3; // 定数？ ただの変数？
// 🔋 高: 規約が一貫し、初見でも予測できる
const MAX_RETRY = 3; // UPPER_SNAKE → 定数だろう
```

```ts:事例： 値なのに動詞
// 🪫 低: 規約がバラバラで、名前から種類を予測できない
// 値なのに動詞
const sendMessage = { to, subject }

// 🔋 高:
const messageToSend = { to, subject }
```

<!-- 有効性 -->

### 理解容易性 (understandability)

_速く・楽に_ 読み解けること。短く一貫した名前ほど速い。予測容易性が「正しく分かる」なら、こちらは「速く分かる」── 別の軸であり、簡潔さと明確さのように両者は衝突することもある。

**事例: 一目で頭に入る名前**

意味は同じでも、長く込み入った名前は読み解くのに時間がかかる。正確さを保ちつつ短くまとめると、一目で頭に入る。

```ts
// 🪫 低: 正確だが、読み解くのに時間がかかる
const theListOfUsersWhoAreCurrentlyActive = getUsers();
// 🔋 高: 一目で頭に入る
const activeUsers = getUsers();
```

<!-- 効率性 -->

### 伝達容易性 (communicability)

名前を _共有語彙_ として、チームの会話・レビューや AI への指示で、正確かつ素早く指し示せること。一語で「`getExpiredDate` を直そう」と言えるか、AI に「`data` を意味のある名前に」と的確に指示できるか。発音可能性（平易性）・明確性・予測容易性・読み手網羅性が寄与する。理解容易性が _黙読_ の速さなら、こちらは _対話・指示_ のしやすさだ ── 冒頭の「AI への指示」という動機がここに効く。

**事例: 一語で指し示せる**

名前が付いていれば、会話や AI への指示で一語で正確に指せる。名前がなければ「ユーザー一覧を絞り込んでいるあの部分」と説明が要る。

```ts
// 🪫 低: 名前がなく、指すのに説明が要る
return users.filter((u) => u.active && !u.deleted);
// 🔋 高: 名前を付ければ「activeUsers を直して」で通じる
const activeUsers = users.filter((u) => u.active && !u.deleted);
return activeUsers;
```

<!-- 効率性 -->

### 到達容易性 (reachability)

必要な人が、その名前に **たどり着ける** こと。

- 同僚が既存のユーティリティ関数を見つけて再利用する（重複実装を防ぐ）
- 使ってほしいライブラリが Web 検索で見つかって採用される

など。

内部検索性・外部検索性が _見つけやすさ_ を、明確性・予測容易性が「これが探していたものだ」という確信を与える。検索性が _見つけられる性質_ なら、到達容易性は _実際にたどり着けた結果_ だ（なお「発見可能性」はデザイン分野で「どんな操作が可能か・どう操作するかを見つけられるか」という別の意味で使われるため、ここでは _到達_ と呼ぶ）。

**事例: 既存の関数にたどり着く**

「日付を整形する関数、どこかにあった気がする」というとき、規約どおりに名付けられていれば、「`format` とか入ってるんじゃないかな～」と、補完や grep で当たりをつけてたどり着け、重複実装を避けられる。場当たりな名前だと存在に気づかず作り直してしまう。

```ts
// 🪫 低: 場当たりな名前。存在を予測できず重複実装してしまう
function toStr2(d: Date) {}
// 🔋 高: 規約どおり。`format` で補完すれば候補に出てたどり着ける
function formatDate(d: Date) {}
```

こういった探し方は AI（私が主に使っているのが Claude Code なのでそれを念頭にしていますが）もよくやっています。

ためしに、手元にあるリポジトリで「このリポジトリに日付をフォーマットする関数はありますか？」と Claude Code に調査を依頼しました。その後 Claude Code 自身に「どうやって（どんなコマンドで）探しましたか？」と聞いてみました。
すると以下のように説明してくれました。

```sh
# 1. ファイル名で当たりをつける（find + grep）
find /repo \
  -type f \( -name "*.ts" -o -name "*.tsx" -o -name "*.js" -o -name "*.jsx" \) \
  | grep -E "(util|lib|helper|format)" | head -20

# 日付フォーマットが置かれそうなファイル名（util, lib, helper, format）でファイル一覧を絞り込み。

# 2. 関数名パターンで全文検索（grep -r）
grep -r "formatDate\|dateFormat\|toDateString\|format.*Date" \
  /repo \
  --include="*.ts" --include="*.tsx" --include="*.js" --include="*.jsx" \
  | grep -v node_modules | head -30

# よくある関数名パターン（formatDate, dateFormat, toDateString, format.*Date）を OR で横断検索。

# ...以降も具体的な定義箇所の調査が続いていたが省略
```

Claude Code 自身が説明しているとおりですが、
「そういうフォーマット関数は `util` `lib` `helper` `format` といった名前のついているファイルにありがちだ」、という推測をもとにファイルを探し、
「そういうフォーマット関数には `formatDate`, `dateFormat`, `toDateString`, `format〇〇Date` といった名前がついていがちだ」という推測を元に関数定義を探していました。

このとき検証したリポジトリには実際に `toDateString` という名前の関数があり、発見に成功していました。

機能を表すような名前（明確性を持つ名前）をつけ、慣習に従った名前（外部識別性のある名前）にすることで、このように発見しやすくなります。

<!-- 有効性 -->

### 安全性 (safety)

紛らわしい名前による **誤用・バグを防げる** こと。

<!-- リスク回避性に対応 -->

- 取り違えを起こさない
- 危険の存在を知らせる

```ts:事例: 取り違えやすい名前を避ける
// 🪫 低: 1 文字違いで紛らわしく、取り違えやすい
deleteAt; // 削除予定時刻？ 削除済み時刻？
deletedAt;
// 🔋 高: 紛れない名前にして取り違えを防ぐ
scheduledDeletionAt; // 削除予定
deletedAt; // 削除済み
```

### 安定性 (stability)

将来の変更で名前を **変えずに済む** こと。意図で名付ければ実装を変えても名前は有効だし（明確性）、スコープに見合った抽象度なら成長しても破綻しない（適量性）。

なお、公開した名前を約束として変えない **連続性** とは別の概念です。連続性が「変えたくても変えない」なのに対し、安定性は「そもそも変える必要が出ない」という結果を指します。

<!-- リスク回避性に対応？ -->

```ts
// 🪫 低:
function getNameAndId(user) {
  return {
    name: user.name,
    id: user.id,
  };
}
// 追加するとしたら・・・
function getNameAndIdAndEmail(user) {
  return {
    name: user.name,
    id: user.id,
    email: user.email,
  };
}
```

#### 事例: 拡張性の低い命名

たとえばサイズ段階に名前を付けるとき、`largest` のような最上級の語を選ぶと、より大きい段階を増やしたくなったときに名前が嘘になり、改名の必要が出てきてしまう。`s`/`m`/`l` のような相対スケールにしておけば、あとから `xl` や `2xl` を足すだけで済み、既存の名前は変えずに済む。適量性で _拡張余地_ を残しておくほど、安定性は上がる（[開放/閉鎖原則 (open/closed principle)](https://ja.wikipedia.org/wiki/%E9%96%8B%E6%94%BE/%E9%96%89%E9%8E%96%E5%8E%9F%E5%89%87) を命名で行うようなイメージ）。

<!-- 要確認：これは開放閉鎖と言って良いのか -->

```ts
// 🪫 低:
sizes = {
  large: "30px",
  larger: "40px",
  largest: "50px",
};
// すでに最上級の名前を使ってしまったが、
// "60px" を追加したいときどうする？
// 既存の名前を変更する？

// 🔋 高:
sizes = {
  lg: "30px",
  xl: "40px",
  "2xl": "50px",
  "3xl": "60px",
};
```

### 変更容易性 (modifiability)

いざ名前を変えるとき、**楽かつ安全に直せる** こと。安定性が「変えずに済む」を最大化するのに対し、こちらは「変えるときのコストを下げる」。
一意でよく弁別された名前は、grep で全使用箇所に漏れなく当たり（内部検索性）、似た名前と紛れず誤爆しない（弁別性）ので、リネーム（一括置換）が安全・確実になる。逆に `data` や `id` のような頻出語は、置換しようとすると無関係な箇所まで巻き込む。

<!-- リスク回避性＋効率性。modifiability は ISO だと保守性（製品側）だが、ここでは利用時の効果として借用 -->

**事例: 一括置換できる名前**

```ts
// 🪫 低: 頻出語。grep が無関係な箇所まで拾い、安全に置換できない
const data = fetchUsers();
// 🔋 高: 一意な名前。grep で全箇所に当たり、安全にリネームできる
const activeUsers = fetchUsers();
```

### 読み手網羅性 (reader coverage)

**ベテランでも新人でも、人間でもAIでも、誰が読んでも通じる** こと。

**事例: 内輪の略語を避ける**

チーム内では通じる独自の略語も、新しく入った人・非母語話者・AI には通じないことがある。読み手の幅を広げたいなら、場当たりな省略より普通の語を選ぶ。

```ts
// 🪫 低: 内輪でしか通じない省略（Ttl は total? title?）
const usrAddr = "";
const calcTtl = () => {};
// 🔋 高: 誰が読んでも分かる
const userAddress = "";
const calculateTotal = () => {};
```

## トレードオフとの向き合い方

ここまで分類してきた命名品質特性は、**全部を同時には満たせません**。なぜなら、特性間にトレードオフがあるから。最終的には優先順位を決める必要があります。

判断の基準は **影響範囲（スコープ）**。ローカル変数のリネームはいくらでもできます。一方で、公開 API・URL・DB カラム・JSON キーなどは変更が難しくなります。影響範囲が広い名前ほど、意味的正確性・一貫性・明確性の側に倒す必要があります。

```ts
d; // 簡潔だが意味が薄い
daysSinceLastLogin; // 明確だが長い
```

どちらが正解かは単体では決められません。その名前が使用される **スコープによって決まります**。3 行のループ内なら `d` で十分ですし、公開 API の引数なら `daysSinceLastLogin` のような明確さがあったほうが望ましいです。

「全特性を満たす名前」ではなく **「その文脈で優先すべき特性を選んだ名前」が良い** ということです。

### 事例: 平易性 vs 明確性

前述の `d` と `daysSinceLastLogin` で見たように、
最も基本的な衝突は、**平易性（簡潔さ・平易さ）vs 明確性** です。

平易性 vs 明確性
狭いスコープなら短い名前で問題ありません。

```ts
for (let i = 0; i < 9; i++) {
  console.log(i);
}

for (let index = 0; index < 9; index++) {
  console.log(index);
}

try {
  ...
} catch (e) {
  console.error(e);
}

try {
  ...
} catch (error) {
  console.error(error);
}
```

このトレードオフはすでに紹介した **適量性** によって測る基準です。

### 事例: 「アンケート」（平易性 vs 言語的正確性）

たとえば、「アンケート」機能があるようなときの命名について。日本では「アンケート」として馴染みがあるが、これはもともとフランス語。`enquete` のような変数名を書くと Code Spell Checker ではスペルミス扱いされる（英語辞書で設定していれば）。英語では **"questionnaire"** と言います。
だからといって questionnaire が絶対的な正解かというとそういうことでもなく、日本語母語話者（「アンケート」で認識している人）が中心の開発チームであれば、直感的にアンケートのことだとわかりやすい enquete を使うのも十分選択肢になるでしょう。

```ts:事例: 「アンケート」の命名
// 英語における「言語的正確性」を取るならこちら
questionnaire;
// 日本人にとっての「平易性」を取るならこちら
enquete;
// より日本人にとっての「平易性」を優先し、
// 「スタイル適合性」（変数名はアルファベットで書くという慣習）を捨てるならこちら
アンケート
```

### 事例: `referer` （言語的正確性 vs 連続性）

HTTP ヘッダの `referer` は、本来英単語として正しい綴りは `referrer`（r が足される）です。
これは仕様策定時にスペルミスをしてしまった結果、それが今でも使われているためです。
いったん広まった綴りを変えないために **連続性**（後方互換）を優先し、**言語的正確性** を捨てている（捨てざるを得ない）例です。

https://zenn.dev/motch/articles/567a456b42cc4f

### 事例: `dangerouslySetInnerHTML` （明確性・安全性 vs 平易性）

React の `dangerouslySetInnerHTML` は、シンプルに書けば `setHtml` で済むところを、あえて長く物々しくして XSS（クロスサイトスクリプティング）の危険があることを **明確** に伝える名前で、**安全性**（誤用への警告）を最優先した名前になっています。

## おわりに：命名は設計へのフィードバック

漠然としていた「良い命名」の観点を分類して見てきました。
分類し、その特性にも「命名」したことでそれぞれの概念を認識できるようになります。

この「名前が付くと（名前を知ると）その存在を認識できる」という現象にもまた名前が付いており、ジョシュアツリーの法則と呼ばれています。『[ノンデザイナーズ・デザインブック](https://book.mynavi.jp/nddb/)』のイントロで触れられている話です。

https://objectclub.jp/technicaldoc/object-orientation/principle/principle04

**名前は、対象を認識・理解することにつながります**。だから命名にこだわることは、コードを読みやすくするだけでなく、設計そのものを良くすることにつながる。

最後に、分類から少し離れた話をひとつ。

**命名が難しいとき、それは設計を見直すべきとき** かもしれません。
`DoStuff` のような名前しか浮かばないのは、その対象が多くの責務を抱えていて、一語で言い表せないからだ。名前を付けようとする行為が、「ここは分けるべきだ」「これは別のものとまとめるべきだ」「この知識はチーム内で共有すべき概念だ」といった設計上の気づきを与えてくれます。

同様に、「これ、名付けるべきでは？」と思うことも重要です。コード中に存在するマジックナンバーはどんな役割を持っているでしょうか？　何らかの意味・役割があってその数字を指定しているのではないでしょうか。その概念に名前をつけましょう。
