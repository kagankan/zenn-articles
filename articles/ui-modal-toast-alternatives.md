## 今日の目標

**STOP！「なんとなくモーダル」✋ STOP！「なんとなくトースト」✋**

Web アプリケーションの UI を検討する際に、他の手段を検討せずにモーダルやトーストを使ってしまっていることはないでしょうか。今回はこれらの使い分けについて考えます。
※使うな、というわけではなく、他の手段も一度考えてみよう、ということです。

## 主な出典

『[Web アプリケーションアクセシビリティ](https://gihyo.jp/book/2023/978-4-297-13366-5)』（技術評論社）より、第 5 章「複雑な UI パターンの改善」の最初の 2 項目であるモーダルダイアログと通知（トースト）を取り上げます。モーダルについては第 8 章にも記載があるのでそこの内容も含まれます。
（筆者による話もします。）

なお、書籍の中ではキーボード操作に関する問題やスクリーンリーダー使用時の問題も触れられていますが、Chakra のコンポーネントを使っていればその点は基本的に考慮されているので、今回はこれらの話はしません。

## 用語の確認

### 「モーダル」

「モーダル」というのは、実はモノの名前ではなくて、**状態の名前**。

操作をしないと進めない状態（モード）に入るのがモーダル。
それに対して、操作が必須でないのはモードレス。

いつもモーダルと呼んでいるのは、正確には **「モーダルダイアログ」**。

### 「トースト」

画面の上部や隅に一時的に表示される通知。

アクションボタンを含む場合はスナックバーと呼ばれたりする。

https://note.com/kanna_life/n/nd04fbc07d6b2

### 「インライン表示」

そこまで多く使われる表現ではないと思うのですが、今後の説明で使うので定義しておきます。

モーダルダイアログやトーストのように、他の要素の上に乗っかって（オーバーレイで）表示される表示方法に対して、**テキストを挿入するような形で表示される表示を「インライン表示」** と表現します。

## 💬 モーダルダイアログの特徴＆問題点

### 特徴

- ページの遷移を必要としない
- 開いている間は、モーダルダイアログしか操作できない

### 「モーダルダイアログの濫用」

『Web アプリケーションアクセシビリティ』の中では、必然性のない場面でもモーダルダイアログを使ってしまう（濫用されている）ことを指摘しています。

> モーダルダイアログは、現在の操作から別のモードに入り、ダイアログ以外の箇所の操作を受け付けないようにするものです。ユーザーを特定の操作に集中させることができる一方、デメリットも多いため、**「モーダルダイアログでなければならない」という必然性があるケースのみで使うべきです**。
> しかし、実際の使われ方を見ると **「1 画面にさまざまな状態を盛り込めるもの」「文脈を維持したまま新しいキャンバスが作れる便利なもの」として濫用される傾向にあります**。

（『Web アプリケーションアクセシビリティ』8.3（466 ページ）より。太字は筆者による）

### デメリット

- ダイアログが URL を持たない場合、開いた状態を再度訪れることができない。ブラウザの戻る・進むが使えない
- ダイアログに対応（フォーム入力や YES/NO の回答）するために背景側の情報が必要な場合、一度閉じてから背景側の情報を記憶し、再度開く必要がある。
- ダイアログの面積が狭かったり、ダイアログ内でスクロールが必要になったりすると、操作がやりにくくなる
  - スマホではスクロールバーが表示されないケースもあり、ダイアログ内でスクロールが発生していること自体に気付けず、下の方にあるボタンを押せない、といったケースがある

### ダイアログの濫用パターン ❌

1. 情報表示のみのダイアログ
2. 編集モードのダイアログ
3. 追加入力のダイアログ
    - 初回ログイン時に出現する、設定を入力させるダイアログ
    - アクション実行前にオプションの入力を要求してくるダイアログ
    - 重要情報へのアクセス時や決済ルートの途中で、ログインやパスワード再入力、CAPTCHA の入力を求めてくるダイアログ
4. 確認ダイアログ（例：「本当に実行しますか？」）
5. 結果報告ダイアログ（例：「登録が完了しました）
6. エラーダイアログ（例：「登録に失敗しました。氏名が未入力です」）

（『Web アプリケーションアクセシビリティ』467 ページ、484 ページより）

### 解決策 ✅

#### 1. 情報表示のみのダイアログ

- インライン表示する
- （スペースを取るのが気になる場合）アコーディオンで隠す
- 情報が多ければ独立したページにする

#### 2. 編集モードのダイアログ

- そもそも編集ボタンを置かず、現在の内容が入った入力フォームが並んでいる詳細画面にしてしまう
    - 例えば、Notion のページはこの形式
- 編集ボタンを押すと、詳細画面がフォームに切り替わる
    - GitHub の例。プルリクエストのタイトル編集では、「Edit」ボタンをクリックするとフォームに切り替わる
        ![GitHub のプルリクエストのタイトル部分](/images/ui-modal-toast-alternatives/2025-11-17-02-10-09.png)
        ![GitHub のプルリクエストのタイトル編集画面（フォームに切り替わった後）](/images/ui-modal-toast-alternatives/2025-11-17-02-10-43.png)
- 編集ボタンを押すと、フォームが挿入される
- 編集ボタンを押すと、編集ページに遷移する

#### 3. 追加入力のダイアログ

- 初回ログイン時の設定は、独立した画面を用意する。
- 実行前オプションは、実行ボタンの近くにインライン表示する。

#### 4. 確認ダイアログ

- チェックボックスで意思を確認する
- （GitHub の例）リポジトリのアーカイブ化を実行する際、リポジトリ名を入力させることで意思を確認する  
  ![GitHub のリポジトリのアーカイブ化ダイアログ](/images/ui-modal-toast-alternatives/2025-11-17-02-11-38.png)


#### 5. 結果報告ダイアログ

- 一覧画面で項目の追加をしたら、その一覧画面を表示すれば追加されていることがわかる
- 一覧画面で項目の削除をしたら、その一覧画面を表示すれば削除されていることがわかる
- それだけでは伝わりにくい場合、結果報告をインラインで表示すればよい

#### 6. エラーダイアログ

- インライン表示にする

## 🍞 トースト通知の特徴＆問題点

先程紹介したモーダルダイアログの使い方のうち、「結果報告ダイアログ」や「エラーダイアログ」の代替として使われることもあるため、トーストについても取り上げます。

### 特徴

- 一定時間で消える
- 画面の端に表示される
- ユーザーの操作を止めない

操作が不要なことから、本の中では「**控えめな**通知」と表現されている。

### 問題点

- 画面の端（＝操作を行っていたのとは別の場所）に表示されるため、気づかないことがある
- 一定時間で消えると、テキストが読み切れないことがある

### トーストの使い方

- そもそもできるだけ使わない
- 使うとしたら、以下に当てはまるようなケースのみ
    - 気づかずに消えても問題ない（必須要件）
    - ユーザーの操作とは無関係なタイミングで表示したい
    - 他の操作を邪魔したくない

### 濫用パターン ❌

- ユーザー登録後、「認証メールを送信しました」
- メール認証時、「メール認証に失敗しました」
- フォーム入力で「エラーが発生しました」

→その後の操作が必要なので、気づいてほしい。

### 改善例 ✅

- インラインで表示する
  - エラーメッセージを、入力フォームのすぐ下に表示する例
    ![「おすすめ度を選択してください」というテキストが、選択UIのすぐ下に表示されている](/images/ui-modal-toast-alternatives/2025-11-17-02-13-14.png)    
  - エラーメッセージを、フォームの上部に表示する例
    ![WordPressのログインフォーム。エラーメッセージがフォームの上部に表示されている](/images/ui-modal-toast-alternatives/2025-11-17-02-15-12.png)

- 専用の表示（ページ）を用意する
  - 処理に失敗したことと、対応方法を伝えるページの例
    ![認証に失敗したことを伝えるページ](/images/ui-modal-toast-alternatives/2025-11-17-02-15-48.png)

## 簡易フローチャート

上記の情報を元に、どんな UI を使うべきか判断するための手順を筆者がざっくりと整理してみたものです。
（じっくり検討したわけじゃないので参考程度に）

### 何かをお知らせするUIを作る場合

1. まずは**インライン表示**できないか考える
2. 量が多いなど、スペースを取るのが気になる場合は**アコーディオン**にする
3. 情報が多いなら**単独のページ**にする
4. 気づかなくてもいい・邪魔したくないなら**トースト**
5. なんとしても気づいてもらう必要があれば**モーダルダイアログ**

### 何かを操作してほしいUIを作る場合

1. まずは**専用のページ**を作ることを考える
2. ページを作れない・ページを遷移させたくない場合、表示を**フォームに置き換え**ることを考える
3. もしくは、**フォームを挿入**する。
4. それが難しい場合、**モーダルダイアログ**
