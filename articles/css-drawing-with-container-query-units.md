CSS の **コンテナクエリ単位**（`cqw`、`cqh` など）、使ったことありますか？
存在を知らなかった人、もしくは知っているけど使いどころに出会わなくて使ったことない人も多いと思います。

この記事では、コンテナクエリ単位の便利さを実感する例として、**CSSお絵かき**を題材に紹介します。
普通のレイアウトでは見えづらいコンテナクエリ単位の嬉しさが、CSSお絵かきの文脈ではくっきり浮き彫りになります。

## CSSお絵かきとは？

HTML と CSS だけでイラストを描く遊びです。
画像も SVG も使わず、`background-color` で色を塗り、 `border-radius` で形を変え、その他色々なプロパティを駆使して絵を描きます。

CSS 好きの変態がやる遊びではあると思うのですが、意外とやっている人が多い領域です。
普段の開発とは違ったテクニックが要求される特殊さゆえに、普段使わない CSS の機能に触れる面白い分野でもあります。
普段 CSS を触る人であればやってみてテクニックを学ぶのは非常に有意義です。

## CSSお絵かき特有の悩み

普通の Web レイアウトと違い、CSSお絵かきには「絵」ならではの制約があります。

- **縦横比を保ったまま拡縮したい**: アスペクト比が崩れたら絵として成立しない
- **キャンバス内の全要素がキャンバスの大きさに連動してほしい**: 絵を大きく描いてもパーツのバランスは崩れてはいけない

つまり、**「キャンバスのサイズが変わっても、すべてのパーツが比例して拡縮する」絵**を作りたいわけです。
これを実現するために、CSSお絵かき勢は時代ごとに違うテクニックを駆使してきました。

実は、CSS お絵かきにとって、使用できるテクニックは次のように進化してきました。

1. **`padding-top` ハック時代**: 縦横比を保ちたければ `padding` の特性を逆手に取るしかなかった
2. **`aspect-ratio` 時代**: 縦横比を宣言的に書けるようになった
3. **コンテナクエリ単位時代**: キャンバス基準で全要素を直接記述できるようになった

それぞれの時代で何が辛くて、次の機能で何が解決されたのか、CSSお絵かきというレンズで見ていきます。


## 時代 1: `padding-top` ハックの暗黒時代

`aspect-ratio` がなかった時代、要素の縦横比を保ったまま横幅基準で拡縮させるのはなかなか難しい問題でした。
そこで定番として使われていたのが、**`padding-top` ハック**です。

### 仕組み

CSS の `padding` 系プロパティのパーセント値は、**親要素の横幅**を基準に計算されます。
これは縦方向の `padding-top`/`padding-bottom` でも変わりません（ここがポイント）。

この性質を逆手に取り、要素に `height: 0` を指定して `padding-top` を縦横比のパーセントで指定すると、縦横比を保った要素が作れます。

非常口のキャンバス部分は、こう書けます。

```css
.exit-sign {
  width: 100%;
  height: 0;
  padding-top: 100%; /* 1:1 の正方形 */
}
```

横幅可変で 1:1 の正方形キャンバスが実現できます。
ハック感の強い書き方ですが、これで初めて「親要素の横幅基準で縦横比を保つ」ことができたのです。

### 子要素のサイズ指定の痛み

そして、キャンバス内のパーツは `position: absolute` でパーセント指定されます。
例えば走る人の手までの構造を見てみましょう。

```css
/* キャンバス */
.exit-sign {
  width: 100%;
  height: 0;
  padding-top: 100%;
}

/* 走る人 */
.person {
  position: absolute;
  top: 10%;
  left: 25%;
  width: 50%; /* 親（キャンバス）の幅の 50% */
  height: 80%; /* 親（キャンバス）の高さの 80% */
}

/* 前方の腕 */
.arm-front {
  position: absolute;
  top: 30%;
  left: 60%;
  width: 35%; /* 親（person）の幅の 35% */
  height: 45%; /* 親（person）の高さの 45% */
}

/* 手 */
.hand-front {
  position: absolute;
  top: 80%;
  left: 30%;
  width: 50%; /* 親（arm-front）の幅の 50% */
  height: 25%; /* 親（arm-front）の高さの 25% */
}
```

ここで「手はキャンバス全体に対して何%の大きさなのか？」を計算してみましょう。

- 横幅: `50% × 35% × 50% ≒ 8.75%`
- 縦幅: `80% × 45% × 25% = 9%`

…と、複数階層のパーセントを掛け算しないと求まりません。

絵を描くときに「キャンバスの中で手をだいたいこの位置にこのくらいの大きさで」と感覚的に決めたいのに、実装するときには「直近の親要素の何%」に変換し直さないといけない。
親要素のサイズを変えたくなったら、その中の子要素全部を再計算しないといけない。

これが、CSSお絵かきの暗黒時代でした。

## 時代 2: `aspect-ratio` という光明

CSS の [`aspect-ratio` プロパティ](https://developer.mozilla.org/ja/docs/Web/CSS/aspect-ratio)が登場し、縦横比を宣言的に指定できるようになりました。

```css
.exit-sign {
  width: 100%;
  aspect-ratio: 1 / 1;
}
```

`height: 0; padding-top: 100%;` という不可思議な 2 行が、`aspect-ratio: 1 / 1;` の 1 行で済みます。
比率がそのままコードに表れていて、何をしているか一目瞭然ですね。

### 解決したこと

- 縦横比の指定が宣言的になった（コードを見れば意図が明らか）
- `padding-top` ハックの不可解さから解放された

### 残った問題

ただし、子要素のサイズ指定は変わっていません。
依然として「親要素基準のパーセント」で書くしかなく、複数階層のパーセント掛け算からは逃れられませんでした。

```css
.person {
  width: 50%;
  height: 80%;
}
.arm-front {
  width: 35%;
  height: 45%;
}
.hand-front {
  width: 50%;
  height: 25%;
}
```

「キャンバス全体に対する手のサイズ」を直接書きたい、という願いは叶わぬままでした。

## 時代 3: コンテナクエリ単位の到来

そして現在。**コンテナクエリ単位**の登場により、状況は劇的に変わります。

### コンテナクエリ単位とは

要素のサイズを「指定したコンテナの何%か」で表現する単位です。
パーセント（`%`）が「直近の親要素」基準だったのに対し、コンテナクエリ単位は **「明示的に指定したコンテナ要素」基準**になります。

| 単位    | 意味                            |
| ------- | ------------------------------- |
| `cqw`   | コンテナの幅の 1%               |
| `cqh`   | コンテナの高さの 1%             |
| `cqi`   | コンテナのインライン軸の 1%     |
| `cqb`   | コンテナのブロック軸の 1%       |
| `cqmin` | `cqi` と `cqb` のうち小さい方   |
| `cqmax` | `cqi` と `cqb` のうち大きい方   |

コンテナにしたい要素には `container-type` プロパティを設定します。

```css
.canvas {
  container-type: size;
}
```

### CSSお絵かきでは `cqw` + `container-type: size` がベスト

コンテナクエリ単位を使うとき、実は単位の選び方には複数の流派があります。

| 方針 | container-type | 単位  | 特徴                                   |
| ---- | -------------- | ----- | -------------------------------------- |
| A    | `inline-size`  | `cqi` | writing-mode に追従する論理単位        |
| B    | `size`         | `cqw` | writing-mode に依存しない物理単位      |

CSSお絵かきの文脈では、**B（`cqw` + `container-type: size`）が適切**です。
理由は次のとおり。

絵は `<img>` のような視覚オブジェクトであり、周囲のテキストが縦書きになったからといって絵そのものが回転するべきではありません。
つまり、絵の座標系は **writing-mode から独立した物理軸** であるべきです。

`cqw` は物理 width を基準にする単位なので、writing-mode が変わっても挙動が変わりません。
ただし `cqw` を縦書きでも正しく機能させるためには、`container-type: size` でブロック軸とインライン軸の両方をコンテインメントする必要があります（`inline-size` だと縦書き時にインライン軸が縦になるため、`cqw` がコンテナを参照できなくなります）。

:::message
通常のレイアウト用途（テキストコンポーネントのレスポンシブなど）では、writing-mode に追従する方が嬉しいので `cqi` + `inline-size` のほうが筋が良いケースが多いです。**「絵か文章か」で選び方が変わる**のが面白いところです。
:::

### なぜ `cqw` だけで書くのか（`cqh` を使わない理由）

`cqh` も使えるのに、なぜ `cqw` だけで書くのか。
これは、**単位を 1 種類に統一したい**からです。

CSSお絵かきは「キャンバス座標上のドット絵を描いている」感覚に近いです。
ドット絵を描くとき、横方向の 1 ドットと縦方向の 1 ドットの大きさが違ったら描きにくいですよね。
`cqw` と `cqh` を混ぜると、キャンバスのアスペクト比に応じて「1 単位の長さ」が方向によって変わってしまうので、座標系を一貫させたい絵描きには不向きです。

`aspect-ratio` でキャンバスの縦横比を固定している前提なら、横幅から縦も比例的に決まるので、`cqw` だけで縦方向のサイズも記述できます。

### 子要素を `cqw` で書き換える

キャンバスを次のように設定します。

```css
.exit-sign {
  width: 100%;
  aspect-ratio: 1 / 1;
  container-type: size; /* ← これを足すだけ */
}
```

そして、子要素のサイズを `cqw` で記述します。

```css
.person {
  position: absolute;
  top: 10cqw;
  left: 25cqw;
  width: 50cqw;
  height: 80cqw;
}

.arm-front {
  position: absolute;
  top: 24cqw;
  left: 30cqw;
  width: 17.5cqw;
  height: 36cqw;
}

.hand-front {
  position: absolute;
  top: 29cqw;
  left: 5cqw;
  width: 9cqw;
  height: 9cqw;
}
```

ここでのポイントは、**全ての値が「キャンバスの何%か」で統一されている**ことです。
`.hand-front` の `width: 9cqw` は、ネストの深さに関係なく「キャンバス幅の 9%」を意味します。
親要素（`.arm-front`）の大きさが何であれ、影響を受けません。

時代 1・時代 2 では `width: 50%` と書いて「実際にはキャンバスの 8.75% だなあ」と頭の中で計算していたところを、**最初からキャンバス基準で `9cqw` と書ける**のです。

:::message
位置の指定（`top`/`left`）も `cqw` で書けます。値の単位がキャンバス基準で統一されるので「1cqw 横にずらしたい」という直感的な操作ができますね。ただし `position: absolute` の起点は依然として親要素なので、「キャンバスから直接の絶対位置」を指定したい場合は、要素を直接キャンバス直下に配置する必要があります。
:::

### 嬉しいこと

- **パーセント複合からの解放**: 親要素のサイズに依存しないので、サイズの計算が不要になる
- **親要素のサイズ変更に強い**: 親要素のサイズを変えても、子要素の見た目のサイズは変わらない
- **ボーダー幅やシャドウ距離も `cqw` で指定できる**: 線の太さを `em` や `px` で指定するとキャンバスサイズに連動しません（`em` はフォントサイズ依存、`px` は固定値）。`cqw` ならボーダー幅まで含めて全部キャンバス相対にでき、本当の意味でスケール不変な絵になります

## 3 つの時代を並べてみる

同じ「手までのスタイル」を 3 つの時代で並べてみます。

### 時代 1: `padding-top` ハック

```css
.exit-sign {
  width: 100%;
  height: 0;
  padding-top: 100%;
}
.person {
  width: 50%;
  height: 80%;
}
.arm-front {
  width: 35%;
  height: 45%;
}
.hand-front {
  width: 50%;
  height: 25%;
}
```

### 時代 2: `aspect-ratio`

```css
.exit-sign {
  width: 100%;
  aspect-ratio: 1 / 1;
}
.person {
  width: 50%;
  height: 80%;
}
.arm-front {
  width: 35%;
  height: 45%;
}
.hand-front {
  width: 50%;
  height: 25%;
}
```

### 時代 3: コンテナクエリ単位

```css
.exit-sign {
  width: 100%;
  aspect-ratio: 1 / 1;
  container-type: size;
}
.person {
  width: 50cqw;
  height: 80cqw;
}
.arm-front {
  width: 17.5cqw;
  height: 36cqw;
}
.hand-front {
  width: 9cqw;
  height: 9cqw;
}
```

時代 1 から時代 2 では「キャンバスの定義」が改善されました。
時代 2 から時代 3 では「子要素の指定」がキャンバス相対になり、ネストの深さに依存しなくなりました。

進化しているのは確かに別の部分なのですが、いずれも CSSお絵かき視点で見ると **「キャンバス基準で考えやすくなった」** という同じ方向の改善になっています。

## CSSお絵かきしない人にも届けたい話

CSSお絵かきという特殊な題材で見てきましたが、コンテナクエリ単位は普通のレイアウトでも便利です。

例えば、カードコンポーネントの中で「カード自体のサイズに連動して文字や余白を変えたい」というケースはよくあります。
`vw` を使うと画面幅基準になってしまいますが、`cqi` を使えばコンポーネント自身のサイズ基準にできます。

```css
.card {
  container-type: inline-size;
}
.card-title {
  font-size: 5cqi; /* カードの幅に応じて文字サイズが変わる */
  padding: 2cqi 4cqi;
}
```

カードを大きく表示すれば文字も大きく、小さく表示すれば文字も小さく、自動的に追従します。
画面幅ではなく **コンポーネント自身の表示サイズ** でレスポンシブできるので、配置場所を選ばないコンポーネントが作れますね。

## まとめ

CSSお絵かきというレンズで、CSS の歴史を振り返ってみました。

| 時代                     | キャンバスの定義     | 子要素の指定                 |
| ------------------------ | -------------------- | ---------------------------- |
| `padding-top` ハック     | わかりにくい         | 親基準のパーセント           |
| `aspect-ratio`           | 宣言的に書ける       | 親基準のパーセント           |
| コンテナクエリ単位       | 宣言的に書ける       | コンテナ基準で直接書ける     |

コンテナクエリ単位（`cqw`、`cqh`、`cqi`、`cqb`）、まだ使ったことがない人はぜひ試してみてください。
CSSお絵かきしなくても、コンポーネント単位のレスポンシブで便利さを実感できると思います。

CSSフリークがしょうもないことに全力を出すと、こんな発見があるという例でした。
