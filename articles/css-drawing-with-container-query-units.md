CSS の **コンテナクエリ単位**（`cqw`、`cqh` など）、使ったことありますか？
存在を知らなかった人、もしくは知っているけど使いどころに出会わなくて使ったことない人も多いと思います。

この記事では、コンテナクエリ単位の便利さを実感する例として、**CSSお絵かき**を題材に紹介します。
普通のレイアウトでは見えづらいコンテナクエリ単位の嬉しさが、CSS お絵かきでは意外と活躍するんです。

## CSSお絵かきとは

HTML と CSS だけでイラストを描く遊びです。
画像も SVG も使わず、`background-color` で色を塗り、 `border-radius` で形を変え、その他色々なプロパティを駆使して絵を描きます。

CSS 好き ~~の変態~~ がやる遊びではあると思うのですが、意外とやっている人は多いように思います。
普段の開発とは違ったテクニックが要求される特殊さゆえに、普段使わない CSS の機能に触れる面白い分野でもあります。
普段 CSS を触る人であれば、やってみてテクニックを学ぶのは非常に有意義です。

## CSSお絵かき特有の要求

普通の Web レイアウトと違い、CSS お絵かきには「イラスト」を表現するからこその要求があります。

- **縦横比を保ちたい**
- **親要素の大きさに連動して拡縮させたい**（レスポンシブ対応）
  - 画像（`img` 要素）のように、横幅 (`width`) を大きくしたらそれに連動してイラスト全体が拡大してほしい
- **パーツをレイヤー管理したい**
  - すべてのパーツをフラットに配置してもいいのですが、「顔を全体的に右に移動させたい」などの操作が発生することを考えると、HTML の階層構造を利用して、パーツをグループ化して管理しておきたい

これらを実現しようと思うと、CSS お絵かきの実装で欲しくなるのは「**あるコンテナの大きさ（横幅）を基準に、あらゆる寸法を書く**」という統一表記です。コンテナのサイズさえ決まれば、縦も子要素の位置・サイズも全部それに連動して決まる、という座標系が理想ですね。

この「横幅基準の統一表記」を実現する方法は、CSS の進化とともに変遷してきました。

1. **`padding-top` ハック**: 親の横幅で縦寸法を書く最初の方法（ハッキーで、要素自身に閉じている）
2. **`aspect-ratio`**: 同じことが宣言的に書けるようになった（依然として要素自身に閉じている）
3. **コンテナクエリ単位（`cqw` など）**: ネストを越えて、任意の祖先の横幅を全要素から参照できるようになった

それぞれを見ていきましょう。

## `padding-top` ハックの時代

`aspect-ratio` がなかった時代、要素の縦寸法を横幅で表現するのはなかなか難しい問題でした。
そこで定番として使われていたのが、**`padding-top` ハック**です。

### 仕組み

`padding` 系プロパティのパーセント値は、**親要素の横幅**（縦幅ではなく）を基準に計算されます。
これは縦方向の `padding-top`/`padding-bottom` でも同じく、横幅が基準になります。

この性質を利用し、以下のように `padding-top` を縦横比のパーセントで指定すると、縦横比を保った要素が作れます。
ハック感の強い書き方ですが、これで「**親の横幅を基準に縦寸法を表現する**」ことができます。

```css:aspect-ratioがない時代の縦横比固定
.square {
  width: 100%;
  height: 0;
  padding-top: 100%; /* 1:1 の正方形 */
}

.small-square {
  width: 50%;
  height: 0;
  padding-top: 50%; /* 1:1 の正方形で、親の半分の大きさ */
}
```

### 問題: 直属の親基準にしかできない

ただし、この技法でできるのはあくまで「**自分自身**の縦寸法を、**自分の親の横幅基準**で書く」ことだけです。子要素の中のさらに子要素から、祖先（キャンバス）の横幅を参照することはできません。
まあ、すべてのパーツを親の直下にフラットに配置してもいいのですが、冒頭にも描いた通り、パーツをレイヤー分けして管理したいと思うと、ネスト構造が必要になってきます。

たとえば `.canvas > .person > .face > .eye` という階層構造で顔のパーツを管理しているとき、`.eye` のサイズを「キャンバスの何%か」で書きたいと思っても、`width: 10%` と書くと「**親要素（.face）基準の10%**」になってしまいます。

これでも、結局は祖先要素の横幅に連動して拡縮してくれるので、実用上は問題ないとも言えます。
ですが、ネスト構造を変えた場合にサイズが変わってしまうのはメンテナンス性が悪く、あまり望ましくありません。

## `aspect-ratio` の登場

CSS の [`aspect-ratio` プロパティ](https://developer.mozilla.org/ja/docs/Web/CSS/aspect-ratio)が登場し、縦横比を宣言的に指定できるようになりました。
（時期としては、Chrome にサポートされたのが 2021 年 1 月、Safari が 2021 年 9 月です）

```css
.square {
  width: 100%;
  aspect-ratio: 1 / 1;
}
```

`padding-top: x%;` という意図の伝わりにくかったハックから、「1:1 にしたい」と伝わる宣言になりました。
比率がそのままコードに表れていて、何をしているか一目瞭然ですね。

### 依然として要素自身に閉じている

ただし、できることの範囲は `padding-top` のハックと変わっていません。`aspect-ratio` も「**自分自身**の縦横比を宣言する」プロパティであり、`width` は親依存の横幅指定ですから、子孫要素から祖先の横幅を参照できるようにはなりません。

子要素のサイズ指定は依然として「親要素基準のパーセント」のままで、「キャンバス全体に対するサイズ」を直接記述することはできません。

## コンテナクエリ単位という強力な武器

そして現在。**コンテナクエリ単位**の登場により、状況は変わります。
「**ネストを越えて、任意の祖先の横幅を全子孫から参照できる**」ようになりました。これによって、お絵かきの全寸法を 1 つの単位（`cqw`）で統一表記できるようになります。

（時期としては、2022 年 9 月に Chrome 105, Safari 16 が対応し、2023 年 2 月に Firefox 110 が対応。現在では Baseline: Widely available となっています）

### コンテナクエリ単位とは

要素のサイズを「指定したコンテナの何%か」で表現する単位です。
パーセント（`%`）が「直属の親要素」基準だったのに対し、コンテナクエリ単位は **「明示的に指定したコンテナ要素」基準**になります。ここが本質的に新しい点で、**親子関係のネストを越えて祖先の寸法を参照できる**仕組みです。

| 単位    | 意味                          |
| ------- | ----------------------------- |
| `cqw`   | コンテナの幅の 1%             |
| `cqh`   | コンテナの高さの 1%           |
| `cqi`   | コンテナのインライン軸の 1%   |
| `cqb`   | コンテナのブロック軸の 1%     |
| `cqmin` | `cqi` と `cqb` のうち小さい方 |
| `cqmax` | `cqi` と `cqb` のうち大きい方 |

https://developer.mozilla.org/ja/docs/Web/CSS/Guides/Containment/Container_queries#%E3%82%B3%E3%83%B3%E3%83%86%E3%83%8A%E3%83%BC%E3%82%AF%E3%82%A8%E3%83%AA%E3%83%BC%E3%81%AE%E9%95%B7%E3%81%95%E5%8D%98%E4%BD%8D

### `vw` ではダメなのか

「幅基準にしたいなら `vw`（ビューポート幅の 1%）でもよくね？」と思うかもしれません。
たしかに、絵を **画面全幅で表示する前提** であれば、`vw` でも横幅基準の統一表記は実現できます。
しかし、せっかくなら `<img>` 要素と同じような使い勝手で、ページ内に絵を埋め込んでも成立するような実装したいですよね。（よね？）
となると、「**親要素のサイズに連動させる**」必要があり、それには、`vw` では対応できないわけです。

### `cqi` （論理単位）ではダメなのか

コンテナクエリ単位を使うとき、実は単位の選び方には複数の選択肢があります。

| 方針 | container-type | 単位  | 特徴                              |
| ---- | -------------- | ----- | --------------------------------- |
| A    | `inline-size`  | `cqi` | writing-mode に追従する論理単位   |
| B    | `size`         | `cqw` | writing-mode に依存しない物理単位 |

横書きの世界のみを考慮するならどちらもあまり変わりませんが、縦書きまで考慮する場合は挙動が変わります。
そのうえで、CSS お絵かきの文脈では **B（物理単位）** を選択します。
理由は次のとおり。

絵は `<img>` のような視覚オブジェクトであり、周囲のテキストが縦書きになったからといって絵そのものが回転するべきではありません。
つまり、絵の座標系は **writing-mode から独立した物理軸** であるべきです。

`cqw` は物理 width を基準にする単位なので、writing-mode が変わっても挙動が変わりません。
ただし `cqw` を縦書きでも正しく機能させるためには、`container-type: size` でブロック軸とインライン軸の両方をコンテナサイズとして使用できるようにする必要があります（`inline-size` だと縦書き時にインライン軸が縦になるため、`cqw` がコンテナを参照できなくなります）。

:::message

正直、お遊びでお絵かきするだけなので、縦書きなんて考慮する必要はないんですけどね。
一方で、お遊びだからこそ、細かいところまでこだわって実装してみたいですよね。

:::

### `cqh` は使わないのか

直感的には、縦方向のサイズを指定するときには `cqh` を使ったほうが良い気がしてしまいますが、それはしません。
これは、**単位を 1 種類に統一したい**からです。

例えばドット絵を描くとき、横方向の 1 ドットと縦方向の 1 ドットの大きさが違ったら描きにくいですよね。縦の 10 ドットと横の 10 ドットの長さが違ってしまうと、直感的に絵を描くのが難しくなります。
`cqw` と `cqh` を混ぜてしまうと、キャンバスのアスペクト比に応じて「1 単位の長さ」が方向によって変わってしまいます。
どちらの方向も、同じ数値が同じ見た目の長さを表すように、単位は統一して使用します。

### `cqw` を使ってみる

コンテナクエリ単位を使用するときは、基準（＝コンテナ）としたい要素（お絵かきでいうところのキャンバス）に `container-type` プロパティを設定します。

```diff css
 .canvas {
   width: 100%;
   aspect-ratio: 1 / 1;
+  container-type: size; /* ← これを足すだけ */
 }
```

そして、子要素のサイズを `cqw` で記述します。

```css
.head {
  position: absolute;
  top: 14cqw;
  left: 34cqw;
  width: 14cqw;
  height: 14cqw;
  border-radius: 50%;
  background-color: var(--color-green);
}

.door {
  position: absolute;
  top: 6cqw;
  left: 19cqw;
  right: 20cqw;
  bottom: 8cqw;
  background-color: var(--color-white);
}

/* ... */
```

:::message

位置の指定（`top`/`left`）も `cqw` で書けます。値の単位がキャンバス基準で統一されるので「1cqw 横にずらしたい」という直感的な操作ができますね。
ただし `position: absolute` の起点は依然として親要素なので、「キャンバスから直接の絶対位置」を指定したい場合は、要素を直接キャンバス直下に配置する必要があります。

:::

ここでのポイントは、**全ての値が「キャンバスの何%か」で統一できる**ことです。
`10cqw` はネストの深さに関係なく「キャンバス幅の 10%」を意味します。

冒頭の非常口のピクトグラムの実装を再掲します。ほぼ全ての寸法が `cqw` で書かれているのがわかると思います。
`.wrapper` で大きさを制限しているので、この要素の `max-width` を変えてみると、全体がそれに合わせて拡縮するのもわかると思います。

## CSSお絵かき以外でも、普通に便利なコンテナクエリ単位

ここまで CSS お絵かきというニッチな題材で見てきましたが、コンテナクエリ単位は普通のレイアウトでも便利です。

例えば、カードコンポーネントの中で「カード自体のサイズに連動して文字や余白を変えたい」というケースはよくあります。
`vw` を使うと画面幅基準になってしまいますが、`cqi` を使えばコンポーネント自身のサイズ基準にできます。
（※ここでは書字方向に依存させるのが適切であるため、`cqi` + `inline-size` の組み合わせを想定しています）

```css
.card {
  container-type: inline-size;
}
.card-title {
  font-size: 5cqi; /* カードの幅に応じて文字サイズが変わる */
  padding: 2cqi 4cqi;
}
```

カードを大きく表示すれば文字も大きく、小さく表示すれば文字も小さく、自動的に追従します。
画面幅ではなく **コンポーネント自身の表示サイズ** でレスポンシブできるので、配置方法を選ばないコンポーネントが作れますね。

## まとめ

CSS お絵かきという視点で、CSS の歴史、そしてコンテナクエリ単位の特徴を紹介してきました。
CSS お絵かきに限らずとも、コンポーネント単位でレスポンシブを意識した実装をするときに便利さを実感できると思います。
