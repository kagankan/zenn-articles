## ガイドラインがあるのは知っているけれど。

Web サイトや Web アプリケーションを作る上で、誰もが使えるようにするためにアクセシビリティを高めることは重要です。
その指針として、[WCAG（Web Content Accessibility Guidelines）](https://waic.jp/translations/WCAG22/)のようなガイドラインが整備されています。

ただ、これらのガイドラインは「達成基準」という**ルール単位**で整理されています。「キーボードで操作できること」「状態が支援技術に伝わること」……といった具合です。
チェック項目としては包括的かつ網羅的で頼りになるのですが、いざ実際に手を動かして UI を組もうとすると、こんな壁にぶつかります。

**今から作るこのコンポーネント / 機能では、どの達成基準を気にすればいいんだ？** 🤔

WCAG 2.2 では達成基準の数は 86 もあり、そこまで詳しくない開発者にとっては、今作っている機能を作るときに気にするべき基準を探すこと、それが実際考慮すべきものなのか判断することはなかなか難しいです。

実装するときには**作っているものを起点に**知りたいのです。

## この本は「逆引き（コンポーネント・機能起点）」でまとめます

そこでこの本では、達成基準から考えるのではなく、**コンポーネント・機能から**考慮すべき点を確認できるように構成しました。「アコーディオンを作る」「タブを作る」といった目的から入り、そこで考慮すべきアクセシビリティの観点を、理由・確認方法とセットで一覧にしています。

:::message

そのコンポーネントに関わる主要な考慮事項のみをまとめるつもりであり、完全に網羅することはできません。
また、「色の使用」など、特定の機能というよりはすべての機能で全般的に影響するような観点は掲載しません。

厳密なアクセシビリティチェックが必要な場合は、やはり WCAG の達成基準を参照して検証を行ってください。

:::

## ビジュアルデザインも諦めない

もうひとつ、この本が大事にしているテーマがあります。

かっこいい・かわいい装飾で目を引くデザインにもしたくても、アクセシビリティを考慮すると、想定したビジュアルデザインやスタイリングの実現が難しくなることがあります。そのため「アクセシビリティ」と「ビジュアルデザイン」は、しばしば対立する要素として語られがちです。
しかし、この課題は技術によって解決できる問題だと考えています。

この本では、アクセシビリティを担保しつつ、特定のビジュアル表現を実装する方法を紹介していきます。

## ライブラリ選定の観点としても

UI コンポーネントについては、自前で実装するだけでなく、公開されているライブラリを使用して実装することも一般的です。UI ライブラリを導入するため、複数のライブラリを比較検討する際にアクセシビリティを考慮するならどんな観点を比較すればよいか知ることで、判断の助けになれば嬉しいです。

## （改めて）この本の目的

- 自分が作ろうとしているコンポーネントに、どのアクセシビリティ観点が関わるのかを逆引きで把握する
- コンポーネントを実装する際、アクセシビリティを担保したうえで特定のビジュアル表現を実現する方法を知る
- UI ライブラリを選定する際の観点（コンポーネントのアクセシビリティやスタイリングの柔軟性など）を理解する

## 記事中で示すコードについての補足

- コンポーネントごとに HTML, CSS, JavaScript の実装例を示します
- React や Vue 等のフレームワークを使用しない、プレーンな実装を想定します。
- CSS コードでは、 [CSS Nesting](https://developer.mozilla.org/ja/docs/Web/CSS/Guides/Nesting) を使用します。
  - Baseline 2023 であり、2026 年 6 月現在では Widely Available（広く利用可能）に切り替わろうとするくらいに時間が経っています。ごく古いブラウザでは動作しないことがありますが、今後の標準的な書き方になっていくと判断して使用します。
- JavaScript コードは TypeScript で記述します
  - ただし、TypeScript 独自の記法は最低限とし、できるだけそのまま JavaScript として動くようにしています。
  - プレーン JS で使う場合は、型定義を削除して使用してください
