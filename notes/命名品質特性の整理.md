# 命名品質特性 — 設計ノート

> 記事「命名品質特性」の設計ノート。記事本体は別ファイルに書き起こす。
> 残す目的: 達成事例の一覧／分類の構造／**各名前を採った理由・没にした理由・検討した選択肢**。
> 元ネタ: https://zenn.dev/kagan/scraps/c5df40eeeda99f

## 達成事例集（命名の具体例・手法）

個々の *達成事例・手法*（こうすると良い／これを避ける、の具体）の集まり。WCAG の達成方法（Techniques）に相当し、後段の「分類（命名品質特性）」がこれらを束ねる。**番号順のフラットなリスト**（末尾が最新＝次の採番番号がすぐ分かる）。網羅性は問わない（後付けで増える）。各行は **#番号 → 絵文字 → 名前** の順で、絵文字がカテゴリ（番号が先頭なので VSCode の行ソートで番号順を保てる。分類未定なら絵文字は後付けでよい）。

凡例: 🎯 意味・正確さ ／ 🔗 区別・一貫性 ／ 💡 読みやすさ・分かりやすさ ／ 🔤 文法・形態 ／ ✍️ 表記・形式 ／ 🔭 スコープ・抽象度 ／ 🔍 発見性・整列・入力 ／ 🌐 文脈・ドメイン・人 ／ ⚙️ メタ観点

- #1 🎯 **正確性**: 名前が指示対象（実体）と意味的に合致しているか（型と名前の矛盾を避ける。綴りの正しさは言語的正確性へ）
- #2 🎯 **誤解を招かない（disinformation を避ける）**: `accountList` が実際は List 型でない等。「情報がない名前」より「嘘をつく名前」の方が有害
- #3 🎯 **冗長語の排除（noise words）**: スコープが既に与える情報を繰り返さない（`user.userName` → `user.name`、`ProductData`／`nameString`）
- #4 🎯 **実装ではなく意図を示す**:「どのように」でなく「何のために」（`getThreeMonthsLater` → `getExpiredDate`）。`And` 連結を避け抽象表現を使う
- #5 🔗 **弁別性**: 似たものと区別しやすいか（`date` → `startDate`）
- #6 🔗 **一貫性**: 同じ概念には同じ名前。リポジトリ全体で統一。世間一般の概念名に合わせる
- #7 🔗 **対義語・対称ペアの統一**: `get/set`・`open/close`・`begin/end`・`min/max`・`source/destination`
- #8 🔗 **視覚的な紛らわしさ**: `l`/`1`/`I`、`O`/`0`、`rn`/`m` など字形レベルの誤読を避ける
- #9 💡 **直感性**: 直感的に理解できるか。できるだけ短く、平易な用語（辞書を引かなくても分かる）
- #10 💡 **ドキュメント不要な命名**: 名前だけで使い方が分かる（`duration` → `durationMs` で単位を明示）
- #11 💡 **発音可能性**: 会話・レビューで口に出せるか（`genymdhms` は議論時に困る）
- #12 🔤 **品詞の一貫性**: 関数・メソッドは動詞句（`getName`・`isValid`）、変数・クラスは名詞句
- #13 🔤 **単数 / 複数**: コレクションは複数形（`users`）、単体は単数（`user`）
- #14 🔤 **真偽値の接頭辞規約**: `is` / `has` / `can` / `should`
- #15 🔤 **否定形・二重否定を避ける**: `isNotEnabled`・`disabled = false` を避け肯定形を基準にする
- #16 🔤 **時制・ライフサイクルの表現**: `isLoading`／`isLoaded`、`willUpdate`／`didUpdate`、`onClick`（受け口）／`handleClick`（実装）
- #17 ✍️ **文字種・形式**: `-` と `_` の使い分け、ファイル名は小文字推奨、React コンポーネントはアッパーキャメル
- #18 ✍️ **略語・頭字語の方針**: 略すかどうか、頭字語のケース統一（`userId`/`userID`）。標準略語（`id`・`url`）は可、場当たり略語は不可
- #19 ✍️ **予約語・組み込みとの衝突回避**: `list`・`type`・`id`（Python）や標準ライブラリ名のシャドーイングを避ける
- #20 🔭 **具体・抽象の度合い**: スコープが狭いなら抽象的でも可、広いなら具体的に
- #21 🔭 **将来の拡張性とのバランス**: 狭すぎるとリネームが必要、広すぎると意味が薄れる
- #22 🔍 **検索しやすさ（greppability / googlability）**: grep で見つかるか、Web 検索での発見性
- #23 🔍 **整列しやすさ（sortability）**: 一覧で関連が固まるか（`PrimaryButton` → `ButtonPrimary`、`image-01` の数値パディング）
- #24 🌐 **ドメイン用語／ユビキタス言語**: 業務・問題領域の語彙、ドメインエキスパートの言い回しに合わせる（DDD）
- #25 🌐 **対象読者・チームの言語**: 英語に統一するか、非母語話者に通じるか、ローマ字を許すか
- #26 🌐 **包摂的命名**: `master/slave` → `primary/replica`、`whitelist/blacklist` → `allowlist/denylist`
- #27 🌐 **フレームワークの規約**: React hooks は `use` から始める、等
- #28 ⚙️ **単独評価 vs 相対評価**: 名前単体での良し悪しと、他の命名との相対での良し悪しの両方を見る
- #29 ⚙️ **影響範囲に投資を比例させる**: ローカル変数のリネームは安いが、公開 API・URL・DB カラム・JSON キーは高コスト
- #30 ⚙️ **観点同士のトレードオフと優先順位付け**: 短さ↔正確さ、具体↔汎用などは衝突する。文脈ごとに優先順位を決める
- #31 🎯 **クライアント視点での命名（SON）**: 実装側でなく「呼び出す側が欲しいもの」で名付ける（出典: [Fujiwo](https://qiita.com/Fujiwo/items/741f7de1255b1e9aefc2)）
- #32 🎯 **実装・型をエンコードしない**: `CustomerList` の型露出やハンガリアン記法を避ける（出典: [Fujiwo](https://qiita.com/Fujiwo/items/741f7de1255b1e9aefc2)）
- #33 🎯 **責務の単一性・境界**: 何をして／何をしないかが名前で分かる（`DoStuff` 回避）（出典: [Fujiwo](https://qiita.com/Fujiwo/items/741f7de1255b1e9aefc2)）
- #34 🔗 **モデル間の垂直一貫性**: 分析→設計→実装の層をまたいでも同じ概念は同じ語で通す（出典: [Fujiwo](https://qiita.com/Fujiwo/items/741f7de1255b1e9aefc2)）
- #35 🔤 **文法の自然さ**: SV/SVO・自動詞/他動詞・三人称単数。`user.hasPermission()` が自然な英文として読める（出典: [Fujiwo](https://qiita.com/Fujiwo/items/741f7de1255b1e9aefc2)）
- #36 🌐 **パターン語彙の活用**: Factory／Builder／Validator／`~er` など確立された役割語彙に乗せる（出典: [Fujiwo](https://qiita.com/Fujiwo/items/741f7de1255b1e9aefc2)）
- #37 🔗 **既知の名前との連続性（後方互換）**: 公開 API・サービス名など、いったん広まった名前を昔と同じに保ち、既に知っている人に伝わり続けるか（一貫性の時間軸）
- #38 🌐 **ユーザの期待・語の含意に合わせる**: `get*` は軽い処理、`size()` は O(1) を期待される等、語が世間で持つ含意に逆らう名前は誤解を生む。「負けを認めて」誤解されない名前にする（リーダブルコード「ユーザの期待に合わせる」）
- #39 🔍 **タイプしやすさ・補完での発見性（typeability）**: IDE 補完で絞り込めるか、共通プレフィックスで補完候補がまとまるか、打鍵しやすいか
- #40 🔗 **比喩・概念モデルの一貫（メタファー一貫性）**: 名前を一貫した比喩体系から引き、混ぜない（郵便比喩 `mailbox`/`deliver` に `stream`/`flush` を混ぜない）。出典: XP「システムメタファー」
- #41 🎯 **リスク・警告を名前に込める（警告的命名）**: 危険・副作用・非推奨を名前で知らせ誤用を抑止する。`dangerouslySetInnerHTML`、`unsafe`/`UNSAFE_`、`unstable_`、`legacy_`、`raw`、`Sync`（同期＝ブロッキング）等。分類＝**明確性（意味の充足・自立）**。利用時の**安全性**を生む手段（means→outcome の好例）。固有の綴り `dangerously` は外部相対・スタイル適合性
- #42 🔗 **構造の反映（従属／並列の命名規則を一致させる）**: 名前の構造を要素どうしの関係に合わせる。従属（包含）は従属と分かる名前（`List`/`ListItem`）、並列は対等な命名規則（`Button`/`ButtonSpecial` は子要素に見え誤解 → `ButtonCommon`/`ButtonSpecial`）。分類＝**内部識別性・一貫側（内部相対）**。利用時の予測可能性にも寄与
- #43 🎯 **連番でなく意味のある名前**: 位置番号でなく中身/固有名で名付ける。動機で効く特性が 3 つに割れる ── (a) **記述名**で中身を語る（`img1`→`imgCircle`）＝**明確性**（副次で検索性・補完容易性）、(b) 互いに**紛れない名前**で取り違えを防ぐ（医薬品の類似名改名「アルマール→」https://www.nikkei.com/article/DGXNASDD160P5_W2A110C1TJ2000/ 、NATO フォネティック、米ハリケーン名）＝**弁別性**→利用時の**安全性**（番号の整列性とトレードオフ）、(c) **なじみのある固有名**で覚えやすく（台風のアジア名）＝**平易性**＋文化尊重で**包摂性**、防災で**読み手網羅性**。※台風アジア名は気象庁の主目的が (c)＝なじみ・文化であり (b) 取り違え防止ではない（当初 WMO の理屈と混同していた）。固有名でも動機で所属が変わる
- #44 🔍 **grep 可能な定型マーカー**: 自由文でなく `TODO:`／`FIXME:`／`HACK:`／`XXX:` など決まった語に揃え、後から横断検索・集計できるようにする。素直には自由なコメントだが、検索のために定型化する。分類＝**検索性**
- #45 🔭 **拡張余地のある名前（最上級・絶対語を避ける）**: 後から上位・下位を足せる名前にする。`s`/`m`/`l`/`xl` は `xxl`/`xs` を足せるが、`largest`/`final`/`newest`/`last` のような最上級・絶対語は上を足せず、足した瞬間に名前が嘘になる（`report_final`→`final_v2` の悲劇）。相対スケール・採番で開いておく。分類＝**適量性（#21 将来の拡張）→ 変更耐性**。最上級は将来 **意味的正確性** も壊す。連続性（過去向き）に対する**未来向き**の双子
- #46 🔍 **既存資産にたどり着ける（内部の到達）**: 同僚・将来の自分が、既にあるユーティリティ／定数を見つけて再利用できるか（重複実装の回避）。内部検索性＋明確性が支え、利用時の**到達可能性**を生む。
- #47 🔍 **外部から見つけてもらえる（外部の到達）**: 使ってほしいライブラリ・公開 API が Web 検索で見つかり採用されるか。外部検索性（ググラビリティ）が支え、利用時の**到達可能性**を生む。
- #48 🎯 **振る舞い・コストの正直さ**: 名前が *何をするか* について嘘をつかない。`getSize()` が O(n)、`get*` が裏で API 通信、`size()` が重い等、語の含意に反する振る舞いを避ける。単一の置き場でなく **意味的正確性（事実）＋外部識別性 #38（語の含意・期待）＋明確性（警告的命名 #41）** の交差で、利用時の **予測可能性・安全性** に効く（独立特性にはしない）。
- #49 ⚙️ **満足性（命名の心地よさ）※見送り**: ISO 利用時の品質の5カテゴリの満足性に当たるが、主観寄り＋コピーライティング（名前で惹きつける）側に滑るため、利用時アウトカムには採らず対象外。記事には「満足性は主観ゆえ対象外」と一文添える方針。

## 分類（命名品質特性）

### その手前: 「何に名付けるか」と Layer 0（床）

- **何に名付けるか**（対象選定・粒度）は設計の話。マジックナンバーや概念の塊をどこで切り出すか。名付けにくさは設計のサイン（凝集が低い・関心が混ざっている疑い）。本記事は「**どう良くするか**（品質）」に集中し、こちらは前段で触れるにとどめる（#33 もこの層に通じる）。
- **Layer 0（床）**: 言語仕様上 valid な識別子であること（コンパイルが通る）。さらに「動く風だが挙動が変わる／誤動作する」**妥当性ゲート**（予約語 #19・FW の必須規約〈React コンポーネント大文字＝小文字だと DOM 扱い〉#27・挙動が変わる文字種 #17）もここ。使える/動くが強制される話で、選んで良くする*質*ではない。

### 主軸: 何と照らして判断するか（製品品質）

名前のよさは「何と照らすか」で分かれる。自立評価 ⇔ 相対評価（内／外）の 2 階層。

```
自立品質        単体で評価が完結（絶対評価）
相対品質
├─ 内部相対品質  コード内の他の名前との関係＝自分との一致（空間内＋時間）
└─ 外部相対品質  コード外の規約・社会との一致＝外との一致
```

#### 自立品質

名前＋指示対象＋固定した前提（英語・採用 FW・対象読者の言語など）だけで判断できる。

- **意味的正確性**（旧正確性｜#1 #2）: 指示対象の事実と意味的に一致し、誤った含意を持たない（綴り・文法の正しさは言語的正確性へ）
- **言語的正確性**（旧適格性｜2 葉: 綴り orthography ＝#1 の綴り分／文法 grammar ＝#12 #13 #14 #15 #16 #35）: 自然言語として正しい。綴り（`recieve`→`receive`）と文法（品詞／単複／真偽値接頭辞／否定回避／時制／自然な英文）。意味的正確性と「意味系/形式系」で対
- **明確性**（意味の充足｜#4 #10 #31 #32 #33 #41 #43a）: 名前だけで意図・使い方が分かる。意図を示す／ドキュメント不要／クライアント視点／実装非エンコード／責務の単一／リスク・警告の明示（#41 `dangerouslySetInnerHTML` 等＝安全性に寄与）／連番でなく記述名（#43 の記述名側 `img1`→`imgCircle`）
- **平易性**（認知のしやすさ｜#9 #11 #8 #25 #43c）: 人が速く正しく読める。短く平易／発音可能／字形が紛れない／なじみのある固有名で覚えやすい（#43c 台風のアジア名。文化尊重で包摂性にも、防災で読み手網羅性にも波及）。※平易さは*固定した読者層*に対して判定（日本語話者向けに Storybook の Story 名を日本語に、等が #25）

#### 内部相対品質

- **内部識別性**（一貫＋弁別｜一貫 #6 #34 #40 #42 / 弁別 #5 #19 #43b）: コード内で「これは何か」を正しく見分けられる
  - 一貫: 内部一貫性(#6) / 垂直一貫(#34) / **比喩一貫性**(#40) / 構造の反映(#42) ※対義語ペア #7 は外部識別性へ移動
  - 弁別: **弁別性**・一意性(#5 #19) / 紛れない名で取り違え回避(#43b。医薬品類似名の改名・NATO フォネティック・米ハリケーン名。利用時の安全性へ。整列性とトレードオフ)
- **スタイル一貫性**（形式・記法の内部一貫｜#17 の内部側）: 形式・記法を自分のコード内で揃える（`UPPER_SNAKE`／`PascalCase`／`handle*`）。内部識別性（語彙）と意味/形式で対。`UPPER_SNAKE`→「定数」のような形式の識別便益は予測可能性（利用時）へ流す（means→end）。揃え先が世間の流儀なら外部の **スタイル適合性**（一貫↔適合）
- **適量性**（スコープに見合った情報量｜#3 #20 #21 #45）: スコープに対し過不足ない量。2方向の葉 ── **簡潔性**（減らす：スコープ既出の情報を繰り返さない・狭スコープは短く｜#3 / #20 狭側）／**十分性**（増やす：広・公開スコープに具体を足す｜#20 広側）。将来の拡張に名前を縛りつけない（#45 最上級・絶対語を避け相対スケールに）のもここ→変更耐性
- **内部検索性**（#22 #44）: grep（repo 内）で一意に当たる。`TODO:`/`FIXME:` のような定型マーカーで横断検索できるようにするのも内部検索性(#44)。外でのググラビリティは外部検索性へ
- **整列性**（#23）: 一覧で関連する名前が固まる
- **補完容易性**（タイプしやすさ｜#39）: IDE 補完で絞れる／打鍵しやすい
- **連続性**（時間軸｜#37）: 過去の自分の名前と同じであり続ける（公開 API・サービス名）

#### 外部相対品質

- **外部識別性**（一貫＋弁別。内部識別性と対称｜一貫 #24 #6 #18 #36 #38 #7 / 弁別 #19）: 外の世界で「これは何か」を正しく見分けられる
  - 一貫: **外部一貫性**（ドメイン用語#24 / 世間の概念名#6 / 略語慣習#18 / パターン語彙#36 / 慣習的な対義語ペア#7 / ユーザの期待#38）
  - 弁別: **外部弁別性**（組み込み・標準ライブラリ・商標・他サービスと紛れない #19）
- **外部検索性**（ググラビリティ｜#22 の外側）: Web 検索・SNS で見つかる。公開パッケージ・リポジトリ・公開 API・サービス名で効く。外部弁別性（外で一意）→外部検索性（外で見つかる）。内部検索性（grep）と内外の対。※*見つかるか*（品質）であって*魅力・コピー*（マーケ）は対象外
- **スタイル適合性**（#27 #17 の外部側）: 動くが合わせると親切な形式・スタイルの流儀（`kebab-case`、`*.test.ts`、React の `on*`/`handle*`/`[x, setX]` 等）。揃え先が世間の流儀。内部の **スタイル一貫性** と一貫↔適合で対
- **包摂性**（#26）: 社会・文化への配慮（差別的・他言語で不適切な語の回避）

### 利用時の品質（アウトカム）

製品品質（上記 3 大分類）が寄与して生まれる結果。複数特性にまたがるため特定の大分類には属さない。ISO 利用時の品質の 5 カテゴリ（有効性・効率性・満足性・リスク回避性・利用状況網羅性）のうち満足性を除く 4 つに対応づけ、7 つに整理（有効性＝予測可能性＋到達可能性、効率性＝理解効率＋伝達容易性、リスク回避性＝安全性＋変更耐性）。

- **予測可能性**（有効性）: 読み手が*正しく*意味を当てられる。内部識別性・外部識別性・スタイル一貫性・スタイル適合性・社会通念が寄与（`UPPER_SNAKE`→定数等、形式からカテゴリを当てる便益もここに集まる）。WCAG 3.2「予測可能」に対応。「誤解を招かない」はこの裏返し
- **到達可能性**（有効性）: 必要な人がその名前にたどり着けること（同僚が既存ユーティリティを見つけ再利用＝重複回避／ライブラリが Web 検索で見つかり採用）。内部検索性・外部検索性＋明確性・予測可能性が寄与。検索性＝*見つけられる性質*／到達可能性＝*たどり着けた結果* の means→outcome。#46 #47 を回収
- **理解効率**（効率性）: *速く・楽に*読み解ける。予測可能性＝「正しく分かる」／理解効率＝「速く分かる」で別軸。簡潔↔明確のように衝突しうる。※〜性で終わらない唯一の例（〜性で揃えるなら理解容易性／可読性。可読性は広すぎ却下経緯あり）。あとで見直す可能性あり
- **伝達容易性**（効率性）: 名前を*共有語彙*として、会話・レビュー・AI への指示で正確かつ素早く指し示せる。発音可能性（平易性）・明確性・予測可能性・読み手網羅性が寄与。理解効率＝*黙読*の速さ／伝達容易性＝*対話・指示*のしやすさ で別。冒頭「AI への指示」動機の回収。効率性が理解効率＋伝達容易性の2つを持つのは、リスク回避性が安全性＋変更耐性の2つを持つのと同型
- **安全性**（リスク回避性）: 紛らわしい名前による誤用・バグを防ぐ（`deleteAt`/`deletedAt`、List でないのに List）。意味的正確性・明確性（とくに警告的命名 #41 `dangerouslySetInnerHTML`）・外部識別性（期待）が寄与。ISO 製品品質の安全性とは別物（こちらは利用時）
- **変更耐性**（リスク回避性）: 明確性（意図を示す #4・実装非エンコード #32＝実装変更耐性）＋適量性（#20 #21＝スコープ変更耐性／#45 最上級を避け拡張余地を残す＝将来拡張耐性）が寄与
- **読み手網羅性**（利用状況網羅性）: 人・AI・新規参加者の誰にも通じる。冒頭の「読み手は人間だけでない（AI も）」の回収
- 〜（候補・見送り）満足性（命名のここちよさ・主観寄り）／認知負荷の低さ（理解効率・予測可能性と重複）

### トレードオフと裁定（記事の山場）

- ISO/IEC 25010 でも品質特性は全部同時に最大化できず、特性間にトレードオフがある。品質モデルは共通語彙を与えるだけで、優先順位は文脈・ステークホルダー要求で決める（トレードオフ分析手法 ATAM）。ISO は**製品品質と利用時の品質を別モデルに分ける**（2023 改訂で利用時品質は ISO/IEC 25019 へ）。本分類の 2 層構成はこれに対応。
- **中心的な衝突**: 簡潔さ・平易さ（平易性）↔ 明確性 ── どちらも自立品質。**スコープ（適量性・内部相対）が裁定**する（狭→簡潔／広・公開→明確）。
- **誤解を招かない**は照合先で割れる: 事実レベル（意味的正確性・自立）／通念レベル（#38・外部。通念を前提固定すれば自立寄り）。
- **裁定の基準**: 影響範囲 #29（公開名ほど慎重・明確側へ）／固定する前提（英語・FW・読者言語・社会通念）で境界が動く。
- **その他の衝突ペア**: 明確性 ↔ 将来の拡張 #21 ／ 内部一貫性 ↔ 個別最適 ／ 自然な英語 ↔ FW 規約 ／ クライアント視点 #31 ↔ 実装の素直さ ／ 実装非エンコード #32 ↔ ドキュメント不要 #10。

## 名前の理由・没案

各名前について、採った理由／没にした案／検討した選択肢を残す。
（大分類の軸名を決める作業そのものが「命名の営み」であり、この記事のテーマの実演。没案と理由を残すこと自体が記事の素材になる＝自己言及。）

### 大分類の軸名（自立品質 / 内部相対品質 / 外部相対品質）

軸は「自立評価（絶対評価）⇔ 相対評価」、相対を内／外に割る。採用は **自立品質 / 内部相対品質 / 外部相対品質**。検討した没案:

- 没・**絶対品質**（自立の対として）:「完璧な品質」に聞こえる。→ 中立な **自立** を採用。
- 没・**自己品質**:「単体で完結」のニュアンスが弱い。→ 自立。
- 没・**的あり**（自立的品質／内部相対的品質）:「相対的」は自然な形容詞だが、ISO の **内部品質／外部品質**（的なし）作法に揃える方が締まり term らしい。「内部相対的品質」は冗長。そもそも「絶対評価・相対評価」と言うので的なしで普通。
- 没・**文脈**（相対→内部文脈品質）: ① 自立の対義が消える ② 検索性・整列性は「文脈」でなく「相対（他と比べて一意）」の話 ③「文脈」は既に適量性の候補名・スコープの説明で使用済みで多重定義になる。※ 文脈は**適量性の葉名**としては有力。
- 没・**依存**（相対→内部依存品質）: 自立⇔依存は最も素直な対義だが、ソフトウェアでは「依存＝モジュール依存・結合」の確立した意味があり*結合度の品質*と誤読される。また依存＝従属で「他と照らして評価（比較）」とニュアンスがずれる。相対＝比較が正確で中立、相対評価の語にも繋がる。

### 明確性（意味の充足）

- 採用: リーダブルコードの「明確にする」と一致し、意図・責務・クライアント視点まで束ねられる。
- 没・**具体性**:「実装の具体」に引っ張られ #4 意図と逆を向きうる（`getThreeMonthsLater` の方が実装的には具体的）。採るなら「実装の具体でなく意図が具体的に立つこと」と定義が要る。
- 他候補: 意図明確性／自己説明性（逆向きリスクなし）／表現性／明瞭性／理解可能性。

### 平易性（認知のしやすさ）

- 採用: 短く平易（#9）に発音 #11・字形 #8 を束ねる。読者言語 #25 もここ（固定読者層に対する平易さ）。
- 没・**可読性**: 広すぎ（記事全体の別名になる）。
- 没・**認知容易性**: カーネマン由来の cognitive ease は核が familiarity（慣れ親しみ）で「処理負荷の低さ」とズレる。さらに「見慣れた体裁」まで含み外部一貫性と被る。出典: https://baigie.me/sogitani/2016/08/webdesign-standard/
- 葉候補・**判読可能性**: #8 字形の葉名には最適だがバケツには狭い（#9 平易・#11 発音を覆えない）。他: 知覚性／低認知負荷性。

### 言語的正確性（旧 適格性。意味的正確性と対）

- 採用: **言語的正確性**（英 Linguistic correctness）。**意味的正確性（旧 正確性）↔ 言語的正確性** で「意味系／形式系」の軸を名前そのもので体現（内部識別性／外部識別性と同型の対称）。中身は 2 葉 ── **綴り（orthography）** と **文法（grammar）**。
- **well-formedness / grammaticality は文法（ならび）の話で綴りを含まない**（[Wikipedia: Well-formedness](https://en.wikipedia.org/wiki/Well-formedness) 確認。当初「綴り＋文法を内包」としたのは誤り）。適格性/well-formedness 単独では綴りがこぼれるため、両方を覆う上位語「言語的正確性」を採用し綴り・文法を葉に分けた。
- 旧採用案 **適格性**: 言語学の適格性（grammaticality / well-formedness、適格文・不適格文）由来で `正確性・明確性・平易性・適格性` と韻も揃ったが、上記のとおり文法限定で綴りを覆えない。→ 言語的正確性へ。
- 別案・**正確性を分けない**: 元正確性へ綴り・文法を戻す案もあったが、意味（対 referent）と形式（対言語規則）は照合先が違うので分離を維持。元正確性は **意味的正確性** に改名して衝突回避。
- 没・**文法性 / grammaticality**: -ity で揃うが綴りを弾く。没・**整形式性 / well-formedness**: 機械寄りで XML の語として注釈が要る＋綴りを含まない。
- 注: 動かない／誤動作する妥当性ゲートはここでなく Layer 0。
- **スペルミス（綴り）は意味的正確性でなく言語的正確性（綴り葉）**（「スペルミスは文法的適格性では？」から）: 綴りは*指示対象*でなく*語形*に照らして判定＝形式の話。意味的正確性は意味（対 referent）の一致に純化。**意味的正確性(意味)↔言語的正確性(形式)** の対が締まる（意味系/形式系の軸と整合）。例外: タイポが別の実在語になる場合（`manager`→`manger`）は意味にも食い込むが、大半は語形の崩れ。
- 事例: **データ（名詞）なのに動詞句**＝品詞ミスマッチ #12。送信予定のメッセージを `sendMessage`（動作＝関数に見える）→ 名詞句 `messageToSend`/`outgoingMessage`。`userName`（関数なのに名詞）の逆向き。英語非母語話者がやりがち（#25 対象読者の言語と関連）。

### 内部識別性

- 識別性＝**一貫＋弁別**の傘。WCAG 3.2.4「一貫した識別性」（上位ガイドライン 3.2 予測可能）由来（着想。コードへの直接適用ではなく借用）。https://waic.jp/translations/WCAG22/Understanding/consistent-identification.html
- 一貫の葉: 内部一貫性 #6・垂直一貫 #34・比喩一貫性 #40・構造の反映 #42（従属/並列の命名規則を関係に合わせる。`ButtonSpecial`→`ButtonCommon`/`ButtonSpecial`）。（対義語ペア #7 は「正しい対語＝慣習・外部の語彙」なので外部一貫性へ移動。コード内でどのペアに揃えるかの内部の側面は #6 がカバー。）
- 弁別の葉: 弁別性／一意性／排他性（取り違えない・衝突しない）。一意性は内部検索性 #22 にも効く。
- **予測可能性は葉名にしない**（利用時の品質へ。複数特性にまたがる「なぜ」だから）。
- 没・**内部相対的識別性／内部文脈的識別性**（単独で分かりにくい懸念から）: 相対的＝親「内部相対品質」と重複し冗長、文脈的＝軸名に相対を採った経緯と不整合。→ 名前は据え置き、本文初出でグロス（「コード内の他の名前と照らした見分けやすさ」）を添える方針。

### 比喩一貫性（#40・内部識別性の一貫側）

- 比喩・語彙体系を混ぜない（概念モデルレベルの一貫）。#6 のトークン一貫とは別層、#34 垂直一貫とも別軸。
- 出典: XP「システムメタファー」。
- 代替: 語彙体系一貫性（広い）／概念モデル一貫性（正確だが #34 と紛らわしい）／メタファー一貫性（漢字の他葉と表記が浮く）。

### 適量性（スコープに見合った情報量。2 葉: 簡潔性／十分性）

- 採用:「スコープに対し過不足ない量」＝双方向（狭→削る／広→足す）を一語で表せる。さらに **2 方向に葉名**をつける（内部識別性の一貫／弁別と同型）。
  - 減らす＝**簡潔性**（Conciseness）: スコープ既出の情報を繰り返さない・狭スコープは短く（#3 / #20 狭側）。
  - 増やす＝**十分性**（Sufficiency）: 広・公開スコープでは名前だけで足りるよう具体を足す（#20 広側）。
- **簡潔性はバケツ名としては没**（片方向に見える）**だが、減らす方向の葉としては最適で採用**。ここでの簡潔は「*スコープに照らした*冗長排除」で、語単体の平易さ（平易性・自立）とは別の相対判断（`userName` 単体は難しくないが `user.` 内では冗長）。
- 十分性（増やす葉）の命名検討:
  - 採用 **十分性**。理由: もう一方の葉が `簡潔性`（平易な常用語）なので register を揃える／記事が平易性を重んじる／「簡潔↔十分」が量の過不足として一読で対になる。
  - 没・**具体性**: ① 明確性の語彙と衝突（明確性セクションが「具体性があること」と既に使用）、② 適量性は*量*の軸で、具体性は「抽象↔具体」の*具体度*という別軸（量でない。具体的でも情報不足はありうる）、③ 具体性にすると減らす葉が「抽象性」になってしまうが、減らす方向の実体は*冗長除去*であって抽象化ではない。
  - 没・**充足性**: term らしく硬い利点はあるが、難語寄り（平易性マイナス）＋「充足感＝満足・情緒」の含意がノイズ。十分性のほうが直感的。
- #45（拡張余地・最上級回避）は*将来方向*の別 facet→変更耐性。簡潔／十分は*現在のスコープ*に対する量。
- バケツ名の代替: 文脈明確性（「減らす方向」が出ない）／スコープ明確性。

### 内部検索性 / 外部検索性 / 整列性 / 補完容易性（発見性）

- 内部検索性: grep で一意に当たる（repo 内）。
- 外部検索性（ググラビリティ）: Web 検索・SNS で見つかる（公開パッケージ・リポジトリ・公開 API・サービス名）。
- **発見性クラスタは製品品質**（利用時に移さない）: 「弁別性と重なる（弁別→発見）から利用時の効果では？」に対し ── 検索性は*名前単体で測れる一意性*の話で複数特性の収束でないので製品。弁別→検索の重なりは*接続*であって outcome 化の根拠でない（移すと整列性・補完容易性も道連れで空洞化）。製品が affordance 名でよいのは ISO 保守性のテスト容易性と同じ（→構造上の判断）。
- **外部検索性を立てる（option B 採用）**: 弁別→検索は内外そろう（内部弁別性→内部検索性〈grep〉／外部弁別性→外部検索性〈ググラビリティ〉）。当初は外部検索性を外部弁別性に吸収（option A）も検討したが、**公開パッケージ・リポジトリ名の発見性は品質側**で記事スコープ（リポジトリ・公開 API）にも入るため B を採用し内外対称に。サービス名の*魅力・コピーライティング・商標取得可否*はマーケ別ドメインで対象外（境界を記事に明記）。
- 整列性: 一覧で関連が固まる。代替・順序性（順番寄りで「塊になる」が薄い）。
- 補完容易性 #39: IDE 補完・打鍵の体験。検索性・整列性と隣接（ツール上の話）だが別概念。代替: 入力容易性／タイプ容易性。

### 連続性（#37・時間軸）

- 採用: 過去の自分の名前と同じであり続ける（公開 API・サービス名）。
- 没・**互換性**: 前後どちらも含み「将来」寄りに読まれる。過去との一致を指すなら連続性が正確。
- 内部相対の理由: 照合先は社会でなく*自分の過去*。内部一貫性（空間内）と対で「自分との一致（空間＋時間）」。
- **外部連続性は立てない（内部に据え置き）**: 「公開 API は互換が崩れると動かない（referer）＝外部では？」という問いに対し ── 連続性の照合先は公開・非公開を問わず常に*自分が過去につけた名前*なので内部のまま。「外部公開だと重い」のは照合先でなく**影響範囲（#29）**の話で、裁定基準で既に扱う。referer を分解すると: 新規実装が定着綴りに合わせる＝**外部一貫性**／仕様側が綴りを保つ＝**連続性 ×影響範囲**。よって外部連続性という別特性は不要（連続性＋外部一貫性＋影響範囲に分配済み）。記事の連続性節にこの判断事例を記載。

### 外部識別性

- 内部識別性と対称（一貫＋弁別）。
- 外部一貫性（一貫）: ドメイン用語 #24／世間の概念名 #6／略語慣習 #18／パターン語彙 #36／ユーザの期待 #38。代替: 慣用性／慣習適合性／語彙整合性／ドメイン適合性。
- 外部弁別性（弁別）: 組み込み・標準ライブラリ・商標・既存サービス・他ライブラリと紛れない #19。

### スタイル一貫性（内部）／スタイル適合性（外部）

- **対で持つ**: 形式・記法の流儀も、語彙の一貫と同じく照合先ごとに版を持つ。内部で自分のコードを揃える＝**スタイル一貫性**（Style consistency）、世間の流儀に合わせる＝**スタイル適合性**（Style conformance）。**一貫（対 自分）↔ 適合（対 外部）**。
- 採用: 形式・スタイルの「流儀」と分かる。自立の「言語的正確性（形式）」と並ぶ形式系で、外部一貫性（語彙・意味）と意味/形式で対。
- 英語: **Style conformance / Style consistency**。没・**Idiomaticity**: プログラミングの idiomatic は「流儀に沿う」で命名にも使う（Effective Go のゲッターが好例）が、① 一般語義の idiom＝語彙の誤読、② 実装パターン・構成まで含み広すぎ、の二点で特性名には不適。idiomatic は地の文の説明語に留める。
- 没・**規約適合性**:「規約」は必須に聞こえるが、必須（動かない）は Layer 0 へ移してある。ここは動くが合わせると親切なものだけ。
- 代替（日本語）: 流儀適合性／慣例適合性。慣用性は外部一貫性（語彙の慣用）と紛れるため不可。
- 代替: 流儀適合性／慣例適合性／プラットフォーム適合性。

### 包摂性

- 採用: 社会・文化への配慮。差別的・他言語で不適切な語の回避。
- 没・**包括性**: 一般の DEI 文脈では inclusion を「包括性」と訳すことも多い（多様性・公平性・包括性）。が、本記事は「名前の性質」を列挙する文脈なので、包括性＝comprehensive/網羅的と読まれ **明確性・適量性（名前が概念を漏れなく覆うか）と衝突** する。包摂＝排除しない（社会的包摂）の方が意味も精密で衝突もない。register の硬さは記事（ISO・WCAG）と整合。
- 他候補: インクルーシブ性（語義衝突は回避するが漢語〇〇性の統一が崩れる）／社会・文化的配慮（誤解は最少だが〜性でない）。
- 記事には「inclusiveness は DEI 文脈で『包括性』と訳されることも多い」と一文添えた（読者の別呼称対応＋採用理由の明示）。

### 利用時の品質（6 アウトカム）

- 当初 2（予測可能性・変更耐性）→ ISO 利用時の品質（有効性・効率性・リスク回避性・利用状況網羅性）に棚卸しして 5 →伝達容易性（協働・AI 指示）で 6 →到達可能性（検索性のアウトカム）で **7** に拡張。
- 予測可能性（有効性）と理解効率（効率性）は近いが別物: 正しく分かる vs 速く分かる。簡潔↔明確と同型の衝突がアウトカム層でも再現。
- 安全性（リスク回避性）と変更耐性（リスク回避性）はどちらもリスク回避だが、誤用・バグの抑止 vs 改名コストの回避で別。
- 読み手網羅性（利用状況網羅性）で「人・AI」の伏線を回収。
- 伝達容易性（効率性・協働版）: 「コミュニケーションのしやすさ＝チーム会話＋AI への指示」を回収。採用 **伝達容易性**（英 Communicability）。候補 コミュニケーション容易性（カタカナで重い）／疎通性（意思疎通寄りで曖昧）／伝達性（簡潔だが「伝わる度合い」が弱い）。読み手網羅性（*誰に届くか*）とは別＝*指し示す道具*としての名前（会話・AI 指示でのハンドル）。
- 到達可能性（有効性）: 検索性（製品）のアウトカム側＝「実際に見つけてもらえた」。採用 **到達可能性**（英 Reachability）。**「発見可能性」を避けた**: デザイン分野では発見可能性＝*どんな操作が可能か・どう操作するかを見つけられるか*（Norman『誰のためのデザイン』）の意で定着しており、本項の「目的の名前にたどり着く」とずれるため、発見を使わず到達を採用（出典: https://note.com/amaino/n/n5bed6c539fef ）。読み手網羅性（見つけた*後*に通じる）／伝達容易性（既知の名を指す）とは別＝*未知にたどり着く*。
- 効果は大分類に置かない（複数特性が寄与する結果だから）。数: 製品品質 15＋利用時 7＝**22**。タイトル/冒頭も 22 に更新済み。
- 名前の宿題: 理解効率は〜性で終わらない唯一の例。揃えるなら理解容易性／可読性。「あとで見直すかも」。

## 構造上の判断（なぜこの分類か）

- **主軸は「何と照らして判断するか」1 本**。自立（絶対）／相対（内・外）。副軸（対象のレイヤー）はマトリクスにせず並び順としてだけ使う。
- **内部相対＝自分との一致**（空間内＝内部一貫性／時間＝連続性）、**外部相対＝外との一致**。
- **製品品質（3 大分類）と利用時の品質（6 アウトカム）の 2 層**（ISO 25010／25019 に倣う）。判別は2基準が一致 ── ① **静的に名前（集合）を見れば測れる＝製品／実際に読む・使う場面で初めて出る（「使ってみないとわからないこと」）＝利用時**、② **単一の名前の性質に還元できれば製品／複数特性の収束なら利用時**。
- **製品/利用時 は「作る側/使う側」でも「内部/外部」でもない**: ISO でも操作性・習得性（ユーザー寄り）すら*製品*、効率性（実利用の結果）が*利用時*。境界は「成果物の静的性質か／使った結果か」。名前の『ユーザー＝読み手』はほぼ開発側なので「利用時＝外部社会」は方向も誤り。
- **照合先軸（自立/内部/外部）と 製品/利用時 層は直交**: 反例 ── 外部相対品質（スタイル適合性等）は照合先＝外部だが*製品*／理解効率は読み手＝内部の開発者だが*利用時*。だから「製品＝内部・利用時＝外部」と重ねてはいけない。
- **製品特性は affordance（活動）で名づけてよい**: ISO 保守性のサブ特性が 解析性・修正性・テスト容易性 と「〜のしやすさ」で名づけつつ製品であるのと同じ。検索性・整列性・補完容易性も「名前を見れば測れる、ツール操作への向き不向き」＝製品。affordance 名は、同じ素の性質に乗った別の関心を見分ける役も果たす（弁別性/検索性＝一意性に乗る *区別/発見*、整列性/補完容易性＝族化に乗る *整列/補完*。素の性質名にすると衝突・融合する）。
- **同じ関心が照合先ごとに「版」を持つ**: 一貫性（内 #6／外）・識別（内／外）・スタイル（内＝スタイル一貫性／外＝スタイル適合性）。誤解を招かないは事実／通念で別軸。
- **分類の刃はメカニズム（意味系／形式系）、効果ではない**: 識別性＝語彙で見分ける／スタイル＝形式を揃える。「識別を助けるか」という*効果*で切ると良い命名がすべて識別性に吸い込まれて軸が崩れる。`UPPER_SNAKE`→「定数」のような形式由来の識別便益は、製品特性（スタイル一貫性）の効果として**予測可能性（利用時）**へ流す。識別性とスタイルは交差するだけで一方が他方を内包しない（kebab ファイル名＝形式だが識別に無関係／`date`→`startDate`＝識別だが形式無関係）ので、兄弟＋means→end で持つ。
- **前提固定で境界が動く**: 英語・React・読者言語・社会通念を固定すると外部相対が自立に寄る（例: React の `on*`/`handle*`/`[x, setX]` は React 前提なら自立、一般論なら外部相対）。
- **妥当性ゲート（動かない・誤動作）は Layer 0**。質ではなく前提。

## 記事化に向けたメモ

- **コンセプト**:「命名品質特性」。ISO 品質特性に倣い、名前のよさを分類（製品品質 3 大分類）し、利用時の品質と特性間の衝突・裁定まで書く。タイトルは「命名品質特性」系。
- **はじめに**: コードの読み手は人間だけでなく AI（コーディングエージェント）も含む、という時代背景（分類でなく導入で触れる）。
- **おわりに**: 命名は設計のフィードバック。名付けようとすると「まとめるべき／分けるべき」に気づける（#33・名付けにくさ＝設計のサインと呼応）。
- **出典**: fujiwo（Qiita /『AI 時代のきれいなプログラムの教科書』）#31-36 ── https://qiita.com/Fujiwo/items/741f7de1255b1e9aefc2 ／ WCAG 3.2.4 ／ ISO/IEC 25010・25019 ／ XP システムメタファー ／ リーダブルコード。

### 記事化での表現上の決定

- **構成**: 冒頭で分類を図（mermaid）＋表で見せる結論先行 → 導入（はじめに）→ 対象/対象外 → 主軸 → 3 大分類 → 利用時の品質 → トレードオフ → おわりに。
- **タイトル**:「命名品質特性 ── 良い名前を測る 3 つの軸と 22 の性質」。数字（3 軸・22 性質）を入れる。22＝製品品質 15＋利用時 7。
- **高低のバッテリー記法**: コード例は ✅/❌ をやめ **🪫＝その観点が低い／🔋＝高い** で表す。「絶対の正解／不正解」でなく観点上の高低（良し悪しは文脈次第）であることを凡例で明示。色覚配慮（赤緑単独を避け、形＝残量でも判別可）。
- **対象の断り**: 対象＝コードの名前（変数・関数・ファイル・リポジトリ・API）。発展で UI・サービス名・記事タイトルにも応用。例は JS/TS/React に偏るが概念は言語非依存。

## 分野横断の命名事例（記事コラム候補）

プログラミング外の命名も同じ命名品質特性で説明できる＝概念の汎用性を示す材料。「素直 → 目的のための逸脱」型と、「番号 vs 固有名」型が多い。

### 素直 → 目的のための逸脱

| 素直なら | 実際 | 動機（取る特性） | 犠牲 |
|---|---|---|---|
| `80586`（採番の続き） | **Pentium** | 数字は商標登録できない→固有名で商標保護＝**外部弁別性** | ― |
| 元素119を `Ununennium`（仮） | **正式名（例 Nihonium）** | 連番プレースホルダ→固有名。一意性＋発見地の**包摂性**（#43 同型） | ― |
| 「スペイン風邪」「武漢ウイルス」 | **COVID-19** | 地名・人名・動物を避ける WHO ガイドライン（2015）＝**包摂性**（差別・風評回避） | 記述性（中身を語らない） |
| `referer`/`creat`（誤綴り・e 抜き） | そのまま固定 | 広まった綴りを保つ＝**連続性**（後方互換） | **言語的正確性** |

### 番号 vs 固有名（#43 の実社会版）

| 対象 | 番号系 | 固有名系 | 軸 |
|---|---|---|---|
| 車 | BMW「3 シリーズ」＝序列の**整列性** | トヨタ「カローラ」＝記憶の**平易性**/ブランド | 整列 ↔ 記憶 |
| 家具 | 品番 | IKEA「BILLY」＝**平易性**＋体系（内部一貫性） | |
| 天体 | HD 48915・`Kepler-22b`＝**一意性・網羅性** | シリウス＝**平易性** | |
| 楽曲 | K.626・BWV＝**外部識別性・一意性** | 「運命」＝**平易性** | |
| 生物 | （和名はゆれる） | 学名・二名法＝**外部識別性・一意性**（言語非依存） | |

### 弁別性・安全性（取り違え防止）

- **Tall Man lettering**: `predniSONE`/`prednisoLONE` を部分大文字で区別＝**弁別性→安全性**（薬剤取り違え対策。#43b の字形版）。
- **ハリケーン名の引退（retire）**: 大災害の名前を再使用しない＝混同回避＋配慮。

### 体系・記号化（網羅性）

- 空港 `NRT`／通貨 `JPY`／言語 `ja`: 短く一意・言語非依存＝**外部一貫性・一意性・読み手網羅性**。
- 地下鉄の路線記号（`G` 銀座線・色つき）: 多言語・色覚配慮＝**読み手網羅性・弁別性**。

### アンチ例

- **Twitter → X**: ブランド刷新で**連続性・外部検索性・一意性**を同時に喪失。「X」は Web 検索最悪で外部弁別性も低い反面教師。
- **`report_final.doc` → `report_final_v2.doc` → `report_FINAL_final.doc`**: 最上級 `final` を使ったため拡張で破綻（#45 の典型的反例）。`report_v1`/`v2`… の採番や日付プレフィックスなら開いていた。「最上級・絶対語は上を足せず、足した瞬間に嘘になる」の好例。
