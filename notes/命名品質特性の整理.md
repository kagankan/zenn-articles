> 記事「命名品質特性」の設計ノート。記事本体は別ファイルに書き起こす。
> 残す目的: 観点の全体像／分類の構造／**各名前を採った理由・没にした理由・検討した選択肢**。

## Phase 1: 観点一覧（全 40）

出典の目印: 🟦 元スクラップ / 🟩 追加 / 🟪 fujiwo の分類由来。

### 意味・正確さ

- #1 🟦 **正確性**: 名前が指示対象（実体）と意味的に合致しているか（型と名前の矛盾を避ける。綴りの正しさは適格性へ）
- #2 🟩 **誤解を招かない（disinformation を避ける）**: `accountList` が実際は List 型でない等。「情報がない名前」より「嘘をつく名前」の方が有害
- #3 🟩 **冗長語の排除（noise words）**: スコープが既に与える情報を繰り返さない（`user.userName` → `user.name`、`ProductData`／`nameString`）
- #4 🟦 **実装ではなく意図を示す**: 「どのように」でなく「何のために」（`getThreeMonthsLater` → `getExpiredDate`）。`And` 連結を避け抽象表現を使う
- #41 🟩 **リスク・警告を名前に込める（警告的命名）**: 危険・副作用・非推奨を名前で知らせ誤用を抑止する。`dangerouslySetInnerHTML`、`unsafe`/`UNSAFE_`、`unstable_`、`legacy_`、`raw`、`Sync`（同期＝ブロッキング）等。分類＝**明確性（意味の充足・自立）**。利用時の**安全性**を生む手段（means→outcome の好例）。固有の綴り `dangerously` は外部相対・スタイル適合性

### 区別・一貫性

- #5 🟦 **弁別性**: 似たものと区別しやすいか（`date` → `startDate`）
- #6 🟦 **一貫性**: 同じ概念には同じ名前。リポジトリ全体で統一。世間一般の概念名に合わせる
- #7 🟩 **対義語・対称ペアの統一**: `get/set`・`open/close`・`begin/end`・`min/max`・`source/destination`
- #8 🟩 **視覚的な紛らわしさ**: `l`/`1`/`I`、`O`/`0`、`rn`/`m` など字形レベルの誤読を避ける
- #42 🟩 **構造の反映（従属／並列の命名規則を一致させる）**: 名前の構造を要素どうしの関係に合わせる。従属（包含）は従属と分かる名前（`List`/`ListItem`）、並列は対等な命名規則（`Button`/`ButtonSpecial` は子要素に見え誤解 → `ButtonCommon`/`ButtonSpecial`）。分類＝**内部識別性・一貫側（内部相対）**。利用時の予測可能性にも寄与
- #37 🟩 **既知の名前との連続性（後方互換）**: 公開 API・サービス名など、いったん広まった名前を昔と同じに保ち、既に知っている人に伝わり続けるか（一貫性の時間軸）
- #40 🟩 **比喩・概念モデルの一貫（メタファー一貫性）**: 名前を一貫した比喩体系から引き、混ぜない（郵便比喩 `mailbox`/`deliver` に `stream`/`flush` を混ぜない）。出典: XP「システムメタファー」

### 読みやすさ・分かりやすさ

- #9 🟦 **直感性**: 直感的に理解できるか。できるだけ短く、平易な用語（辞書を引かなくても分かる）
- #10 🟦 **ドキュメント不要な命名**: 名前だけで使い方が分かる（`duration` → `durationMs` で単位を明示）
- #11 🟩 **発音可能性**: 会話・レビューで口に出せるか（`genymdhms` は議論時に困る）

### 文法・形態

- #12 🟩 **品詞の一貫性**: 関数・メソッドは動詞句（`getName`・`isValid`）、変数・クラスは名詞句
- #13 🟩 **単数 / 複数**: コレクションは複数形（`users`）、単体は単数（`user`）
- #14 🟩 **真偽値の接頭辞規約**: `is` / `has` / `can` / `should`
- #15 🟩 **否定形・二重否定を避ける**: `isNotEnabled`・`disabled = false` を避け肯定形を基準にする
- #16 🟩 **時制・ライフサイクルの表現**: `isLoading`／`isLoaded`、`willUpdate`／`didUpdate`、`onClick`（受け口）／`handleClick`（実装）

### 表記・形式

- #17 🟦 **文字種・形式**: `-` と `_` の使い分け、ファイル名は小文字推奨、React コンポーネントはアッパーキャメル
- #18 🟩 **略語・頭字語の方針**: 略すかどうか、頭字語のケース統一（`userId`/`userID`）。標準略語（`id`・`url`）は可、場当たり略語は不可
- #19 🟩 **予約語・組み込みとの衝突回避**: `list`・`type`・`id`（Python）や標準ライブラリ名のシャドーイングを避ける

### スコープ・抽象度

- #20 🟦 **具体・抽象の度合い**: スコープが狭いなら抽象的でも可、広いなら具体的に
- #21 🟩 **将来の拡張性とのバランス**: 狭すぎるとリネームが必要、広すぎると意味が薄れる

### 発見性・整列・入力

- #22 🟦 **検索しやすさ（greppability / googlability）**: grep で見つかるか、Web 検索での発見性
- #23 🟦 **整列しやすさ（sortability）**: 一覧で関連が固まるか（`PrimaryButton` → `ButtonPrimary`、`image-01` の数値パディング）
- #39 🟩 **タイプしやすさ・補完での発見性（typeability）**: IDE 補完で絞り込めるか、共通プレフィックスで補完候補がまとまるか、打鍵しやすいか

### 文脈・ドメイン・人

- #24 🟩 **ドメイン用語／ユビキタス言語**: 業務・問題領域の語彙、ドメインエキスパートの言い回しに合わせる（DDD）
- #25 🟩 **対象読者・チームの言語**: 英語に統一するか、非母語話者に通じるか、ローマ字を許すか
- #26 🟩 **包摂的命名**: `master/slave` → `primary/replica`、`whitelist/blacklist` → `allowlist/denylist`
- #27 🟦 **フレームワークの規約**: React hooks は `use` から始める、等
- #38 🟩 **ユーザの期待・語の含意に合わせる**: `get*` は軽い処理、`size()` は O(1) を期待される等、語が世間で持つ含意に逆らう名前は誤解を生む。「負けを認めて」誤解されない名前にする（リーダブルコード「ユーザの期待に合わせる」）

### メタ観点（観点そのものの使い方）

- #28 🟦 **単独評価 vs 相対評価**: 名前単体での良し悪しと、他の命名との相対での良し悪しの両方を見る
- #29 🟩 **影響範囲に投資を比例させる**: ローカル変数のリネームは安いが、公開 API・URL・DB カラム・JSON キーは高コスト
- #30 🟩 **観点同士のトレードオフと優先順位付け**: 短さ↔正確さ、具体↔汎用などは衝突する。文脈ごとに優先順位を決める

### fujiwo の分類より

- #31 🟪 **クライアント視点での命名（SON）**: 実装側でなく「呼び出す側が欲しいもの」で名付ける
- #32 🟪 **実装・型をエンコードしない**: `CustomerList` の型露出やハンガリアン記法を避ける
- #33 🟪 **責務の単一性・境界**: 何をして／何をしないかが名前で分かる（`DoStuff` 回避）
- #34 🟪 **モデル間の垂直一貫性**: 分析→設計→実装の層をまたいでも同じ概念は同じ語で通す
- #35 🟪 **文法の自然さ**: SV/SVO・自動詞/他動詞・三人称単数。`user.hasPermission()` が自然な英文として読める
- #36 🟪 **パターン語彙の活用**: Factory／Builder／Validator／`~er` など確立された役割語彙に乗せる

## Phase 2: 分類

### その手前: 「何に名付けるか」と Layer 0（床）

- **何に名付けるか**（対象選定・粒度）は設計の話。マジックナンバーや概念の塊をどこで切り出すか。名付けにくさは設計のサイン（凝集が低い・関心が混ざっている疑い）。本記事は「**どう良くするか**（品質）」に集中し、こちらは前段で触れるにとどめる（#33 もこの層に通じる）。
- **Layer 0（床）**: 言語仕様上 valid な識別子であること（コンパイルが通る）。さらに「動く風だが挙動が変わる／誤動作する」**妥当性ゲート**（予約語 #19・FW の必須規約〈React コンポーネント大文字＝小文字だと DOM 扱い〉#27・挙動が変わる文字種 #17）もここ。使える/動くが強制される話で、選んで良くする*質*ではない。

### 主軸: 何と照らして判断するか（製品品質）

名前のよさは「何と照らすか」で分かれる。自立評価 ⇔ 相対評価（内／外）の 2 階層。

```
自立品質        単体で評価が完結（絶対評価）
相対品質
├─ 内部相対品質  コード内の他の名前との関係＝自分との一致（空間内＋時間）
└─ 外部相対品質  コード外の規約・社会との一致＝外との一致
```

#### 自立品質

名前＋指示対象＋固定した前提（英語・採用 FW・対象読者の言語など）だけで判断できる。

- **正確性**（正確さ｜#1 #2）: 指示対象の事実と意味的に一致し、誤った含意を持たない（綴り・形式の正しさは適格性へ）
- **明確性**（意味の充足｜#4 #10 #31 #32 #33 #41）: 名前だけで意図・使い方が分かる。意図を示す／ドキュメント不要／クライアント視点／実装非エンコード／責務の単一／リスク・警告の明示（#41 `dangerouslySetInnerHTML` 等＝安全性に寄与）
- **平易性**（認知のしやすさ｜#9 #11 #8 #25）: 人が速く正しく読める。短く平易／発音可能／字形が紛れない。※平易さは*固定した読者層*に対して判定（日本語話者向けに Storybook の Story 名を日本語に、等が #25）
- **適格性**（綴り・形式・文法の正しさ｜#1 の綴り分 #12 #13 #14 #15 #16 #35）: 英語＋固定した言語/FW で語として正しく整っている。綴り（`recieve`→`receive`）／品詞／単複／真偽値接頭辞／否定回避／時制／文法の自然さ

#### 内部相対品質

- **内部識別性**（一貫＋弁別｜一貫 #6 #34 #40 #42 / 弁別 #5 #19）: コード内で「これは何か」を正しく見分けられる
  - 一貫: 内部一貫性(#6) / 垂直一貫(#34) / **比喩一貫性**(#40) / 構造の反映(#42) ※対義語ペア #7 は外部識別性へ移動
  - 弁別: **弁別性**・一意性(#5 #19)
- **適量性**（スコープに見合った情報量｜#3 #20 #21）: スコープに対し過不足ない量（狭→削る／広→足す）
- **検索性**（#22）: grep で一意に当たる（repo 内が主。派生: ググラビリティ／エゴサビリティ）
- **整列性**（#23）: 一覧で関連する名前が固まる
- **補完容易性**（タイプしやすさ｜#39）: IDE 補完で絞れる／打鍵しやすい
- **連続性**（時間軸｜#37）: 過去の自分の名前と同じであり続ける（公開 API・サービス名）

#### 外部相対品質

- **外部識別性**（一貫＋弁別。内部識別性と対称｜一貫 #24 #6 #18 #36 #38 #7 / 弁別 #19）: 外の世界で「これは何か」を正しく見分けられる
  - 一貫: **外部一貫性**（ドメイン用語#24 / 世間の概念名#6 / 略語慣習#18 / パターン語彙#36 / 慣習的な対義語ペア#7 / ユーザの期待#38）
  - 弁別: **外部弁別性**（組み込み・標準ライブラリ・商標・他サービスと紛れない #19）
- **スタイル適合性**（#27 #17）: 動くが合わせると親切な形式・スタイルの流儀（`kebab-case`、`*.test.ts`、React の `on*`/`handle*`/`[x, setX]` 等）
- **包摂性**（#26）: 社会・文化への配慮（差別的・他言語で不適切な語の回避）

### 利用時の品質（アウトカム）

製品品質（上記 3 大分類）が寄与して生まれる結果。複数特性にまたがるため特定の大分類には属さない。ISO 利用時の品質（有効性・効率性・リスク回避性・利用状況網羅性）の 5 観点にマッピングして 5 つに整理（満足性は主観寄りで見送り）。

- **予測可能性**（有効性）: 読み手が*正しく*意味を当てられる。内部一貫性＋外部一貫性・スタイル適合性・社会通念が寄与。WCAG 3.2「予測可能」に対応。「誤解を招かない」はこの裏返し
- **理解効率**（効率性）: *速く・楽に*読み解ける。予測可能性＝「正しく分かる」／理解効率＝「速く分かる」で別軸。簡潔↔明確のように衝突しうる。※〜性で終わらない唯一の例（〜性で揃えるなら 理解容易性／可読性。可読性は広すぎ却下経緯あり）。あとで見直す可能性あり
- **安全性**（リスク回避性）: 紛らわしい名前による誤用・バグを防ぐ（`deleteAt`/`deletedAt`、List でないのに List）。正確性・明確性（とくに警告的命名 #41 `dangerouslySetInnerHTML`）・外部識別性（期待）が寄与。ISO 製品品質の安全性とは別物（こちらは利用時）
- **変更耐性**（リスク回避性）: 明確性（意図を示す #4・実装非エンコード #32＝実装変更耐性）＋適量性（#20 #21＝スコープ変更耐性）が寄与
- **読み手網羅性**（利用状況網羅性）: 人・AI・新規参加者の誰にも通じる。冒頭の「読み手は人間だけでない（AI も）」の回収
- 〜（候補・見送り）満足性（命名のここちよさ・主観寄り）／認知負荷の低さ（理解効率・予測可能性と重複）

### トレードオフと裁定（記事の山場）

- ISO/IEC 25010 でも品質特性は全部同時に最大化できず、特性間にトレードオフがある。品質モデルは共通語彙を与えるだけで、優先順位は文脈・ステークホルダー要求で決める（トレードオフ分析手法 ATAM）。ISO は**製品品質と利用時の品質を別モデルに分ける**（2023 改訂で利用時品質は ISO/IEC 25019 へ）。本分類の 2 層構成はこれに対応。
- **中心的な衝突**: 簡潔さ・平易さ（平易性）↔ 明確性 ── どちらも自立品質。**スコープ（適量性・内部相対）が裁定**する（狭→簡潔／広・公開→明確）。
- **誤解を招かない**は照合先で割れる: 事実レベル（正確性・自立）／通念レベル（#38・外部。通念を前提固定すれば自立寄り）。
- **裁定の基準**: 影響範囲 #29（公開名ほど慎重・明確側へ）／固定する前提（英語・FW・読者言語・社会通念）で境界が動く。
- **その他の衝突ペア**: 明確性 ↔ 将来の拡張 #21 ／ 内部一貫性 ↔ 個別最適 ／ 自然な英語 ↔ FW 規約 ／ クライアント視点 #31 ↔ 実装の素直さ ／ 実装非エンコード #32 ↔ ドキュメント不要 #10。

## 名前の理由・没案

各名前について、採った理由／没にした案／検討した選択肢を残す。
（大分類の軸名を決める作業そのものが「命名の営み」であり、この記事のテーマの実演。没案と理由を残すこと自体が記事の素材になる＝自己言及。）

### 大分類の軸名（自立品質 / 内部相対品質 / 外部相対品質）

軸は「自立評価（絶対評価）⇔ 相対評価」、相対を内／外に割る。採用は **自立品質 / 内部相対品質 / 外部相対品質**。検討した没案:

- 没・**絶対品質**（自立の対として）: 「完璧な品質」に聞こえる。→ 中立な **自立** を採用。
- 没・**自己品質**: 「単体で完結」のニュアンスが弱い。→ 自立。
- 没・**的あり**（自立的品質／内部相対的品質）: 「相対的」は自然な形容詞だが、ISO の **内部品質／外部品質**（的なし）作法に揃える方が締まり term らしい。「内部相対的品質」は冗長。そもそも「絶対評価・相対評価」と言うので的なしで普通。
- 没・**文脈**（相対→内部文脈品質）: ① 自立 の対義が消える ② 検索性・整列性は「文脈」でなく「相対（他と比べて一意）」の話 ③「文脈」は既に適量性の候補名・スコープの説明で使用済みで多重定義になる。※ 文脈は**適量性の葉名**としては有力。
- 没・**依存**（相対→内部依存品質）: 自立⇔依存 は最も素直な対義だが、ソフトウェアでは「依存＝モジュール依存・結合」の確立した意味があり*結合度の品質*と誤読される。また依存＝従属で「他と照らして評価（比較）」とニュアンスがずれる。相対＝比較が正確で中立、相対評価の語にも繋がる。

### 明確性（意味の充足）

- 採用: リーダブルコードの「明確にする」と一致し、意図・責務・クライアント視点まで束ねられる。
- 没・**具体性**: 「実装の具体」に引っ張られ #4 意図と逆を向きうる（`getThreeMonthsLater` の方が実装的には具体的）。採るなら「実装の具体でなく意図が具体的に立つこと」と定義が要る。
- 他候補: 意図明確性／自己説明性（逆向きリスクなし）／表現性／明瞭性／理解可能性。

### 平易性（認知のしやすさ）

- 採用: 短く平易（#9）に発音 #11・字形 #8 を束ねる。読者言語 #25 もここ（固定読者層に対する平易さ）。
- 没・**可読性**: 広すぎ（記事全体の別名になる）。
- 没・**認知容易性**: カーネマン由来の cognitive ease は核が familiarity（慣れ親しみ）で「処理負荷の低さ」とズレる。さらに「見慣れた体裁」まで含み外部一貫性と被る。出典: https://baigie.me/sogitani/2016/08/webdesign-standard/
- 葉候補・**判読可能性**: #8 字形の葉名には最適だがバケツには狭い（#9 平易・#11 発音を覆えない）。他: 知覚性／低認知負荷性。

### 適格性（形式・文法の正しさ）

- 採用: **適格性**。言語学の「適格性（grammaticality / well-formedness、適格文・不適格文）」由来。中身が自然言語の文法的正しさ（品詞・単複・時制・自然な英文 #35）なので、自然言語寄りのこの語が合う。`正確性・明確性・平易性・適格性` と 3 文字で韻も揃う。難点「適格性＝資格・eligibility」とも取れるが、文脈と補足で解消。
- 没・**整形式性**: well-formed（XML・形式言語由来。構文規則に準拠の意）。厳密だが機械寄りで、XML の語として注釈が要る。
- 他候補: 文法性／文法的正しさ／文法的適格性（明示的だが長い）。
- 注: 動かない／誤動作する妥当性ゲートはここでなく Layer 0。
- **スペルミス（綴り）は正確性でなく適格性**（「スペルミスは文法的適格性では？」から）: 綴りは*指示対象*でなく*語形*に照らして判定＝形式の話。正確性は意味（対 referent）の一致に純化。**正確性(意味)↔適格性(形式)** の対が締まる（意味系/形式系の軸と整合）。例外: タイポが別の実在語になる場合（`manager`→`manger`）は意味にも食い込むが、大半は語形の崩れ。
- 事例: **データ（名詞）なのに動詞句**＝品詞ミスマッチ #12。送信予定のメッセージを `sendMessage`（動作＝関数に見える）→ 名詞句 `messageToSend`/`outgoingMessage`。`userName`（関数なのに名詞）の逆向き。英語非母語話者がやりがち（#25 対象読者の言語と関連）。

### 内部識別性

- 識別性＝**一貫＋弁別**の傘。WCAG 3.2.4「一貫した識別性」（上位ガイドライン 3.2 予測可能）由来（着想。コードへの直接適用ではなく借用）。https://waic.jp/translations/WCAG22/Understanding/consistent-identification.html
- 一貫の葉: 内部一貫性 #6・垂直一貫 #34・比喩一貫性 #40・構造の反映 #42（従属/並列の命名規則を関係に合わせる。`ButtonSpecial`→`ButtonCommon`/`ButtonSpecial`）。（対義語ペア #7 は「正しい対語＝慣習・外部の語彙」なので外部一貫性へ移動。コード内でどのペアに揃えるかの内部の側面は #6 がカバー。）
- 弁別の葉: 弁別性／一意性／排他性（取り違えない・衝突しない）。一意性は検索性 #22 にも効く。
- **予測可能性は葉名にしない**（利用時の品質へ。複数特性にまたがる「なぜ」だから）。
- 没・**内部相対的識別性／内部文脈的識別性**（単独で分かりにくい懸念から）: 相対的＝親「内部相対品質」と重複し冗長、文脈的＝軸名に相対を採った経緯と不整合。→ 名前は据え置き、本文初出でグロス（「コード内の他の名前と照らした見分けやすさ」）を添える方針。

### 比喩一貫性（#40・内部識別性の一貫側）

- 比喩・語彙体系を混ぜない（概念モデルレベルの一貫）。#6 のトークン一貫とは別層、#34 垂直一貫とも別軸。
- 出典: XP「システムメタファー」。
- 代替: 語彙体系一貫性（広い）／概念モデル一貫性（正確だが #34 と紛らわしい）／メタファー一貫性（漢字の他葉と表記が浮く）。

### 適量性（スコープに見合った情報量）

- 採用: 「スコープに対し過不足ない量」＝双方向（狭→削る／広→足す）を表せる。
- 没・**簡潔性**: 片方向（削るだけ）に見える。かつ簡潔さ自体は平易性（自立）の仲間で、スコープの話ではない。
- 代替: 文脈明確性（明確性と対で見せられるが「減らす方向」が出ない）／スコープ明確性。

### 検索性 / 整列性 / 補完容易性（発見性）

- 検索性: grep で一意に当たる（repo 内）が主。派生のググラビリティ／エゴサビリティは開発との関連が薄く補足扱い。
- 整列性: 一覧で関連が固まる。代替・順序性（順番寄りで「塊になる」が薄い）。
- 補完容易性 #39: IDE 補完・打鍵の体験。検索性・整列性と隣接（ツール上の話）だが別概念。代替: 入力容易性／タイプ容易性。

### 連続性（#37・時間軸）

- 採用: 過去の自分の名前と同じであり続ける（公開 API・サービス名）。
- 没・**互換性**: 前後どちらも含み「将来」寄りに読まれる。過去との一致を指すなら連続性が正確。
- 内部相対の理由: 照合先は社会でなく*自分の過去*。内部一貫性（空間内）と対で「自分との一致（空間＋時間）」。

### 外部識別性

- 内部識別性と対称（一貫＋弁別）。
- 外部一貫性（一貫）: ドメイン用語 #24／世間の概念名 #6／略語慣習 #18／パターン語彙 #36／ユーザの期待 #38。代替: 慣用性／慣習適合性／語彙整合性／ドメイン適合性。
- 外部弁別性（弁別）: 組み込み・標準ライブラリ・商標・既存サービス・他ライブラリと紛れない #19。

### スタイル適合性（エコシステム規約）

- 採用: 形式・スタイルの「流儀」と分かる。自立の「形式・文法の正しさ」と意味/形式で対（外部一貫性＝語彙・意味／スタイル適合性＝形式・スタイル）。
- 没・**規約適合性**: 「規約」は必須に聞こえるが、必須（動かない）は Layer 0 へ移してある。ここは動くが合わせると親切なものだけ。
- 代替: 流儀適合性／慣例適合性／プラットフォーム適合性。

### 包摂性

- 採用: 社会・文化への配慮。差別的・他言語で不適切な語の回避。
- 没・**包括性**: 一般の DEI 文脈では inclusion を「包括性」と訳すことも多い（多様性・公平性・包括性）。が、本記事は「名前の性質」を列挙する文脈なので、包括性＝comprehensive/網羅的 と読まれ **明確性・適量性（名前が概念を漏れなく覆うか）と衝突** する。包摂＝排除しない（社会的包摂）の方が意味も精密で衝突もない。register の硬さは記事（ISO・WCAG）と整合。
- 他候補: インクルーシブ性（語義衝突は回避するが漢語〇〇性の統一が崩れる）／社会・文化的配慮（誤解は最少だが〜性でない）。
- 記事には「inclusiveness は DEI 文脈で『包括性』と訳されることも多い」と一文添えた（読者の別呼称対応＋採用理由の明示）。

### 利用時の品質（5 アウトカム）

- 当初 2（予測可能性・変更耐性）→ ISO 利用時の品質（有効性・効率性・リスク回避性・利用状況網羅性）に棚卸しして **5** に拡張。
- 予測可能性（有効性）と理解効率（効率性）は近いが別物: 正しく分かる vs 速く分かる。簡潔↔明確と同型の衝突がアウトカム層でも再現。
- 安全性（リスク回避性）と変更耐性（リスク回避性）はどちらもリスク回避だが、誤用・バグの抑止 vs 改名コストの回避で別。
- 読み手網羅性（利用状況網羅性）で「人・AI」の伏線を回収。
- 効果は大分類に置かない（複数特性が寄与する結果だから）。数: 製品品質13＋利用時5＝**18**。タイトル/冒頭も 18 に更新済み。
- 名前の宿題: 理解効率 は〜性で終わらない唯一の例。揃えるなら 理解容易性／可読性。「あとで見直すかも」。

## 構造上の判断（なぜこの分類か）

- **主軸は「何と照らして判断するか」1 本**。自立（絶対）／相対（内・外）。副軸（対象のレイヤー）はマトリクスにせず並び順としてだけ使う。
- **内部相対＝自分との一致**（空間内＝内部一貫性／時間＝連続性）、**外部相対＝外との一致**。
- **製品品質（3 大分類）と利用時の品質（予測可能性・変更耐性）の 2 層**（ISO 25010／25019 に倣う）。判別: 単一の製品特性に還元できれば製品品質、複数特性の寄与で生まれる結果なら利用時の品質。
- **同じ関心が照合先ごとに「版」を持つ**: 一貫性（内 #6／外）・識別（内／外）。誤解を招かないは事実／通念で別軸。
- **前提固定で境界が動く**: 英語・React・読者言語・社会通念を固定すると外部相対が自立に寄る（例: React の `on*`/`handle*`/`[x, setX]` は React 前提なら自立、一般論なら外部相対）。
- **妥当性ゲート（動かない・誤動作）は Layer 0**。質ではなく前提。

## 記事化に向けたメモ

- **コンセプト**: 「命名品質特性」。ISO 品質特性に倣い、名前のよさを分類（製品品質 3 大分類）し、利用時の品質と特性間の衝突・裁定まで書く。タイトルは「命名品質特性」系。
- **はじめに**: コードの読み手は人間だけでなく AI（コーディングエージェント）も含む、という時代背景（分類でなく導入で触れる）。
- **おわりに**: 命名は設計のフィードバック。名付けようとすると「まとめるべき／分けるべき」に気づける（#33・名付けにくさ＝設計のサイン と呼応）。

### 記事化での表現上の決定

- **構成**: 冒頭で分類を図（mermaid）＋表で見せる結論先行 → 導入（はじめに）→ 対象/対象外 → 主軸 → 3 大分類 → 利用時の品質 → トレードオフ → おわりに。
