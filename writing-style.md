# 執筆スタイルガイド：技術記事の書き方分析

このドキュメントは、articles 配下の記事を分析して抽出した、執筆スタイルの特徴をまとめたものです。

## 1. 文体的特徴 (Writing Style Characteristics)

### 文の長さと構成

- **短文と中文のバランス**: 一文は基本的に短めで、1 つの文で 1 つのことを伝える明快なスタイル
- **箇条書きの積極活用**: 複数の項目を並べる際は文章で繋げず、箇条書き（`-` や番号付きリスト）で明示
- **段落は短め**: 1 段落は 3〜5 文程度で区切り、読みやすさを重視

### 敬体と常体の使い分け

- **基本は常体（だ・である調）**: 技術記事では「〜です・〜ます」より「〜だ・〜である」を使用
- **「しましょう」「〜てください」の活用**: 読者への呼びかけや推奨事項では丁寧な表現を使用
- **「〜ですね」「〜ですよね」**: 親しみやすさを出すため、時折語りかけるトーンを挟む

### 技術用語の扱い

- **初出時の説明**: 技術用語は初出時に簡潔に説明を加える
  - 例：「DOM（どむ、Document Object Model）」
  - 例：「AOM （えーおーえむ、Accessibility Object Model）」
- **カタカナと英語の併記**: 「Widening と呼ばれる型推論の仕組み」のように、カタカナ表記と英語を併記
- **日本語での説明を優先**: 可能な限り日本語で説明し、専門用語は補足的に使用

### 読者への配慮

- **「実は」「ただし」「なお」**: 補足情報を自然に挿入
- **「ちなみに」「参考までに」**: 脱線や補足を明示的に示す
- **「例えば」「具体的には」**: 抽象的な説明の後に具体例を提示

## 2. 構造パターン (Structure Patterns)

### 記事全体の構造

**標準的な構成パターン：**

```markdown
---
title: "タイトル"
emoji: "絵文字"
type: "tech" or "idea"
topics: [トピック配列]
published: true/false
---

## 概要 / はじめに / この記事の概要

（記事の目的と内容を簡潔に説明）

## 前提 / 確認した環境 / 動作確認環境

（必要に応じてバージョン情報や前提条件）

## 本文セクション 1

（メインコンテンツ）

## 本文セクション 2

...

## まとめ / おわりに

（要点の再確認や締めの言葉）

## 参考 / 参考リンク集 / 参考記事

（引用元や関連リンク）
```

### 見出しの使い方

- **h1 は記事タイトルのみ**: 本文は h2 から開始
- **h2 が主要セクション**: 大きなトピックの区切り
- **h3 以降は h2 の詳細**: 階層構造を意識し、飛び級しない（h2 の次は h3、h4 は使用しない）
- **見出しは体言止めではなく動詞句**: 「〜を理解する」「〜の使い方」「〜とは」

### コードサンプルの配置

**基本パターン：**

1. 問題提起・状況説明
2. コード例（Before/NG 例）
3. 解説
4. 改善コード（After/OK 例）
5. 解説・メリットの説明

**コードブロックのラベル：**

- `Before/After` または `NG例/OK例` で対比
- 絵文字を活用：`😖Before` `✅️After`、`❌ NG例` `✅ OK例`
- ファイル名を明示：` ```tsx:App.tsx `

## 3. トーンと視点 (Tone and Perspective)

### 技術説明のトーン

- **親しみやすく、でも正確**: 堅苦しくなく、でも技術的に正確な説明
- **「〜ですね」「〜ですよね」**: 共感を呼ぶ語尾の使用
- **「実は」「意外と」**: 驚きや発見を共有するトーン
- **「なんと」「めちゃめちゃ」**: 強調したいポイントで口語的な表現

### 読者への配慮の表現

- **「お気をつけください」**: 注意喚起
- **「ぜひ〜してみてください」**: 推奨アクション
- **「〜かもしれません」**: 断定を避ける柔らかい表現
- **「個人的には」「私の場合は」**: 主観的な意見であることを明示

### 問題提起の方法

- **「こんなことってないですか？」**: 読者の共感を呼ぶ
- **「〜で困っていませんか？」**: 問題意識の共有
- **「実は ○○ なのです」**: 意外な事実の提示
- **クイズ形式**: 「以下のコードを実行するとどう出力されるでしょうか？」

## 4. 技術的な表現 (Technical Expression)

### コード例の説明方法

**段階的な説明：**

1. **概要の提示**: 何をしたいのか
2. **コード提示**: 実際のコード
3. **詳細解説**: コードの各部分の説明
4. **メリットの明示**: なぜこの方法が良いのか

**比較による説明：**

- Before/After パターン
- NG/OK パターン
- 複数の解答例の提示（「例 1: 〜」「例 2: 〜」）

### 技術概念の説明スタイル

**抽象 → 具体のパターン：**

```markdown
## 概念の説明

（抽象的な定義）

### 具体例

（実際のコード例）

### メリット

（なぜ重要なのか）
```

**たとえ話の活用：**

- 「エアコンのリモコン」で「宣言的」「命令的」を説明
- 「本を読むときに目次を見て」で見出しの重要性を説明
- 日常的な例を技術概念の説明に活用

### 比喩と例の使用

- **身近な例**: 技術的な概念を日常の物事で例える
- **具体的な型の例**: `Pokemon` 型で TypeScript の型を説明
- **料理レシピ**: HTML のセマンティック構造を説明

## 5. その他の特徴 (Other Characteristics)

### よく使われる表現

**導入表現：**

- 「突然ですが」
- 「改めて…」
- 「さて」「では」
- 「というわけで」

**強調表現：**

- 「**重要**」：太字で強調
- 「めちゃめちゃ〜」
- 「なんと」
- 「実は」

**補足表現：**

- 「ちなみに」
- 「なお」
- 「ただし」
- 「（参考まで）」

### 絵文字の使用パターン

**記事メタデータ：**

- 各記事に関連する絵文字を 1 つ設定

**コード例：**

- `😖` : 改善の余地あり
- `✅` : 良い例
- `❌` : 悪い例
- `🚀` : パフォーマンス向上
- `😊` : 嬉しい・ポジティブな結果

**本文中：**

- 使用は控えめ
- 括弧内で使用：「（嬉しい 😌）」
- リスト項目で使用：「- 🎉 新機能」

### リンクと引用の扱い

**外部リンク：**

- MDN やドキュメントへのリンクは積極的に提示
- 参考記事は「参考」「参考リンク集」セクションにまとめる
- 記事中では `https://...` の形式でそのまま提示

**内部引用：**

- 自分の過去記事への言及：「以前公開した記事で〜」「別の記事で〜」

**引用の形式：**

```markdown
> 引用文
>
> 引用元 URL
```

### 注意喚起・補足情報の表現

**Zenn のメッセージボックス活用：**

```markdown
:::message
補足情報や注意事項
:::

:::details タイトル
詳細な説明（折りたたみ）
:::
```

**使用パターン：**

- `:::message` : 重要な補足、前提条件、注意事項
- `:::details` : 長い補足、コード例、追加情報

## 6. 記事タイプ別の特徴

### 技術解説記事（type: "tech"）

- 問題提起 → 解決策 → コード例 → メリット説明
- バージョン情報を明記
- 実行例・動作確認を含める

### アイデア記事（type: "idea"）

- 体験談や感想を含める
- 「〜してみた」「〜を受けてきた」
- より口語的で柔軟な文体

### クイズ形式記事

1. クイズ提示
2. 考える時間（「:::details 答え」で折りたたみ）
3. 解説
4. ポイントの明示

## 7. 特徴的なスタイル要素

### 「最初にまとめ」パターン

結論先行型の記事では、冒頭に要点をまとめて提示：

```markdown
## 最初にまとめ

この記事で説明するポイントは以下の 5 つです：

1. ポイント 1
2. ポイント 2
   ...

詳細は以降で説明します。
```

### 「想定読者」の明示

技術記事では対象読者を明確に：

```markdown
## 想定読者

- JavaScript を書くことには慣れている
- TypeScript はこれから学ぶ
- 〜についてすでに知っている
```

### 実践的な演習の提供

理論だけでなく、実際に試せる内容を提供：

- TypeScript Playground へのリンク
- CodePen の埋め込み
- 実際のコード例とその実行結果

## まとめ：執筆の 3 つの重要ポイント

この分析に基づいて記事を書くことで、読みやすく、技術的に正確で、親しみやすい技術記事を執筆できます。特に重要なのは以下の 3 点です：

1. **結論を先に書く**: 読者が記事の価値をすぐに理解できるように、要点を冒頭で示す
2. **具体例を豊富に使う**: 抽象的な概念は必ず具体的なコード例や身近な例で説明する
3. **読者への配慮を忘れない**: 親しみやすく、でも正確に、技術的な内容を伝える
